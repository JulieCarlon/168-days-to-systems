1. 抓包的本质：数据包是怎么被"复制"出来的

在能用工具之前，先搞清楚一件事：抓包（packet capture）到底在抓什么。你的网卡（NIC）正常工作时，会过滤掉不是发给自己的帧——它看以太网头里的目标 MAC 地址，不是自己的就丢弃。抓包工具做的第一件事，就是让网卡进入混杂模式（promiscuous mode）或监听模式，把经过它的所有帧都收下来，并在内核里复制一份交给用户态程序。

类比：把网卡想象成公司前台。正常模式下，前台只签收写着"你"名字的快递，别人的直接退回。混杂模式相当于前台把每一个经过大厅的快递都拆开拍照存档一份再放行。注意"经过大厅"这个限定——这正是关键：你只能抓到物理上流经你这块网卡的流量。在交换机（switch）组成的现代局域网里，交换机只把帧送到目标端口，所以默认情况下你抓不到隔壁同事的流量——除非你做了 ARP 欺骗把流量引到自己这儿（这就是后面 MITM 的前提），或者在交换机上配了端口镜像（SPAN）。

抓下来的字节按协议分层封装，像俄罗斯套娃，这是你 Day 15 学过的封装在真实字节上的样子：

[ 以太网帧 Ethernet ] 目标MAC | 源MAC | 类型(0x0800=IPv4) └─[ IP 包 ] 版本 | 源IP | 目标IP | 协议(6=TCP) └─[ TCP 段 ] 源端口 | 目标端口 | seq | ack | flags(SYN/ACK...) └─[ 应用层 ] HTTP 文本 / TLS 记录 / DNS 查询 ... 抓包工具的工作 = 把这一层层壳剥开，每一层都解析成人类可读的字段

底层支撑这一切的是一套叫 BPF（Berkeley Packet Filter）的内核机制——你在 Day 13 听过它的现代版 eBPF。BPF 让你在内核里就先过滤掉不关心的包，只把符合条件的拷贝到用户态，避免海量流量把工具淹死。Linux 上对应 libpcap 库，macOS 也用它，Windows 上叫 Npcap。tcpdump、Wireshark、Python 的 scapy 全都站在 libpcap 之上——所以学会 BPF 过滤语法，等于一把钥匙开所有门。

关键事实
  • 抓包 = 网卡混杂模式 + 内核 BPF 过滤 + 拷贝到用户态，三步。
  • 交换网络里默认只能抓到自己的流量；要抓别人的需 ARP 欺骗或端口镜像。
  • tcpdump / Wireshark / scapy 共享同一套底层（libpcap）和同一套 BPF 捕获过滤语法。
  • 抓包文件标准格式是 .pcap / .pcapng，可以保存后离线分析、跨工具交换。

2. tcpdump 与 BPF 过滤表达式（通用语言）

tcpdump 是命令行抓包的瑞士军刀。没有图形界面，但正因为如此它能在任何一台服务器、容器、跳板机上直接用——这是你日后排查线上问题的看家本领。它的命令结构是：tcpdump [选项] [BPF过滤表达式]。

先跑起来

# macOS / Linux 都需要 root（抓包是特权操作）
# -i 指定网卡；en0 是 mac 的 WiFi，Linux 上常是 eth0/wlan0/ens33
# -n 不做 DNS 反解（否则每个 IP 都去查域名，又慢又产生额外流量）
sudo tcpdump -i en0 -n

# 列出所有可抓的网卡
tcpdump -D            # Linux: ip link 也能看
预期：满屏滚动，每行一个包，形如
14:22:01.345 IP 192.168.1.10.51234 > 142.250.1.1.443: Flags [S], seq 12345, win 65535
[S] 就是 SYN——你亲眼看到了 Day 16 学的三次握手第一步。[S.] 是 SYN+ACK，[.] 是纯 ACK，[P.] 带 PUSH 表示有数据，[F.] 是 FIN。

BPF 过滤表达式：抓包的"WHERE 子句"

满屏滚动没法看，真正的功力在于过滤。BPF 表达式像 SQL 的 WHERE，用三类原语（primitives）组合：

类别关键词例子
主机/网络host net src dsthost 8.8.8.8、src net 10.0.0.0/8
端口port portrangeport 443、portrange 8000-9000
协议tcp udp icmp arpudp port 53（DNS）

用 and / or / not 组合，括号需转义或加引号：

# 只看与某主机的 HTTPS 流量
sudo tcpdump -i en0 -n 'host 142.250.1.1 and port 443'

# 只看 DNS 查询（UDP 53）——抓 DNS 投毒的第一步
sudo tcpdump -i en0 -n 'udp port 53'

# 只看 TCP 握手与挥手包（排除大量数据包），看连接的"骨架"
# tcp[tcpflags] 直接读 TCP flags 字节，SYN=0x02 FIN=0x01 RST=0x04
sudo tcpdump -i en0 -n 'tcp[tcpflags] & (tcp-syn|tcp-fin|tcp-rst) != 0'

# 抓某网段进出但排除 SSH（不想看自己的远程登录流量刷屏）
sudo tcpdump -i en0 -n 'net 192.168.1.0/24 and not port 22'

看内容，不只看头

# -A 以 ASCII 打印 payload（看明文 HTTP 最爽）
sudo tcpdump -i en0 -n -A 'tcp port 80'

# -X 同时打印十六进制+ASCII（分析二进制协议、TLS 记录）
sudo tcpdump -i en0 -n -X 'tcp port 443'

# -w 存盘成 pcap，之后用 Wireshark 慢慢看（生产环境标准做法）
sudo tcpdump -i en0 -n -w /tmp/capture.pcap 'host 142.250.1.1'
#   抓够了 Ctrl-C，然后：
tcpdump -r /tmp/capture.pcap -n      # -r 读回文件离线分析
预期：抓 80 端口加 -A，你能直接读到 GET / HTTP/1.1、Host: ...、Cookie: session=... 这些明文——HTTP 的一切（包括 Cookie、密码）在网线上是裸奔的。抓 443 加 -X，你只看到一堆十六进制乱码，因为 TLS 加密了——这正是第 4 节要破的局。
常用选项速查
  • -i 网卡 · -n 不反解 DNS · -nn 连端口名也不解（443 不显示成 https）
  • -c N 抓 N 个包就停 · -s 0 抓完整包（老版本默认截断）· -e 显示以太网头（看 MAC，查 ARP 欺骗用）
  • -v / -vv 更详细 · -A ASCII · -X hex+ASCII · -w 存盘 · -r 读盘
  • 记忆要点：捕获过滤（BPF，在内核，省资源）和显示过滤（Wireshark 里，事后筛）是两套不同语法，别混。

3. Wireshark：显示过滤与"追踪 TCP 流"

tcpdump 适合服务器和快速过滤，但要看一段完整对话、把几百个乱序的包拼成"人话"，图形化的 Wireshark 无可替代。安装：mac 上 brew install --cask wireshark，Linux apt install wireshark。它和 tcpdump 共享 libpcap，所以 tcpdump -w 出来的 pcap 可以直接拖进 Wireshark。

捕获过滤 vs 显示过滤——必须分清的两套语法

捕获过滤 Capture Filter显示过滤 Display Filter
语法BPF（同 tcpdump）Wireshark 专属（点分字段名）
时机抓之前，决定抓不抓抓之后，决定显不显示
例子tcp port 443tcp.port == 443
丢弃的数据不符合的永久丢弃只是隐藏，随时改条件

显示过滤是 Wireshark 的灵魂，它能精确到协议的每一个字段。几个高频例子：

http                          # 只看 HTTP
http.request.method == "POST" # 只看 POST 请求
tls.handshake.type == 1       # 只看 TLS ClientHello（握手开始）
dns.qry.name contains "google"# DNS 查询里含 google 的
ip.addr == 192.168.1.10       # 某 IP 的所有流量（src 或 dst）
tcp.flags.syn == 1 && tcp.flags.ack == 0   # 纯 SYN（新连接发起）
tcp.analysis.retransmission   # Wireshark 帮你标出的重传包（网络差的信号）
http.cookie || http.set_cookie # 抓 Cookie
小技巧：在包详情面板里随便点一个字段（比如某个 Cookie 值），右键 → "Apply as Filter" → "Selected"，Wireshark 自动帮你写好显示过滤表达式。不用背字段名，点出来即可。

追踪 TCP 流（Follow TCP Stream）——把碎片拼成对话

一次 HTTP 下载可能拆成几十个 TCP 段，在包列表里散落各处。右键任一包 → Follow → TCP Stream，Wireshark 会按 seq 号把同一条 TCP 连接的所有 payload 重组，用两种颜色（红=客户端发的，蓝=服务器发的）拼成完整对话。

预期：对一次 HTTP 请求做 Follow TCP Stream，你会看到一整段：
GET /search?q=secret HTTP/1.1（红）
Host: example.com
Cookie: SESSIONID=abc123...
↓
HTTP/1.1 200 OK（蓝）
……完整的网页 HTML。这就是为什么公共 WiFi 上用 HTTP 等于把账号密码喊出来。

对应的兄弟功能：Follow → TLS Stream（解密后，见第 4 节）、Follow → HTTP/2 Stream、Follow → QUIC Stream。命令行也有等价物——tshark 是 Wireshark 的 CLI 版：

# tshark = 命令行 Wireshark，可在服务器上跑显示过滤
tshark -r capture.pcap -Y 'http.request' -T fields -e http.host -e http.request.uri
# -Y 显示过滤，-T fields 只输出指定字段，做日志分析很顺手

4. 为什么 HTTPS 抓到也是乱码 + SSLKEYLOGFILE 解密

到这里你应该已经实验出那个让人困惑的现象：抓 80 端口什么都看得见，抓 443 端口全是乱码。原因你 Day 18 学过——TLS 在 TCP 之上加了一层加密，应用数据（HTTP）在离开你电脑前就被对称密钥加密成了密文。tcpdump/Wireshark 抓到的是网线上的密文，它们没有密钥，自然解不开。

那 HTTPS 是不是就绝对抓不到明文了？不是。密钥就在你自己的浏览器进程内存里——只要能把那把会话密钥（session key）导出来喂给 Wireshark，它就能解密。这正是 SSLKEYLOGFILE 的作用。

原理：会话密钥日志

Chrome、Firefox、curl 等基于 NSS/BoringSSL/OpenSSL 的程序，都支持一个约定：如果环境变量 SSLKEYLOGFILE 指向一个文件，它们会在每次 TLS 握手后，把本次连接协商出的主密钥（master secret）以标准格式追加写进去。Wireshark 读这个文件，就能对相应连接解密。

注意：这只能解密你自己机器上、你自己启动的那些进程的流量——因为密钥来自你自己的内存。你不能用它解密别人的 HTTPS。这是合法调试手段，不是攻击。（TLS 1.3 用了前向保密 ECDHE，光抓服务器私钥也解不了历史流量，所以 SSLKEYLOGFILE 这种导会话密钥的方式才是现在唯一的本机调试途径。）

# 1) 指定密钥日志文件并启动浏览器（mac，以 Chrome 为例）
export SSLKEYLOGFILE=$HOME/tls-keys.log
open -a "Google Chrome"        # 必须从这个 shell 启动，才能继承环境变量
#   Linux:  SSLKEYLOGFILE=~/tls-keys.log google-chrome &

# 2) 同时用 tcpdump 抓包
sudo tcpdump -i en0 -n -w /tmp/https.pcap 'tcp port 443'

# 3) 在浏览器里访问一个 https 站点，然后 Ctrl-C 停止抓包

# 4) curl 也支持，更适合做干净的实验：
SSLKEYLOGFILE=~/tls-keys.log curl -s https://example.com > /dev/null

然后在 Wireshark 里：Preferences → Protocols → TLS → "(Pre)-Master-Secret log filename" 填入 tls-keys.log 的路径。重新打开 pcap——奇迹发生：原本乱码的 443 流量，Wireshark 现在显示一个新的 Decrypted TLS 标签页，里面是明文 HTTP！对它做 Follow → HTTP Stream，你就看到了加密连接里的真实请求与响应。

预期：解密后，过滤 http 能直接看到 HTTPS 站点里的 GET /、响应头、甚至 JSON API 的 body。加密保护的是"传输中被第三方偷看",但对持有密钥的端点（也就是你自己）始终是透明的。这一点对理解后面所有 LLM API 流量调试至关重要。

5. 中间人攻击与 sslstrip：HTTPS 降级

法律与道德红线：本节及后两节的攻击技术只能在你拥有完全控制权的实验环境（自己的两三台虚拟机/树莓派组成的隔离网络）或书面授权的渗透测试中演示。在公共网络、公司网络、或对他人设备实施，等同于犯罪。我们学它，是因为不懂攻击就写不出防御——这是你做 Agent 安全的本职。

什么是中间人（MITM）

中间人攻击：攻击者把自己插到通信双方中间，让 A 以为在和 B 说话、B 以为在和 A 说话，实际所有流量都经过攻击者。在局域网里，最经典的做法是 ARP 欺骗（ARP spoofing）：

正常： 受害者 ──→ 网关(路由器) ──→ 互联网 ARP 欺骗后： 攻击者持续广播假 ARP："网关的 IP 对应的是我的 MAC" 受害者信了（ARP 没有认证）→ 把发往网关的包发给攻击者 攻击者转发给真网关 → 受害者无感知，但流量全被看到 受害者 ──→ 攻击者(中间人) ──→ 网关 ──→ 互联网 │ └─ 抓包 / 篡改 / 降级

ARP 欺骗之所以可行，是因为你 Day 15 学的 ARP 协议完全没有认证——谁吼一嗓子"这个 IP 是我"，别人就信。工具如 arpspoof（dsniff 套件）、bettercap 几条命令就能做到。一旦成为中间人，攻击者就能抓到受害者所有流量——但 HTTPS 还是密文。怎么办？这就引出 sslstrip。

sslstrip：把 HTTPS 偷偷换成 HTTP

sslstrip（Moxie Marlinspike 2009 年发布）的思路极其巧妙：它不去破解 TLS，而是不让 TLS 发生。很多人访问网站时输入的是 example.com 而非 https://example.com，浏览器先发 HTTP，服务器再用 301 重定向或 HSTS 把你升级到 HTTPS。sslstrip 作为中间人，拦下这个升级：

  1. 受害者→攻击者：明文 HTTP 请求 example.com
  2. 攻击者→服务器：攻击者自己用 HTTPS 去访问真站点（攻击者和服务器之间是加密的，正常）
  3. 服务器→攻击者：返回页面，里面所有 https:// 链接
  4. 攻击者把页面里的 https:// 全改写成 http:// 再发给受害者
  5. 结果：受害者全程用 HTTP 和攻击者通信，浏览器地址栏没有锁图标但很多人不看；攻击者却用 HTTPS 和真站点通信，两边都"正常工作"，密码、Cookie 全在受害者那一侧明文流过攻击者。

防御：HSTS 与 HSTS Preload

sslstrip 的克星是 HSTS（HTTP Strict Transport Security）。服务器返回响应头 Strict-Transport-Security: max-age=31536000; includeSubDomains，浏览器收到后会记住"这个域名以后只准用 HTTPS",哪怕用户敲 http，浏览器自己在本地就改成 https，根本不发明文请求——sslstrip 没东西可拦。更进一步，HSTS Preload List 把一批域名硬编码进浏览器源码，第一次访问就强制 HTTPS，连首次的明文窗口都堵死。你可以验证：

# 看一个站点是否启用 HSTS
curl -sI https://github.com | grep -i strict-transport
#   Strict-Transport-Security: max-age=31536000; includeSubdomains; preload

# Chrome 内部页面查看 preload 状态
#   浏览器地址栏访问： chrome://net-internals/#hsts
关键事实：sslstrip 攻击之所以现在威力大减，正是因为主流站点普遍上了 HSTS + Preload，且现代浏览器默认尝试 HTTPS。但没上 HSTS 的内部系统、老旧设备、自建服务仍是重灾区——这在企业内网渗透里非常常见。你做 Agent 安全时，自家的内部 API 务必上 HSTS。

6. DNS 投毒：把你导向假服务器

MITM 是物理上插到中间，DNS 投毒（DNS spoofing / cache poisoning）则是更隐蔽的"指错路":不拦截你的流量，而是在你问"bank.com 的 IP 是多少"时，抢先给你一个假 IP，让你自己心甘情愿地连到攻击者的服务器。

类比：DNS 是互联网的电话簿（Day 21 会深入）。DNS 投毒就是趁你查电话簿时，有人把"银行客服"那一行的号码偷偷换成了诈骗电话——你拨过去时一切正常，但接电话的是骗子。

两种投毒

类型攻击点原理
本地 DNS 欺骗同局域网（需先 MITM）攻击者看到你的 UDP 53 查询，抢在真 DNS 服务器之前回一个伪造响应。UDP 无连接、DNS 靠 16 位事务 ID 匹配，抢答只要 ID 对就被接受。
缓存投毒递归 DNS 服务器污染上游解析器的缓存，一次投毒影响所有用它的用户。经典的 Kaminsky 攻击（2008）靠暴力猜事务 ID + 抢答实现，催生了源端口随机化这一缓解措施。

为什么 DNS 这么好骗

  • UDP 无连接：没有握手，伪造一个响应包成本极低，不需要建立连接。
  • 仅靠 16 位事务 ID + 端口匹配：理论上 65536 种 ID，本地 MITM 下攻击者能直接看到 ID（抓包就行），命中率 100%；远程则靠猜（Kaminsky）。
  • 明文：传统 DNS（Do53）查询和响应全是明文，谁都能看你在查什么——既泄露隐私（你访问了哪些网站一览无余），又便于精准投毒。

本地实验里，dnsspoof（dsniff）或 bettercap 的 dns.spoof 模块几行配置就能把任意域名解析到你指定的 IP。

观察你自己的 DNS（无害实验）

# 抓自己机器的 DNS 查询，看看明文长什么样
sudo tcpdump -i en0 -n -A 'udp port 53'
#   另开终端访问几个网站，回来看抓到的查询/响应，域名清清楚楚

# dig 看完整 DNS 响应（事务 ID、各段记录）
dig +noall +answer +stats www.example.com
#   注意输出里的 "id" 字段——那 16 位就是投毒要匹配的事务 ID
防御视角：DNS 投毒的根本治法是给响应加上认证或加密。DNSSEC 用数字签名保证响应"确实来自权威服务器且未被篡改"（但它不加密，旁观者仍看得到你查什么）；而 DoH/DoT（下一节）则把整条 DNS 通道加密，旁观者连你查什么都看不到，更无从抢答。两者解决的问题不同：DNSSEC 防篡改，DoH/DoT 防偷窥+防本地抢答。

7. DoH / DoT：给 DNS 也穿上 TLS

传统 DNS（Do53，即 53 端口的明文 UDP/TCP）的两大原罪——明文可窥探、易被抢答投毒——的现代答案是把 DNS 装进 TLS 隧道。两种主流方案：

DoT (DNS over TLS)DoH (DNS over HTTPS)
端口853（专用）443（和普通 HTTPS 混在一起）
封装DNS 报文直接塞进 TLSDNS 报文塞进 HTTPS 请求体
可识别性专用端口，网络管理员能看出"这是加密 DNS"并可封混在 443 的海量 HTTPS 里，极难区分普通网页流量
典型使用者系统级解析器（Android 默认）浏览器内置（Firefox/Chrome）

两者都做到了：第三方（包括你的 ISP、同局域网的 MITM）既看不到你在查什么域名，也无法抢答伪造响应，因为通道有 TLS 的完整性和身份保护。这是隐私和抗投毒的巨大进步。

# 用 curl 直接打 DoH 端点（Cloudflare 的 1.1.1.1）
curl -sH 'accept: application/dns-json' \
  'https://1.1.1.1/dns-query?name=example.com&type=A' | python3 -m json.tool
#   返回 JSON 形式的解析结果，全程走 443 加密

# DoT 测试（需要 kdig，knot-dnsutils 提供；mac: brew install knot）
kdig -d @1.1.1.1 +tls-ca +tls-host=cloudflare-dns.com example.com
安全的双刃剑——这点对你做 Agent 安全极重要：DoH 把 DNS 藏进 443，对隐私是好事，对企业防御是噩梦。传统上安全团队靠监控 DNS 查询来发现恶意软件回连 C2（很多木马用 DNS 解析找主控）、发现数据外泄（DNS tunneling——把偷的数据编码进域名查询里偷偷送出去，你 Day 27 会专门学）。一旦终端走 DoH，这些查询全混进正常 HTTPS，传统 DNS 监控彻底失明。所以现代企业要么禁用终端自带 DoH、强制走自己的受控解析器，要么在 TLS 层做检测。攻击者也爱 DoH——用它做隐蔽 C2 信道。你设计 Agent 沙箱的出网策略时，必须把 DoH 这条隐蔽信道考虑进去。

8. 实战：抓一次 HTTPS 请求并完整解密

把今天的工具串成一条完整链路。目标：抓一次真实 HTTPS 请求，从密文一路解密到明文 HTTP，并定位其中每一层。建议在终端边敲边对照。

# ── 准备 ──
export SSLKEYLOGFILE=$HOME/lab-keys.log
rm -f $HOME/lab-keys.log /tmp/lab.pcap

# ── 1) 开抓包（后台），只抓与目标的 443 ──
sudo tcpdump -i en0 -n -s 0 -w /tmp/lab.pcap 'tcp port 443 and host example.com' &
TCPDUMP_PID=$!
sleep 2

# ── 2) 发一次会写密钥日志的 HTTPS 请求 ──
curl -sv https://example.com/ -o /dev/null
#   -v 会在 stderr 打印 TLS 握手摘要：协议版本、加密套件、证书链
#   你能看到 "* SSL connection using TLSv1.3 / TLS_AES_256_GCM_SHA384"

# ── 3) 停止抓包 ──
sleep 1
sudo kill $TCPDUMP_PID

# ── 4) 命令行先确认抓到了密文 ──
tcpdump -r /tmp/lab.pcap -n -c 10
#   能看到 SYN/SYN-ACK 握手 + 一堆 Application Data（密文）

# ── 5) 确认密钥日志已生成 ──
wc -l $HOME/lab-keys.log     # 应有若干行 CLIENT_TRAFFIC_SECRET / SERVER_... 之类

然后用 tshark 带上密钥文件直接在命令行解密（不用开 GUI）：

tshark -r /tmp/lab.pcap \
  -o tls.keylog_file:$HOME/lab-keys.log \
  -Y 'http' -V | head -60
#   -o 指定密钥日志，-Y http 过滤出解密后的 HTTP，-V 展开全字段
#   你会看到原本加密的 GET / HTTP/1.1、Host、User-Agent、响应 200 OK 全部明文
预期成果：你完成了一条从"网线上的乱码"到"可读 HTTP 报文"的完整解密链路。复盘时在脑中标出每一层：以太网帧 → IP 包（src/dst IP）→ TCP 段（端口 443、SYN/ACK 握手）→ TLS 记录（ClientHello/ServerHello/证书/Application Data）→ 解密后的 HTTP。这条链路就是你日后调试任何网络服务、排查任何"为什么连不上/为什么慢/这流量是不是恶意的"问题的标准心智模型。

与 AI / Agent 安全的连结

1. 调试 LLM 流式 API 必备 SSLKEYLOGFILE。 你的 Agent 调 OpenAI/Claude 这类 HTTPS API，出现"流式响应卡住""SSE 断流""超时重连"时，光看应用日志看不到传输层真相。用 SSLKEYLOGFILE + tcpdump 抓下来解密，你能精确看到是 TLS 握手慢、是服务器迟迟不发第一个 token（TTFT 问题）、还是 TCP 重传——把 Day 17 学的 SSE/HTTP 和今天的解密结合，是排查推理服务的硬功夫。

2. DoH 是 Agent 出网的隐蔽信道，必须纳入沙箱出网策略。 你给 Agent 配代码执行沙箱（Day 157 会深入沙箱设计）时，如果只在 DNS 层（53 端口）做出网管控，一个被 prompt injection 劫持的 Agent 完全可以用 DoH（443）绕过你的 DNS 监控，把窃取的数据或 C2 指令藏进正常 HTTPS 流量。设计出网白名单时，必须连 443 的目标域名/SNI 一起管，否则等于没管。

3. 抓包验证 Agent 工具调用没有"偷偷"外泄数据。 做 Agent 安全审计时，最硬的证据不是看代码，而是抓包：把 Agent 跑一遍，解密它所有出站 HTTPS，逐条核对——它是不是只连了它该连的 API？有没有把用户隐私数据发到第三方？有没有意料外的回连？这种"用网线上的真相核对代码声称"的方法，是发现供应链后门和数据外泄的金标准。

4. MITM 思路用于搭建 Agent 流量审计代理。 防御性地使用中间人技术：在 Agent 和外部世界之间放一个你控制的 TLS 解密代理（如 mitmproxy），所有出站请求都经过它解密、记录、按策略放行或拦截。这本质上是"善意的 sslstrip 反面"——你在自己授权的环境里做合法 MITM，给 Agent 的每一次外呼装上摄像头。理解了第 5 节的 MITM 原理，你才知道这种审计代理怎么搭、TLS 证书怎么处理、有哪些盲区（比如 Agent 若做了证书固定 pinning 就会拒绝你的代理）。

5. DNS 投毒 = 针对 Agent 的"工具地址劫持"。 你的 Agent 通过域名访问工具服务（向量库、检索 API、MCP server）。如果它所在网络的 DNS 被投毒，Agent 会把请求连同凭据一股脑发给攻击者的假服务器，而它自己毫无察觉——这是 indirect injection（Day 132 重点）之外另一条供应链攻击面。防御就是给 Agent 的关键依赖上 DoT/DoH + 证书校验，并在审计代理里固定关键服务的 IP/证书指纹。
