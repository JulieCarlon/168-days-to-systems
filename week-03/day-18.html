1. 对称 vs 非对称：为什么必须两个一起用

加密分两大流派，理解它们的区别是理解 TLS 的前提。

对称加密（symmetric）：加密和解密用同一把钥匙。就像家门钥匙——同一把锁门、开门。代表算法 AES、ChaCha20。优点是极快，现代 CPU 有 AES-NI 硬件指令，每秒能加密几个 GB。缺点是个致命的"先有鸡还是先有蛋"问题：你怎么把这把钥匙安全地交给对方？如果通过网络发，中间人就能截获，那加密就白做了。

非对称加密（asymmetric / 公钥加密）：有一对钥匙——公钥（public key）和私钥（private key）。公钥加密的只有私钥能解，反之亦然。代表算法 RSA、ECDSA、Ed25519。神奇之处：公钥可以大大方方公开，谁都能用你的公钥给你加密，但只有持有私钥的你能解密。这就解决了钥匙分发问题。缺点是慢——RSA 加密比 AES 慢几百上千倍，不可能用它加密整个 HTTP 流量。

类比：非对称加密像一个"任何人都能投信、只有你有钥匙开"的邮筒；对称加密像两人共享的密码本，谁有都能读写。

TLS 的核心智慧：用慢但能安全分发钥匙的非对称加密，去传递一把快但需要安全分发的对称密钥。握手阶段用非对称协商出一个临时对称密钥（叫 session key / 会话密钥），之后所有数据用这把对称密钥加密。鱼和熊掌兼得。

还有第三个角色：密钥交换（key exchange）。现代 TLS 不再用"公钥直接加密会话密钥"（那样私钥泄露后历史流量全完蛋），而是用 Diffie-Hellman（DH / ECDHE）算法。它的魔法是：双方各自生成临时随机数，通过几次数学运算，在公开信道上各自算出同一个共享秘密，而窃听者即使看到所有交换的消息也算不出来。每次连接用全新的临时密钥，这个特性叫前向保密（Forward Secrecy）——就算服务器私钥未来泄露，攻击者也解不开之前录下的流量。

2. PKI 与证书链：信任的传递

非对称加密解决了"怎么安全传钥匙"，但留下一个新问题：你拿到的这个"bank.com 的公钥"，凭什么相信它真是 bank.com 的，而不是中间人换上去的？这就是PKI（Public Key Infrastructure，公钥基础设施）要解决的"身份认证"问题。

解法是引入可信第三方——CA（Certificate Authority，证书颁发机构）。逻辑像现实里的"公证处"：

  1. bank.com 生成自己的公私钥对，把公钥 + 域名信息打包成"证书签名请求（CSR）"交给 CA。
  2. CA 验证 bank.com 确实拥有这个域名（DV 证书查 DNS / 邮箱即可），然后用 CA 自己的私钥给这份证书签名。
  3. 浏览器/操作系统出厂时内置了一批受信任 CA 的公钥（叫根证书 / Root CA store）。
  4. 你连接 bank.com 时，它发来证书，浏览器用内置的 CA 公钥验证签名——签名对得上，就证明"一个我信任的 CA 担保了这个公钥属于 bank.com"。

但 CA 不会拿宝贵的根私钥直接签每张证书（根私钥一旦泄露整个体系崩塌，所以离线保存）。实际是证书链（chain of trust）：

Root CA（根证书，离线，浏览器内置） │ 签名 ▼ Intermediate CA（中间证书，在线签发） │ 签名 ▼ bank.com 的叶子证书（leaf / end-entity）

浏览器验证时逐级往上验签：叶子证书由中间 CA 签 → 中间 CA 由根 CA 签 → 根 CA 在我的信任库里。只要链条能一路连到一个我信任的根，整条链就可信。服务器有义务把"叶子 + 中间"证书都发给你（漏发中间证书是最常见的部署错误，表现为"部分浏览器报错、部分正常"——因为有些客户端缓存了中间证书）。

信任也能被撤销：如果私钥泄露，CA 通过 CRL（撤销列表）或 OCSP（在线状态查询）宣布某证书作废。现代多用 OCSP Stapling——服务器自己定期取 OCSP 应答并在握手时一并发给你，省去浏览器额外查询、也保护隐私。

3. X.509：证书里到底装了什么

X.509 是证书的标准格式。一张证书本质是"一段结构化数据 + CA 对这段数据的签名"。用 openssl 拆开看：

# 从一个网站抓证书并打印明文
echo | openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | openssl x509 -noout -text

# 只看关键字段
echo | openssl s_client -connect example.com:443 2>/dev/null \
  | openssl x509 -noout -subject -issuer -dates

关键字段含义：

字段含义
Subject证书主体（给谁的）。CN=example.com
Issuer签发者（哪个 CA 签的）。自签证书里 Subject==Issuer
Validity (Not Before / Not After)有效期。过期证书浏览器直接拒绝
Subject Alternative Name (SAN)真正生效的域名列表。现代浏览器只看 SAN，不再看 CN！一张证书可覆盖 example.com + *.example.com
Public Key主体的公钥（RSA 2048/4096 或 EC P-256）
Signature AlgorithmCA 用什么算法签的，如 sha256WithRSAEncryption
Key Usage / Extended Key Usage这把钥匙能干嘛（数字签名 / 服务端认证 / 客户端认证）

证书文件常见两种编码：PEM（Base64 文本，-----BEGIN CERTIFICATE----- 开头，最常见）和 DER（二进制）。一个 .pem 文件里可以拼接多张证书形成证书链。

按验证强度，证书分三档：DV（域名验证）只证明你控制这个域名（Let's Encrypt 免费的就是 DV）；OV（组织验证）验证公司真实存在；EV（扩展验证）最严格。但 2019 年后浏览器取消了 EV 的"绿色公司名"特殊显示——对普通用户来说三者地址栏看起来一样，所以现在绝大多数网站用免费 DV 就够。

4. TLS 1.2 握手：一步步建立加密通道

把前面的零件组装起来，看一次完整的 TLS 1.2 握手（HTTP/2 之前的主流，至今仍大量在用）。它需要 2 个 RTT（往返）：

客户端 服务器 │ │ │──ClientHello────────────────────────────►│ 我支持这些 TLS 版本/加密套件 │ (随机数 client_random, 支持的 cipher) │ + SNI（要访问哪个域名） │ │ │◄─────────────────────ServerHello─────────│ 选定版本/套件 + server_random │◄─────────────────────Certificate─────────│ 我的证书链 │◄──────────────ServerKeyExchange──────────│ DH 参数（ECDHE 用） │◄──────────────────ServerHelloDone────────│ │ │ │──ClientKeyExchange───────────────────────►│ 我的 DH 公开值 │──ChangeCipherSpec────────────────────────►│ 之后开始加密 │──Finished (加密)─────────────────────────►│ │ │ │◄──────────────────ChangeCipherSpec───────│ │◄────────────────────Finished (加密)──────│ │ │ │═══════ 之后全是对称加密的应用数据 ═════════│

逐步拆解发生了什么：

  1. ClientHello：客户端说"我支持 TLS 1.2/1.3、这些加密套件、这是我的随机数 client_random"，并通过 SNI 扩展告诉服务器要访问哪个域名。
  2. ServerHello + Certificate：服务器从客户端列表里挑一个双方都支持的加密套件（cipher suite），发回 server_random 和自己的证书链。客户端此时验证证书（第 2、3 节讲的链验证）。
  3. 密钥交换：双方用 ECDHE 交换 DH 公开值，各自算出预主密钥（pre-master secret），再结合两个随机数派生出最终的对称会话密钥。这个会话密钥从未在网络上传输过。
  4. Finished：双方各发一个用会话密钥加密的 Finished 消息，内容是"到目前为止所有握手消息的哈希"。如果中间人篡改过任何握手消息，这个哈希对不上，握手失败——这保证了握手过程本身的完整性。

一个 cipher suite 名字编码了所有选择，例如 TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 拆开是：密钥交换 ECDHE + 身份认证 RSA + 对称加密 AES_128_GCM + 哈希 SHA256。

5. TLS 1.3：更快、更安全、砍掉的东西

2018 年 TLS 1.3 发布，是十年来最大的升级。两个关键词：更快和更安全。

更快：1-RTT 握手 + 0-RTT 恢复

TLS 1.2 要 2 个 RTT 才能开始传数据。TLS 1.3 把流程压缩到 1 个 RTT：客户端在 ClientHello 里直接附带它猜测服务器会用的 DH 公开值（key_share），服务器一次回复就能完成密钥协商。对恢复的连接更进一步——0-RTT：客户端可以在第一个包里就带应用数据（用上次会话缓存的密钥）。这就是为什么 HTTP/3（基于 QUIC，内置 TLS 1.3）连接这么快。

TLS 1.2（2-RTT） TLS 1.3（1-RTT） Client → Hello Client → Hello + key_share Server → Hello,Cert,... Server → Hello,key_share,Cert,Finished Client → KeyExchange Client → Finished + 应用数据 Server → Finished （已经可以传数据了） Client → 应用数据

更安全：做减法

TLS 1.3 最大的安全提升来自删除不安全的旧选项——历史上 TLS 漏洞大多源于"为了兼容保留的弱算法"：

  • 删掉 RSA 密钥交换：强制使用 (EC)DHE，所有连接默认前向保密。
  • 删掉 CBC 模式、RC4、3DES、MD5、SHA-1：只保留 AEAD 认证加密（AES-GCM、ChaCha20-Poly1305），从根上消灭 BEAST、Lucky13、POODLE 这类针对 CBC 的攻击。
  • 删掉压缩：消灭 CRIME 压缩侧信道。
  • 删掉重协商（renegotiation）：消灭一类中间人注入攻击。
  • 加密更多握手信息：ServerHello 之后的证书等消息都加密了（1.2 里证书是明文的）。

到 2026 年，TLS 1.3 已是主流，TLS 1.0/1.1 被所有主流浏览器废弃，PCI-DSS 等合规标准也要求至少 TLS 1.2。部署 LLM 服务时，应当只开 TLS 1.2 + 1.3，关闭一切更老版本。

6. SNI 与 ALPN：握手里的两个旁路信息

ClientHello 里有两个常被忽视、但对运维和安全都关键的扩展。

SNI（Server Name Indication）

一台服务器（一个 IP）上往往托管几百个域名（虚拟主机）。但 TLS 握手在 HTTP 之前发生——服务器还没看到 HTTP 的 Host header，怎么知道该返回哪个域名的证书？答案是 SNI：客户端在 ClientHello 里明文写上要访问的域名，服务器据此选证书。

问题就在"明文"二字。即使整个 TLS 加密了内容，SNI 仍然暴露了你访问的是哪个网站——运营商、防火墙、网络监控者都能看到。这是 GFW 和企业网络做"按域名封锁"的主要手段之一。

解决方案 ECH（Encrypted Client Hello）正在推广——把 SNI 也加密，但部署依赖 DNS 配合且尚未普及。在 2026 年，SNI 明文泄露仍是现实，做隐私敏感的 Agent 服务要意识到这点。

# 观察 SNI 明文：抓握手包能直接看到域名
sudo tcpdump -nn -i any -A 'tcp port 443' | grep -a -i "example"

# 用 openssl 指定 SNI（多域名服务器必须指定，否则可能拿到默认证书）
openssl s_client -connect 1.2.3.4:443 -servername api.example.com

ALPN（Application-Layer Protocol Negotiation）

ALPN 让客户端和服务器在 TLS 握手时顺便协商上层用什么协议，省去一次额外往返。客户端在 ClientHello 里列出支持的协议标识，服务器选一个：

  • h2 → HTTP/2
  • http/1.1 → HTTP/1.1
  • h3 → HTTP/3（在 QUIC 上）

这就是昨天讲的"浏览器只在 TLS 上跑 HTTP/2"的实现机制——没有 ALPN 协商出 h2，就回退到 HTTP/1.1。gRPC 也靠 ALPN 协商 h2。

7. mTLS：双向认证与服务身份

普通 HTTPS 是单向认证：客户端验证服务器（"你真是 bank.com 吗"），但服务器不验证客户端是谁（客户端身份靠后续的 cookie/token）。mTLS（mutual TLS，双向 TLS）则要求双方都出示证书——服务器在握手时额外发一个 CertificateRequest，客户端也必须提供自己的证书并证明持有对应私钥。

单向 TLS（浏览器场景） mTLS（服务间场景） Client 验证 Server Client 验证 Server Server 不验 Client → Server 也验证 Client 证书 （身份靠 token/cookie） （身份就是证书本身）

mTLS 的核心价值是把"网络身份"绑定到密码学证书上，而不是可被盗用的密码或 token。典型应用场景：

  • 微服务间通信：服务 A 调服务 B，双方用证书互认，确保不是冒充的服务在调用。Service Mesh（Istio、Linkerd）默认给每个 Pod 注入证书、自动做 mTLS。
  • 零信任网络：不再假设"内网就安全"，每次调用都验证身份（这是 Week 16 的零信任专题主线）。
  • 高安全 API：银行、支付的对接常要求客户端证书。
# 自己做一个最小 mTLS 实验：先建一个 CA
openssl genrsa -out ca.key 4096
openssl req -x509 -new -key ca.key -days 365 -out ca.crt -subj "/CN=MyTestCA"

# 给服务器签证书
openssl genrsa -out server.key 2048
openssl req -new -key server.key -out server.csr -subj "/CN=localhost"
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 365

# 给客户端签证书（同样用这个 CA）
openssl genrsa -out client.key 2048
openssl req -new -key client.key -out client.csr -subj "/CN=client-A"
openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out client.crt -days 365

# 启动要求客户端证书的 TLS 服务
openssl s_server -accept 8443 -cert server.crt -key server.key -CAfile ca.crt -Verify 1

# 客户端必须带证书才能连上
openssl s_client -connect localhost:8443 -cert client.crt -key client.key -CAfile ca.crt

注意 mTLS 的运维成本：证书轮转是大难题——客户端证书会过期，几百个服务的证书手工换是灾难。这正是 Istio/SPIFFE 这类系统自动签发短期证书（几小时一换）的意义所在。

8. 经典漏洞：TLS 三十年的伤疤

理解漏洞才能理解 TLS 为什么长成今天这样。下面这些都是真实震动业界的事件：

漏洞攻击的是什么教训 / 缓解
Heartbleed (2014)OpenSSL 心跳扩展的越界读，能泄露服务器内存（含私钥、session）实现 bug，非协议 bug。私钥可能泄露 → 必须吊销重发证书
POODLE (2014)逼迫降级到 SSLv3，利用 CBC padding 漏洞解密彻底禁用 SSLv3；TLS 1.3 删掉 CBC
BEAST / Lucky13CBC 模式的 padding / 时间侧信道用 AEAD（GCM）替代 CBC
CRIME / BREACH压缩 + 加密，通过密文长度推断 cookie 内容TLS 1.3 删掉 TLS 压缩；HTTP 压缩需谨慎
FREAK / Logjam逼迫降级到出口级弱 RSA/DH（512 位）后暴力破解禁用所有弱密钥长度；最小 2048-bit RSA / 256-bit EC
降级攻击 (downgrade)中间人篡改 ClientHello 让双方用更弱的版本TLS 1.3 用特殊随机数标记 + Finished 哈希检测降级

提炼出一条贯穿性的规律：几乎所有 TLS 漏洞要么来自"为兼容而保留的弱算法"，要么来自"实现 bug"，而不是核心密码学被破解。这就是 TLS 1.3 激进做减法的根本动机——能删的弱选项全删，把攻击面压到最小。

中间人攻击（MITM）到底攻击哪一步？核心永远是证书验证。如果你能让客户端相信一个伪造的证书（比如往受害者机器装一个恶意根 CA，或客户端代码图省事写了 verify=False），中间人就能解密一切。所以"是否严格校验证书链"是 TLS 安全的命门——下面练习和连结会反复强调。

🔗 与 AI / Agent 安全的连结

1. verify=False 是 LLM 工具调用的头号隐患。Agent 经常需要去调外部 HTTP API、抓网页、连数据库。开发者为了"先跑通"经常写 requests.get(url, verify=False) 或 httpx.Client(verify=False)，这等于关掉了证书验证，把整条链路暴露给中间人。一个被劫持的 API 响应可以注入恶意指令（间接 prompt injection）。生产 Agent 必须强制证书校验，对自建服务用自己的 CA bundle 而非关验证。

2. 自建 CA 是内网 Agent 服务的正确姿势。你的 LLM 推理服务（vLLM）、向量库、网关在 K8s 内网互调，用公网 CA 签证书既贵又不必要。正确做法是建一个内部 CA、给每个服务签证书、客户端信任这个内部 CA——既加密又认证，还能上 mTLS 做服务身份。这正是第 7 节那套 openssl 命令的实战用途，也是 Istio 自动帮你做的事。

3. SNI 泄露暴露你的 Agent 在调用哪些模型/服务。如果你的 Agent 调用 api.openai.com、api.anthropic.com 或某个特定向量服务，即使流量加密，SNI 明文也会让网络监控方知道你在用哪些服务——这对合规审计是把双刃剑（既能监控也会泄密）。隐私敏感场景需关注 ECH 进展或走专线/代理。

4. 证书过期会让整个 Agent 链路雪崩。mTLS 短期证书、Let's Encrypt 90 天证书，一旦轮转脚本挂了，到期当天所有服务调用集体 TLS 握手失败。这是真实发生过的大规模事故模式。LLM 服务的监控里必须有"证书剩余有效期"告警（用 openssl x509 -enddate 配合 Prometheus exporter）。

5. 模型权重/数据传输的前向保密很重要。如果攻击者今天录下了你传输加密模型权重或敏感训练数据的流量，未来某天拿到服务器私钥，有前向保密（ECDHE）就解不开，没有就全泄。这就是为什么 TLS 1.3 强制 ECDHE 是对长期机密数据的关键保护——传大文件（checkpoint、数据集）务必确认走的是支持前向保密的套件。
