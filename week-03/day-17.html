1. UDP：极简主义传输层

UDP（User Datagram Protocol，用户数据报协议）是 TCP 在传输层的同伴，但设计哲学完全相反。如果说 TCP 是"我把所有脏活累活全包了、给你一条干净流"，UDP 就是"我什么都不管，你自己看着办"。它的头只有 8 字节，是网络协议里最简洁的之一。

0 7 8 15 16 23 24 31 +--------+--------+--------+--------+ | Source | Destination | | Port | Port | +--------+--------+--------+--------+ | Length | Checksum | +--------+--------+--------+--------+ | Data (可选) | +-----------------------------------+

四个字段就完了：源端口、目的端口、长度、校验和。对比 TCP 的 20+ 字节、加上 11 个状态、加上拥塞控制和滑动窗口——UDP 简直简陋。UDP 不做的事情比它做的事情多得多：

  • 不建立连接——发就发，没有握手
  • 不保证到达——丢了就丢了，不重传
  • 不保证顺序——后发的可能先到
  • 不去重——同一个包到达两次你会收到两次
  • 不做拥塞控制——你拼命发，网络拥塞了也是你自己的问题
  • 不做流量控制——发再快也不管对方接收缓冲是不是满了

听起来 UDP 是个"残废协议"，那为什么还有人用它？因为它没有 TCP 的开销，更没有 TCP 的"队头阻塞"。在某些场景下，TCP 的"可靠保证"反而是负担：

场景为什么用 UDP
DNS 查询请求/响应都小（一个 UDP 包就够），重传由应用层做，省去 TCP 三次握手
音视频实时通话丢一个包不如延迟一个包。WebRTC、SIP、RTP 全是 UDP
在线游戏玩家位置 100ms 后过时了，丢了也不需要重传——下一个状态包就来了
NTP 时间同步请求/响应模式，应用层自己处理重试
QUIC / HTTP/3在 UDP 之上自己实现一个比 TCP 更好的可靠传输（后面详讲）
SNMP / syslog监控指标，丢一个无所谓

一个细节常被忽略：UDP 没有 MTU 自动分片。一个 IP 包超过 MTU（典型 1500 字节）需要 IP 层分片，分片中丢任何一片整个 UDP 数据报都失效。所以应用层用 UDP 时通常把包控制在 1200-1400 字节以内——QUIC 也遵守这个上限。

2. HTTP/1.0 与 1.1：keep-alive 救了世界

HTTP（HyperText Transfer Protocol）1990 年代由 Tim Berners-Lee 设计时极其简单：客户端发"GET /page"，服务端返 HTML，TCP 关闭。下一个请求重开 TCP。这就是 HTTP/0.9 和 1.0 的本来面目。

问题：一个网页有几十个图片、CSS、JS。1996 年浏览器加载一个页面要打开几十个 TCP 连接、每个都要三次握手 + TLS（如果有的话），总延迟里 TCP 握手能占一半以上。当时浏览器还有"并发 6 个 TCP"的硬限制，剩下的资源排队等。

HTTP/1.1 的关键改进：keep-alive

1997 年 HTTP/1.1 把 persistent connection（持久连接）设为默认。一个 TCP 连接可以连续发多个 HTTP 请求/响应，省下重复握手成本。客户端发：

GET /index.html HTTP/1.1
Host: example.com
Connection: keep-alive

Server 答复后不关 TCP，客户端可以继续在这个连接上发下一个请求。直到任意一方发 Connection: close 或者 idle 超时（Nginx 默认 75s）才关。

HTTP/1.1 的另一个尝试：pipelining

持久连接还是串行——必须等上一个响应回来才能发下一个请求。HTTP/1.1 提了 pipelining：客户端连续发好几个请求，服务端按顺序响应。理论上很好，实践中是灾难：

  • 必须严格按发送顺序响应——后面快的请求必须等前面慢的（队头阻塞 head-of-line blocking）
  • 中间代理、负载均衡常常不支持，会乱序或者断连

结果 pipelining 在主流浏览器里几乎全部默认关闭。所以"一个 HTTP/1.1 连接上同一时刻只能跑一个请求"是事实上的限制，浏览器只能开 6 个并发连接绕过这个限制。

分块传输 Chunked Transfer-Encoding

HTTP/1.1 引入了 chunked encoding：响应不需要预先知道总长度，可以一段一段发，每段前缀长度。这是后来 SSE（Server-Sent Events）和 LLM 流式输出的协议基础——你的 ChatGPT 流式响应底层就是 chunked HTTP/1.1 或 HTTP/2 data frames。

HTTP/1.1 200 OK
Content-Type: text/event-stream
Transfer-Encoding: chunked

1a
data: {"token":"hello"}

1c
data: {"token":" world"}

0

3. HTTP 方法、状态码、关键 Header

方法（Methods / Verbs）

方法语义幂等？典型用途
GET读取资源是查询，可被代理缓存
POST创建 / 非幂等操作否新建资源、登录、触发动作
PUT整体替换是"用这份完整数据覆盖资源 X"
PATCH部分修改否（视实现）"修改 X 的某些字段"
DELETE删除是"删掉资源 X"
HEAD只要响应头是探活、看大小
OPTIONS询问支持的方法是CORS 预检
CONNECT建立隧道—HTTPS 代理

幂等性是个被忽视的概念：同样的请求做 N 次和做 1 次效果一样。客户端在网络抖动时可以放心重试幂等请求，但不能重试 POST——否则可能创建两条记录。所以"用 POST 还是 PUT"不是品味问题，是语义问题。

状态码大类

  • 1xx 信息：100 Continue、101 Switching Protocols（WebSocket、HTTP/2 升级）
  • 2xx 成功：200 OK、201 Created、204 No Content、206 Partial（断点续传）
  • 3xx 重定向：301 Moved Permanently、302 Found、304 Not Modified（条件请求命中缓存）、307/308（保持方法的重定向）
  • 4xx 客户端错误：400 Bad Request、401 Unauthorized（没认证）、403 Forbidden（认了但没权）、404、409 Conflict、429 Too Many Requests（限流）
  • 5xx 服务端错误：500 Internal、502 Bad Gateway、503 Service Unavailable、504 Gateway Timeout

常被搞混：401 是"你没告诉我你是谁"、403 是"我知道你是谁但你没权限"。生产中很多 API 把所有失败都返 500 是设计糟糕——客户端没法区分该重试还是该改请求。

关键 Header（必须熟）

  • Host：HTTP/1.1 强制要求——一个 IP 上跑多个域名（虚拟主机）就靠它
  • Content-Type + Content-Length：消息体的类型和长度
  • Accept / Accept-Encoding / Accept-Language：内容协商
  • Authorization：Bearer <token>（OAuth/JWT）、Basic base64(user:pass)
  • Cookie / Set-Cookie：会话状态。属性 HttpOnly 防 JS 读取、Secure 只在 HTTPS 传、SameSite 防 CSRF
  • Cache-Control / ETag / If-None-Match / Last-Modified：HTTP 缓存的核心
  • X-Forwarded-For / X-Real-IP：经反代后保留客户端真实 IP（注意可伪造，要看是不是从可信代理来的）
  • Strict-Transport-Security（HSTS）、Content-Security-Policy（CSP）：Web 安全 header

4. HTTP/2：二进制分帧 + 多路复用 + HPACK

2015 年 HTTP/2 发布（基于 Google 的 SPDY）。它没有改变 HTTP 语义（方法、状态码、Header 都一样），只重写了传输方式。核心三件套：

1. 二进制分帧

HTTP/1.1 是纯文本协议，解析要找 \r\n、要处理大小写——慢且容易出 bug。HTTP/2 改成二进制分帧（frame），每个 frame 有固定头：

+-----------------------------------------------+ | Length (24) | +---------------+---------------+---------------+ | Type (8) | Flags (8) | +---+-----------+---------------+---------------+ |R| Stream Identifier (31) | +---+-------------------------------------------+ | Frame Payload | +-----------------------------------------------+

关键字段 Stream Identifier——这是多路复用的根基。一条 TCP 连接里可以并行有几百个 stream（请求/响应对），frame 之间可以任意交错，靠 stream id 区分。

2. 多路复用（multiplexing）

HTTP/1.1 在一个 TCP 上同一时刻只能跑一个请求；HTTP/2 在一个 TCP 上可以同时跑 N 个请求，frame 交错传输。一个网页几十个资源用 1 个 TCP 就能并行拉完——握手成本被均摊到几乎为零。

HTTP/1.1 HTTP/2 单连接串行 单连接多路复用 req1 ────► req1 ───[frame]──► ◄──── resp1 ◄──[frame]── req2 ────► req2 ───[frame]──► ◄──── resp2 req3 ───[frame]──► ◄──[frame]── resp1 ◄──[frame]── resp3 ◄──[frame]── resp2

注意：HTTP/2 解决了 HTTP 层的队头阻塞，但 TCP 层的队头阻塞还在——任何一个 frame 在 TCP 流里丢了，整个 TCP 流都得等重传，所有 stream 都被卡。HTTP/3 就是为了解决这个才换到 UDP。

3. HPACK 头部压缩

HTTP 请求里 Header 经常比 body 还大（一堆 cookie、user-agent）。HPACK 用两招压：

  • 静态表 + 动态表：常见 header 名（:method、:path、user-agent）有固定 id，传 id 不传字符串；动态表为连接周期内的重复 header 建索引
  • Huffman 编码：剩下的字符串用 Huffman 压

HPACK 让请求 Header 经常从几 KB 压到几十字节。但它带来了安全问题：CRIME / BREACH 类压缩侧信道——攻击者通过观察密文长度推断 cookie。HTTP/3 的 QPACK 加了缓解措施，但根本风险仍在，所以对敏感 cookie 不要全靠压缩省带宽。

HTTP/2 的其他特性

  • Server Push：服务端预先推客户端"可能要"的资源。实践证明效果差、缓存难、bug 多，Chrome 已经在 2022 年默认禁用
  • 流优先级：客户端可以告诉服务端"先返 CSS 再返图"
  • 强制 TLS：标准上没强制，但所有主流浏览器只支持 HTTPS 上的 HTTP/2（h2 over TLS，对应 ALPN 标识 h2；明文是 h2c）

5. HTTP/3 与 QUIC：0-RTT 与抛弃 TCP

2022 年 HTTP/3 正式标准化。它最反直觉的一点：不跑在 TCP 上，跑在 UDP 上。为什么？因为 TCP 改不动了。

TCP 是内核协议，要给一个 TCP 加新特性意味着推全世界的 Linux/Windows/macOS 内核更新——10 年也推不完。中间盒（防火墙、NAT、运营商盒子）对 TCP 的"奇怪选项"经常直接丢弃。于是 Google 想了个办法：把传输层整个搬到用户态、跑在 UDP 上——这就是 QUIC。

HTTP/2 栈 HTTP/3 栈 ┌─────────┐ ┌─────────┐ │ HTTP/2 │ │ HTTP/3 │ ├─────────┤ ├─────────┤ │ TLS │ │ QUIC │ ← 用户态 ├─────────┤ │ (含 TLS │ 传输层 │ TCP │ │ 1.3) │ ├─────────┤ ├─────────┤ │ IP │ │ UDP │ └─────────┘ ├─────────┤ │ IP │ └─────────┘

QUIC 的几大特性

  1. 0-RTT / 1-RTT 握手：QUIC 把 TLS 1.3 集成在传输层，第一次连接 1 个 RTT就建立加密连接（vs TCP+TLS 的 2-3 RTT）；恢复连接 0-RTT——客户端可以在第一个包里就带应用数据。
  2. 真正的多路复用，没有 TCP 队头阻塞：QUIC 在用户态实现独立的 stream 重传，stream A 丢包不影响 stream B。
  3. 连接迁移（connection migration）：QUIC 用 Connection ID 而不是四元组标识连接。你从 WiFi 切到 4G，IP 变了——TCP 必须重连，QUIC 不用断。
  4. 抗中间盒：因为 QUIC 整个 payload 包括头大部分都加密，中间盒看不到细节，没法搞 "TCP 优化" 把协议玩坏。
  5. 用户态可迭代：拥塞控制可以选 BBR / Cubic / 自定义，发布新版本不用等内核。

HTTP/3 的现实

到 2026 年，全球约 30% 的网站启用了 HTTP/3，CDN（Cloudflare、Fastly、Akamai）全面支持。客户端通过 HTTP 响应 header Alt-Svc: h3=":443" 知道服务器支持 QUIC，下次主动用 UDP。一些企业网仍封 UDP 443，需要回退到 HTTP/2。

什么时候 HTTP/3 真的快：
1. 高丢包率链路（弱网/移动）——TCP 队头阻塞被消除，收益最大
2. 跨大洋长连接——0-RTT 恢复省一个 RTT
3. 多资源并发加载的网页
反之，单请求/低丢包内网场景，HTTP/3 因为用户态 UDP 的 CPU 开销比 TCP 高 30-50%，反而稍慢。

6. CORS：浏览器跨域的全景

CORS（Cross-Origin Resource Sharing）是 Web 安全里最常被骂、最常被误解的机制。要理解它，先理解它解决什么问题。

同源策略（Same-Origin Policy）

浏览器对 JS 的硬规则：脚本只能访问"同源"的资源，"同源"= 协议 + 域名 + 端口三者全相同。如果没这条规则，恶意网站 evil.com 的 JS 可以背着你向 bank.com 发请求，浏览器会自动带上你的 cookie——你银行账号就被搬空了。

CORS 是同源策略的"安全放宽"

但现代 Web 又需要跨域：前端在 app.example.com、API 在 api.example.com。CORS 提供一套机制让服务端"显式同意"某些跨域请求。

规则分两类：

  1. 简单请求（simple request）：GET/POST/HEAD + 仅安全 header + Content-Type 是 form/text 之一。浏览器直接发，但响应里必须有 Access-Control-Allow-Origin 才允许 JS 读取。
  2. 预检请求（preflighted）：所有其他请求（PUT/DELETE/PATCH、JSON Content-Type、自定义 header）。浏览器先发一个 OPTIONS 预检问："我能不能发这个请求？"
# 浏览器先发预检
OPTIONS /api/users HTTP/1.1
Origin: https://app.example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization

# 服务器同意
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400   # 预检结果可缓存的秒数

几个易踩的坑：

  • Access-Control-Allow-Origin: * + Access-Control-Allow-Credentials: true 是非法组合——浏览器会拒绝。带 credential 必须明确指定 origin。
  • 预检只能用 OPTIONS，不会带认证 cookie，不要在服务端的认证中间件里拒绝 OPTIONS。
  • CORS 是浏览器单方面的执行——curl、Python requests 完全不受 CORS 影响。它只防"浏览器里的恶意脚本"。
  • CORS 不能替代认证授权——它只是"允许跨域"，安全决策（这个 token 能不能改这条数据）还是要服务端检查。

🔗 与 AI / Agent 安全的连结

1. SSE 流式 LLM 接口跨 HTTP 版本的差异。你的 ChatGPT 风格流式响应（data: ...\n\n）在 HTTP/1.1 上每个流式连接独占一个 TCP（浏览器 6 个并发上限会卡）；在 HTTP/2 上多个流式响应共享 TCP，但任意一个 stream 因 TCP 重传卡住会拖累所有 stream；HTTP/3 上完全独立。如果你做面向移动端的 LLM 应用、用户在弱网，优先支持 HTTP/3 能显著降低 TTFT 和 token 之间的卡顿。

2. UDP 反射放大攻击是经典 DDoS 手法。攻击者伪造受害者 IP 给一堆 DNS/NTP/Memcached 服务器发小请求，服务器把巨大响应打到受害者那里——放大比可达几百倍。Memcached 反射攻击曾打出 1.7Tbps 的流量。对 Agent 服务这意味着：你自己的 UDP 服务（QUIC、DNS-over-UDP、自定义协议）必须做 amplification 防护——比如 QUIC 强制要求服务端首包不超过客户端首包的 3 倍。

3. CORS 错配是 LLM API 常见漏洞。很多团队上线 Agent 后端图省事开 Access-Control-Allow-Origin: *，前期没事——但当你加了 cookie 认证或 session 认证后，攻击者构造的恶意页面就可能借用户身份调用 Agent API。正确做法：白名单具体 origin、cookie 走 SameSite=Strict、敏感操作走 token + CSRF token 双验证。

4. HPACK / QPACK 侧信道泄露 prompt。如果你的 LLM 网关把用户 prompt 放进 HTTP header（少见但有），HPACK 的压缩会让密文长度暴露 prompt 内容——攻击者通过反复触发请求观察响应大小可以推测内容。对 LLM 流量，prompt 永远放 body，header 只放短的元数据。

5. HTTP/2 RST flood（CVE-2023-44487）是 2023 年震动业界的事件——攻击者快速打开/RST stream，让服务器陷入资源消耗。所有主流 web server（Nginx、Envoy、Go net/http）都中招。教训："多路复用 + 状态机"是攻击面富集地。部署 LLM 服务时，nginx/envoy 一定要打上对应补丁、配 http2_max_concurrent_streams、http2_max_requests。

# 看你 curl 支持哪些

310 分钟 · 思考题

场景：你做了一个 LLM Agent SaaS，前端在 app.youragent.com、API 在 api.youragent.com，用户登录后浏览器 cookie 里存 session。一位渗透测试报告说："你 API 的 CORS 配置是 Access-Control-Allow-Origin: *，存在风险"。

问题：（1）这个配置在你目前的架构下，能不能被利用？为什么？提示：跨域 + credentials 的浏览器限制。（2）正确的 CORS 配置应该是什么？要不要允许 Authorization header？要不要允许 cookie？（3）如果用户的 Agent 会调用第三方 webhook URL，怎么防 SSRF（这不是 CORS 能解决的）？
