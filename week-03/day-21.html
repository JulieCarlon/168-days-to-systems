1. DNS 协议：互联网的电话簿

互联网上一切通信最终都靠 IP 地址定位（回忆 Day 15：IP 是网络层的寻址单位）。但 142.250.196.78 这种数字人类记不住，所以我们用域名 www.google.com。DNS（Domain Name System，域名系统）就是把「人类可读的域名」翻译成「机器可用的 IP 地址」的分布式数据库。它生于 1983 年，至今仍是整个互联网最关键、也最脆弱的基础设施之一。

类比：DNS 像一本超大号电话簿，但没有任何一本完整的簿子存在某台机器上——它被切成无数小块，分散在全球数百万台服务器里，谁负责哪一块由「谁拥有这个域名」决定。你查一个号码时，不是翻一本书，而是层层问路：先问「.com 区的总机在哪」，再问「google.com 的负责人是谁」，最后问那个负责人「www 这台机器号码多少」。

域名的层级结构（从右往左读）

域名是倒序的树。最右边是根，越往左越具体：

. (根 root，隐藏的尾点) /|\ com org cn ... ← 顶级域 TLD (Top-Level Domain) / \ google openai ← 二级域（你花钱注册的部分） / \ \ www mail chat ← 子域 subdomain（自己随便建） 完整写法： www.google.com. ← 末尾那个点就是「根」，平时省略 └┬┘└──┬─┘└┬┘└┘ 主机 二级 TLD 根

这棵树的每一层都有人「负责」（authoritative，权威）。根服务器负责告诉你各个 TLD 的服务器在哪；TLD 服务器（如 .com 的）负责告诉你各二级域的权威服务器在哪；二级域的权威服务器（你买域名时配置的那台）才真正存着 www 对应的 IP。没有任何一台机器知道全部答案，但顺着树往下问，总能问到。

DNS 报文：跑在 UDP 53 上的二进制

DNS 默认用 UDP 端口 53（回忆 Day 17：UDP 无连接、无重传、开销小——非常适合「一问一答」的小查询）。当响应太大（超过 512 字节，或带 DNSSEC 签名）时，会回退到 TCP 53。报文结构：

+---------------------+ | Header | 12 字节：ID / 标志位 / 各区段计数 +---------------------+ | Question | 你问什么：域名 + 类型(A) + 类(IN) +---------------------+ | Answer | 权威/缓存的回答：一条条 RR (Resource Record) +---------------------+ | Authority | 负责该区的 NS 服务器 +---------------------+ | Additional | 附加信息(如 NS 的 IP，省一次查询) / EDNS +---------------------+

Header 里有几个关键字段值得记住：ID（16 位事务号，请求和响应靠它配对——后面讲投毒时是关键）、QR（query/response 标志）、RD（Recursion Desired，「请帮我递归查」）、RA（Recursion Available）、rcode（返回码，NOERROR/NXDOMAIN「域名不存在」/SERVFAIL）。

动手：看一眼真实报文

# dig 是 DNS 调试的瑞士军刀（macOS/Linux 自带）
dig www.google.com

# 输出关键部分：
# ;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 39021
# ;; flags: qr rd ra; QUERY: 1, ANSWER: 6, ...
# ;; ANSWER SECTION:
# www.google.com.   142  IN  A  142.250.196.68
#                   ↑TTL ↑类 ↑类型 ↑值

# +short 只看结果
dig +short www.google.com
关键事实：① DNS 默认 UDP 53，大响应回退 TCP 53；② 一次查询用 16 位 ID 配对请求/响应；③ 每条记录有 TTL（生存时间，秒），决定能被缓存多久——这是后面所有「为什么改了 DNS 没生效」问题的根源；④ NXDOMAIN 表示域名不存在，NOERROR 但 ANSWER 为空表示域名存在但没有该类型记录。

2. 记录类型：DNS 不只返回 IP

DNS 的每条记录叫 RR（Resource Record，资源记录），由「名字 + TTL + 类 + 类型 + 值」组成。类几乎永远是 IN（Internet）。真正多样的是类型——DNS 远不止「域名→IP」，它是一个通用的键值存储：

类型含义值示例用途
A域名 → IPv4142.250.196.68最常见，网页访问
AAAA域名 → IPv62404:6800:...IPv6 访问（读作 quad-A）
CNAME别名 → 另一域名ghs.google.com.「我指向那个域名，去问它要 IP」
MX邮件交换10 mail.x.com.收邮件投到哪（带优先级）
TXT任意文本"v=spf1 ..."SPF/DKIM 反垃圾、域名所有权验证
NS该区的权威服务器ns1.google.com.把子区「委派」给别人管
SOA区的起始授权主 NS + 序列号 + 各种 TTL每个区有且仅一条，定义区元数据
PTRIP → 域名（反向）www.google.com.反向解析，邮件服务器验证常用
SRV服务定位0 5 5060 sip.x.com指明某服务的主机+端口
CAA证书颁发授权0 issue "letsencrypt.org"限制谁能给你签 TLS 证书（接 Day 18）

CNAME：CDN 的入口

CNAME 特别重要，因为它是 CDN 接入的标准方式。当你把 www.yourapp.com 配成 CNAME → yourapp.cdn-provider.net.，相当于说「别问我了，去问 CDN 厂商，它会根据访客位置给最优 IP」。解析器拿到 CNAME 后会继续解析那个目标域名，直到拿到 A/AAAA 记录。

dig www.github.com
# www.github.com.        IN  CNAME  github.github.io.
# github.github.io.      IN  A      185.199.108.153   ← 链条最终落到 IP

# 看某域名的 MX（邮件）
dig +short gmail.com MX
# 看 TXT（常含 SPF / 域名验证记录）
dig +short google.com TXT
关键事实：① CNAME 不能与其他记录共存于同一名字（所以根域 example.com 不能直接 CNAME，需用厂商的 ALIAS/ANAME 变通）；② TXT 是「万能口袋」——也正因如此被攻击者用来藏外泄数据和 C2 指令；③ NS 记录的存在意味着「这一层把管辖权委派给了下一层」，是 DNS 分布式的核心机制。

3. 递归 vs 迭代：一次解析的完整旅程

这是今天最核心的概念。理解它，你就理解了 DNS 为什么「分布式但能工作」。区分两个角色：

  • 递归解析器（Recursive Resolver）：你电脑配置的那台 DNS（如 8.8.8.8 或公司内网 DNS、运营商 DNS）。它代你跑完全程，最后把答案打包给你。
  • 权威服务器（Authoritative Server）：根 / TLD / 域名所有者的服务器。它们只回答「我负责的那块」，不替你跑腿。

递归（recursive）是你对解析器的请求方式：「我只问一次，你给我最终答案」（RD=1）。迭代（iterative）是解析器对各权威服务器的请求方式：「你不知道完整答案没关系，告诉我下一步该问谁」。一次「递归请求」内部，是由解析器执行多次「迭代查询」完成的。

你的电脑 递归解析器(8.8.8.8) 权威服务器们 │ │ │ ① 递归请求 │ │ "www.google.com 的 A?" │ │ ───────────────────────>│ │ (RD=1，我只想要答案) │ ② 迭代：问根 │ │ ─────────────────> 根服务器(.) │ │ <───────────────── "去问 .com 的 NS" │ │ ③ 迭代：问 .com │ │ ─────────────────> TLD 服务器(.com) │ │ <───────────────── "去问 google.com 的 NS" │ │ ④ 迭代：问 google.com 权威 │ │ ─────────────────> 权威(ns1.google.com) │ │ <───────────────── "A = 142.250.196.68" │ ⑤ 返回最终答案 │ │ <───────────────────────│ (并按 TTL 缓存，下次秒回) │ │

缓存：让这套层层问路不至于太慢

如果每次访问网页都要从根开始问，互联网会被 DNS 查询淹没。缓存是 DNS 能 scale 的关键。每条记录带 TTL，解析器在 TTL 内直接用缓存回答。缓存分布在多处：浏览器缓存 → 操作系统缓存（nscd/systemd-resolved）→ 递归解析器缓存。这也解释了一个经典运维痛点：你改了 DNS 记录，但全球用户「看到新 IP」要等旧 TTL 过期（这就是为什么换服务器前要提前把 TTL 调小到 60 秒）。

# +trace 让 dig 自己走迭代全过程，亲眼看层层委派
dig +trace www.google.com
# 你会依次看到：. → com. → google.com. → 最终 A 记录

# 指定用某台解析器查询（@ 后面跟解析器 IP）
dig @1.1.1.1 www.cloudflare.com
dig @8.8.8.8 www.cloudflare.com   # 对比不同解析器返回的 IP（CDN 会不同！）

# 查询根提示
dig . NS +short
关键事实：① 全球有 13 组逻辑根服务器（a..m.root-servers.net），但靠 Anycast 实际是上千台物理机；② 「递归」描述客户端→解析器，「迭代」描述解析器→权威；③ 负缓存：NXDOMAIN 也会被缓存（按 SOA 里的负 TTL），所以打错域名后改对了有时要等一会；④ dig +trace 是理解迭代过程最直观的工具。

4. EDNS / DNSSEC / 0x20：协议的现代扩展

原始 DNS 设计于 1983 年，UDP 512 字节上限、无身份验证。后来打了几个重要补丁：

EDNS(0)：突破 512 字节

EDNS（Extension Mechanisms for DNS）用一条特殊的伪记录 OPT（放在 Additional 区）来协商更大的 UDP 报文（如 4096 字节）和声明新能力，避免动不动就回退到 TCP。其中最有名的子选项是 EDNS Client Subnet（ECS）：递归解析器把「用户的大致网段」转发给权威服务器，这样 CDN 能基于用户真实位置（而非解析器位置）返回更近的节点——这是 CDN 调度精度的关键，但也带来隐私争议（把你的网段暴露给了权威服务器）。

DNSSEC：给 DNS 加签名

原始 DNS 没有身份验证——你怎么知道收到的回答没被篡改？DNSSEC用公钥密码学给记录加数字签名（RRSIG/DNSKEY/DS 记录），形成从根到叶的「信任链」。它能防篡改和缓存投毒，但不加密内容（只保证「真实未被改」，不保证「没被看到」），且部署复杂、采用率仍不高。

0x20 编码：穷人的防投毒

由于 DNS 域名大小写不敏感，有些解析器会把查询域名随机大小写化（wWw.GooGle.cOm），响应必须原样返回——攻击者要伪造响应就得猜对每个字母的大小写，配合 16 位 ID，把伪造难度从 2^16 提到更高。这是一种轻量的熵增强抗投毒手段。

关键事实：① EDNS 用 OPT 伪记录扩展能力，ECS 让 CDN 调度更准；② DNSSEC 保证完整性（防篡改）但不保证机密性（不加密）；③ 加密查询是另一回事——DoH（DNS over HTTPS，443） 和 DoT（DNS over TLS，853） 才解决「查询内容被运营商窥视」的问题（这俩昨天 Day 20 已专门讲过）。

5. CDN 原理：把内容搬到用户家门口

CDN（Content Delivery Network，内容分发网络）解决一个物理问题：光速有限。回忆 Day 16，跨太平洋一个 RTT 约 150ms，连接建立、TLS 握手都要几个来回。如果你的服务器在弗吉尼亚，东京用户每个请求都要绕地球半圈。CDN 的核心思想：把内容复制到全球几百个「边缘节点（edge / PoP, Point of Presence）」，让用户访问离自己最近的那一个。

类比：与其让全国人民都去北京总仓库买货（慢、拥堵），不如在每个城市开连锁便利店（边缘节点），把畅销品提前铺货。顾客就近取货（命中缓存）；只有便利店没有的冷门货，才回总仓调货（回源 origin）。

┌─ 边缘节点(东京) ─┐ ← 日本用户就近访问 用户 ──DNS解析就近IP──> │ 缓存了静态资源 │ └────────┬─────────┘ │ miss 时回源 ▼ ┌─ 源站 origin ─┐ ← 你真正的服务器 │ (弗吉尼亚) │ 只承受 miss 流量 └───────────────┘ ↑ 关键：用户怎么被导到「最近的边缘」？答案就是 DNS / Anycast

CDN 怎么把你导到最近的节点？

这正是 DNS 和 CDN 的结合点。两种主流机制：

  1. DNS 调度：你的域名 CNAME 到 CDN，CDN 的权威 DNS 根据「请求来自哪个解析器/哪个网段（靠 ECS）」返回不同的边缘 IP。这就是为什么前面 dig @8.8.8.8 和 dig @1.1.1.1 查同一个 CDN 域名会得到不同 IP。
  2. Anycast：所有边缘节点共用同一个 IP，靠路由把你导到最近的（见下一节）。

CDN 还顺带做了很多事：缓存静态资源（图片/JS/CSS）、TLS 终止（在边缘做握手，离用户近、快）、压缩、DDoS 清洗、WAF（Web 应用防火墙）、甚至边缘计算（Cloudflare Workers / Lambda@Edge）。今天的大厂入口几乎都在 CDN 后面。

关键事实：① CDN 命中率（cache hit ratio）是核心指标，决定回源压力；② 动态内容（如 LLM 流式响应）无法缓存，但 CDN 仍提供就近 TLS 终止 + 优质骨干网回源加速；③ CDN 同时是安全屏障：隐藏源站真实 IP、抗 DDoS、WAF 拦截——这点对你做 Agent 安全产品的暴露面收敛很重要。

6. Anycast：一个 IP，全球就近接入

前面我们一直假设「一个 IP 对应一台机器」。Anycast打破了这个假设：同一个 IP 地址同时由全球多个地点的服务器宣告，网络（BGP 路由）会自动把你的包送到「路由意义上最近」的那个。

对比三种播送方式：Unicast（一对一，常规）、Multicast（一对多组播）、Broadcast（一对全网，已基本废弃）、Anycast（一对「最近的一个」）。Anycast 的实现靠 BGP：多个 PoP 都向互联网宣告同一段 IP 前缀，运营商路由器按 BGP 选最优路径，天然把用户送到最近的入口。

类比：拨打 110 报警电话。全国统一一个号码（一个 IP），但你打通的永远是你所在城市的接警中心（最近的节点），不是北京总台。你不需要知道本地号码，「最近」由电话网络自动完成。

# 8.8.8.8、1.1.1.1 都是 Anycast。从不同地区 ping，延迟都很低
ping -c 3 1.1.1.1

# 同一个根服务器 IP，不同地方 traceroute 会走向不同物理机房
traceroute 1.1.1.1     # macOS/Linux
# mtr 综合 ping+traceroute，持续观察路径与丢包
mtr 1.1.1.1

Anycast 的优势：天然就近（低延迟）、天然容灾（某节点挂了，BGP 自动收敛到其他节点）、天然抗 DDoS（攻击流量被分散到全球各 PoP，而非砸向一台机器——这是为什么 1.1.1.1、根服务器能扛住巨量攻击）。难点在于 TCP 长连接：如果路由中途切换到另一个节点，TCP 状态对不上会断连，所以 Anycast 最适合 DNS 这种「无连接、一问一答」的场景；HTTP/TCP 用 Anycast 需要稳定的路由 + 边缘做连接保持。

关键事实：① Anycast = 多地宣告同一 IP，BGP 选最近；② 对 DNS（UDP）天然契合，是根服务器和公共解析器扛量的基础；③ 把「就近」从应用层（DNS 调度）下沉到了路由层，更快、更抗攻击；④ 13 组根服务器靠 Anycast 实际是全球上千台。

7. HTTP 缓存：浏览器与 CDN 的省流大法

CDN 能加速，前提是「内容可以被缓存」。这套规则由 HTTP 缓存头定义（接 Day 17 的 HTTP）。理解它，你才知道 CDN 到底缓存了什么、缓存多久。

强缓存：根本不发请求

Cache-Control 是核心头。Cache-Control: max-age=3600 表示「这个资源 1 小时内直接用本地副本，别来问我」。配套关键字：public（CDN 可缓存）/ private（仅浏览器，CDN 不缓存，如含用户数据的页面）/ no-cache（可缓存但每次用前必须回源校验）/ no-store（绝不缓存，敏感数据用）/ immutable（永不变，常配合带哈希的文件名）。

协商缓存：问一句「变了吗」

强缓存过期后，浏览器带上「指纹」回源问：变了就给新的，没变回个 304 省下传输。两套指纹机制：

  • ETag（内容指纹，如哈希）+ 请求头 If-None-Match：服务器比对，没变回 304 Not Modified（空 body，超省流）。
  • Last-Modified（最后修改时间）+ 请求头 If-Modified-Since：按时间判断。
# 看响应里的缓存头（-I 只取 header）
curl -I https://www.cloudflare.com/
# 关注：cache-control / etag / age / cf-cache-status: HIT/MISS

# 带 ETag 发协商请求，命中会回 304
curl -I -H 'If-None-Match: "abc123"' https://example.com/style.css

CDN 边缘节点其实就是一个遵守这套 HTTP 缓存语义的大号共享缓存。响应头里的 Age（资源在缓存里待了多久）、厂商自定义的 X-Cache: HIT/MISS 或 cf-cache-status 能让你看清这次到底是边缘命中还是回源了。

关键事实：① 强缓存（max-age）不发请求，协商缓存（ETag/304）发请求但不传 body；② 典型最佳实践：HTML 用 no-cache（每次校验），带哈希文件名的 JS/CSS 用 max-age=31536000, immutable（永久缓存 + 改名失效）；③ private vs public 决定 CDN 是否参与，含敏感数据务必 private 或 no-store，否则可能被 CDN 缓存后泄露给其他用户（真实事故频发）。

🔗 与 AI / Agent 安全的连结

1. DNS 隧道外泄（DNS Tunneling Exfiltration）——你的头号监控对象。 假设一个被攻陷的 Agent（或它运行的容器）要把窃取的数据传出去，但出口防火墙封了大部分端口——唯独 DNS（53）几乎永远放行，因为封了它整个网络就瘫了。攻击者把数据编码进子域名：c2lzZW5zaXRpdmU.attacker.com，每次「查询」其实是在外传一个 base32 数据块，权威服务器（攻击者控制）从查询里解码。TXT 记录则用来回传 C2 指令。检测信号：异常长/高熵的子域名、单域名超高查询频率、大量 TXT/NULL 查询、不存在域名的疯狂解析。这正是你应该在 Agent 沙箱里埋点监控的行为。

2. SSRF + DNS Rebinding 攻破 Agent 工具。 当 Agent 有「访问 URL」类工具时，DNS 是绕过 IP 白名单的利器。DNS Rebinding：攻击者域名第一次解析返回合法外网 IP（通过校验），TTL 设为 0，第二次解析（实际取数据时）返回 169.254.169.254（云元数据服务）或内网 IP，从而让 Agent 替攻击者读取云凭证。防御：工具侧做「解析后校验 IP 是否私网」并锁定该 IP 发起连接（pin），而不是只校验域名。

3. CDN/Anycast 是 LLM 推理服务的护城河，也是误配陷阱。 你把推理网关放在 CDN 后面：隐藏源站真实 IP（攻击者无法直接 DDoS 你的 GPU 集群）、边缘 TLS 终止降低延迟、WAF 拦注入。但要警惕缓存误配——如果把含用户会话/API key 的响应错误标成 public, max-age，CDN 会把 A 用户的响应缓存后发给 B 用户（跨用户数据泄露），LLM 场景里这可能是把别人的对话内容串台。务必给所有含用户数据的端点设 Cache-Control: private, no-store。

4. ECS 隐私 vs RAG 检索投毒的类比思维。 EDNS Client Subnet 把用户网段暴露给权威服务器以换调度精度——这是「便利换隐私」的经典权衡，和你设计 RAG 时「把更多上下文喂给检索器以提召回，但扩大了 indirect prompt injection 面」是同构的。理解这种权衡的形状，能迁移到你对 Agent 数据流的安全设计：每多暴露一点信息给下游，就多一分被滥用的面。

5. DNS/CDN 是供应链与可用性的单点。 历史上多次大规模宕机（Dyn DNS 被 Mirai DDoS、各大 CDN 配置推送事故）证明：DNS 解析商和 CDN 是你整个 AI 服务可用性的隐形单点。做威胁建模（Week 24 会正式学 STRIDE）时，「DNS 提供商被攻击/被劫持」「CDN 全局配置错误」必须进你的攻击树。缓解：多 DNS 提供商、合理 TTL（太长不利于切换、太短增加查询和暴露）、关键域名启用 DNSSEC + CAA 记录锁定证书颁发者。

今日小练习

1动手：对一个 CDN 域名（如 www.cloudflare.com 或你常用的某个网站）执行 dig +trace，把「根 → TLD → 权威 → 最终 IP」的委派链抄下来。然后分别用 dig @8.8.8.8 和 dig @1.1.1.1 查同一域名，记录返回的 IP 是否不同，并解释原因。再用 curl -I 看这个站的 cache-control 和 cf-cache-status，判断它在缓存什么。

2观察：用 dig www.github.com 追踪 CNAME 链条，画出从 www.github.com 到最终 A 记录中间经过了哪些别名。再对比 dig +short example.com A（看 TTL）连续查两次——第二次的 TTL 数字会变小吗？为什么？（提示：本地/解析器缓存 + 倒计时）

3思考：你负责一个 LLM 推理服务，要给一个被攻陷可能性较高的 Agent 容器做出口监控。你只能看到容器发出的 DNS 查询日志（域名 + 类型 + 时间戳），看不到载荷。请设计 3 条检测规则来识别「DNS 隧道外泄」，并各举一个可能的误报场景（提示：考虑子域名长度/熵、查询频率、记录类型分布、目标域名信誉）。
