1. 问题起源：C10K → C10M

1999 年，Dan Kegel 写了一篇著名文章叫 The C10K problem：怎么让一台机器同时处理 10000 个并发连接？当时主流服务器是"一个连接一个进程"（Apache prefork）或"一个连接一个线程"——10000 个连接就要 10000 个线程，光线程栈就要吃掉几十 GB，上下文切换更是雪崩。

到 2010 年代问题升级成 C10M（千万并发），到今天的 LLM 流式推理网关，每个 SSE 长连接可能挂几分钟甚至几小时，"每连接一线程"彻底破产。解决路径只有一条：让少量线程通过 I/O 多路复用同时管很多个 fd。

演进时间线： 1983 4.2BSD select() ← 第一代 1986 System V poll() ← 干掉 fd 1024 上限 2000 FreeBSD kqueue() ← BSD 阵营事件机制 2002 Linux 2.6 epoll ← Linux 高并发基石 2014 Linux SO_REUSEPORT ← 多 worker 共享端口 2019 Linux 5.1 io_uring ← 真异步、零 syscall

你今天每天用的所有"高性能"东西——Nginx、Redis、Node.js、Netty、Go runtime、Python asyncio、Envoy——本质都是把这些机制包了一层。理解它们，你才能读懂为什么 vLLM 用 ZeroMQ + epoll、为什么 Triton 选 gRPC + HTTP/2、为什么 PyTorch 的分布式后端用 NCCL 而不是普通 socket。

2. 两组维度：同步/异步、阻塞/非阻塞

这是新手最容易混的概念。问题是中文翻译让它更乱了——"同步阻塞"、"同步非阻塞"、"异步阻塞"、"异步非阻塞"听起来像在玩绕口令。先把维度拆开。

维度 A：阻塞 vs 非阻塞（关于调用者要不要等）

你调用 read(fd, buf, n)，数据还没到。阻塞模式下，这个 syscall 不返回，你的线程睡在内核里，直到数据来了；非阻塞模式下，syscall 立刻返回 -1 + errno = EAGAIN，意思是"现在没数据，你自己看着办"。

# 设为非阻塞
import fcntl, os
flags = fcntl.fcntl(fd, fcntl.F_GETFL)
fcntl.fcntl(fd, fcntl.F_SETFL, flags | os.O_NONBLOCK)

维度 B：同步 vs 异步（关于数据怎么交付）

同步：你主动去读，数据从内核 buffer 拷贝到你的 buffer 这个动作由你的线程同步完成。异步：你提交一个请求，内核自己去办；办完后主动把数据放到你指定的 buffer 里，然后通知你。你完全不参与数据搬运。

判别要点：看"数据从内核空间到用户空间的拷贝由谁触发"。如果是你自己 read() 触发的——同步；如果内核自己完成拷贝、你只是收通知——异步。epoll 是同步的（它只通知你"可以读了"，你还得自己 read），io_uring 才是真异步（提交后内核全包，完成时给你通知）。

3. 五种经典 I/O 模型

UNIX Network Programming（W. R. Stevens 那本经典）把 I/O 处理方式正式分成 5 种。理解这 5 种，你才能在面试或架构讨论里不被绕晕。

  1. 阻塞 I/O（blocking I/O）：read() 没数据就睡。一个线程一个连接。
  2. 非阻塞 I/O（non-blocking I/O）：read() 没数据立刻 EAGAIN。配合 busy poll 用，几乎没人这么干。
  3. I/O 多路复用（I/O multiplexing）：一次性问内核"这堆 fd 里哪些可读"，得到列表后再分别 read。select / poll / epoll / kqueue 都属此类。这是现代服务的主流。
  4. 信号驱动 I/O（signal-driven I/O）：fd 准备好时内核发 SIGIO 通知你。理论存在，工程上几乎不用——信号语义太脆弱。
  5. 异步 I/O（asynchronous I/O）：提交后内核全包，完成时通知。POSIX aio 长期实现糟糕，io_uring 才让它真正可用。
五种模型时序对比（用户视角）： 阻塞: [---睡眠等待---][拷贝][返回] 非阻塞轮询: [问][EAGAIN][问][EAGAIN]...[问][有][拷贝][返回] I/O 多路复用: [select][发现某 fd 就绪][read 拷贝][返回] 信号驱动: [设 SIGIO][...做别的...][收到信号][read 拷贝][返回] 真异步: [提交][...做别的...][收到完成事件 + 数据已在用户 buf]

注意前 4 种都是同步——拷贝阶段都由你的线程做。只有第 5 种是真异步。I/O 多路复用经常被错叫成"异步 I/O"，要警惕这个用词混乱。

4. select 和 poll：第一代 I/O 多路复用

select：1983 年的设计

fd_set rfds;
FD_ZERO(&rfds);
FD_SET(sock1, &rfds);
FD_SET(sock2, &rfds);
// ... 把所有关心的 fd 加进去 ...
int n = select(maxfd + 1, &rfds, NULL, NULL, &timeout);
// 返回后遍历所有 fd 检查是哪个就绪了
for (int fd = 0; fd <= maxfd; fd++) {
    if (FD_ISSET(fd, &rfds)) { /* 处理 fd */ }
}

select 的三大致命缺陷：

  • fd 数量上限 1024（FD_SETSIZE，内核里写死的位图大小）。你想监听一万个连接？不行。
  • 每次调用都要把整个 fd 集合从用户态拷贝到内核态。1024 个 fd 也得拷 128 字节，频繁调用 CPU 浪费严重。
  • 返回后你不知道是哪个 fd 就绪，得 O(N) 遍历整个集合。10000 个 fd 里只有 1 个就绪，你也得遍历 10000 次。

poll：1986 年的修补

poll 把固定位图换成数组，干掉了 1024 上限，但前两个缺陷（每次拷贝、O(N) 遍历）依然存在。在 fd 数到上万时，select/poll 的 CPU 几乎全花在"检查谁就绪"上，实际能处理的 QPS 反而下降。

struct pollfd fds[10000];
fds[0].fd = sock1; fds[0].events = POLLIN;
// ...
int n = poll(fds, 10000, -1);
// 依然要遍历整个数组找 revents != 0 的

2000 年前后，所有要做高并发的人都被这两条卡住了——直到 Linux 2.6 / FreeBSD 4.1 时代的两个发明。

5. epoll：让 C10M 成为可能

epoll 是 Linux 内核 2002 年合入的彻底重做，核心思路是：把"关心哪些 fd"这件事一次性注册到内核，之后由内核主动维护"就绪列表"，用户态只取列表。三个 syscall：

int epfd = epoll_create1(0);                   // 在内核里建一个 epoll 实例

struct epoll_event ev = {.events = EPOLLIN, .data.fd = sock};
epoll_ctl(epfd, EPOLL_CTL_ADD, sock, &ev);     // 注册关心的 fd（一次性）

struct epoll_event events[1024];
int n = epoll_wait(epfd, events, 1024, -1);    // 阻塞等就绪事件
// n 就是就绪 fd 数量，events[0..n] 是就绪列表，无需遍历所有 fd

内部机制（红黑树 + 就绪链表）

内核里 epoll 实例维护两个结构：

  • 红黑树：存所有 epoll_ctl 注册过的 fd，O(log N) 增删查。
  • 就绪链表：fd 上有事件发生时（比如收到 TCP 数据），内核的中断回调把它挂到就绪链表上。epoll_wait 只看这个链表，O(就绪数) 而不是 O(总数)。

这就是 epoll 比 select/poll 快几个数量级的根本原因——无关心 fd 总数，只与就绪事件数挂钩。10 万个连接里只有 100 个有数据，epoll_wait 返回 100，select 还得遍历 10 万。

LT 模式 vs ET 模式（必考）

水平触发 LT (Level-Triggered)边缘触发 ET (Edge-Triggered)
触发条件只要 fd 当前可读就一直通知只在状态变化的那一刻通知一次
默认是（兼容 poll 语义）需要 EPOLLET flag
用法读一部分也行，下次 epoll_wait 还会再来必须一次性读完（循环 read 直到 EAGAIN），否则永远漏
性能简单，可能多触发几次更少 epoll_wait 调用，但代码必须严谨
典型用户大多数应用nginx、libevent 高性能模式
ET 模式的坑：fd 必须设为非阻塞。原因：ET 只在"从无到有"那一刻通知一次，你必须循环 read 把所有数据读完。如果 fd 是阻塞的，最后一次 read 会卡住整个事件循环。这是 ET 模式 90% bug 的根源。 
# 用 Python 的 selectors（内部就是 epoll/kqueue 选最佳）
import selectors, socket
sel = selectors.DefaultSelector()
srv = socket.socket(); srv.bind(("", 8080)); srv.listen(); srv.setblocking(False)

def accept(sock, mask):
    conn, _ = sock.accept(); conn.setblocking(False)
    sel.register(conn, selectors.EVENT_READ, read)

def read(conn, mask):
    data = conn.recv(4096)
    if data: conn.sendall(data)
    else: sel.unregister(conn); conn.close()

sel.register(srv, selectors.EVENT_READ, accept)
while True:
    for key, mask in sel.select():
        key.data(key.fileobj, mask)

6. kqueue：BSD 阵营的另一条路

FreeBSD 4.1（2000 年）发布的 kqueue，从设计上比 epoll 更通用——不只支持 socket fd，还能监听文件变化、信号、定时器、子进程退出等。Linux 后来分别用 inotify、signalfd、timerfd、pidfd 补齐这些能力，但 BSD 一次性就给齐了。

int kq = kqueue();
struct kevent change;
EV_SET(&change, sock, EVFILT_READ, EV_ADD | EV_ENABLE, 0, 0, NULL);
struct kevent events[64];
int n = kevent(kq, &change, 1, events, 64, NULL);

macOS 内核就是 XNU（FreeBSD + Mach），所以在 Mac 上跑的所有"高性能"框架（Node、Go、libuv、Python asyncio）底下用的都是 kqueue 而不是 epoll。你的 macOS 笔记本上 strace 等价物 dtruss 抓 Node.js 进程，看到的就是 kevent。

对应用层来说，epoll 和 kqueue 是同一类东西——都是同步的 I/O 多路复用，都把"就绪事件"集中起来给你。差别在 API 和能覆盖的事件类型，不在范式。所以 libuv、libevent 这些库都提供了统一封装，编一次就能在 Linux 和 macOS 上跑。

7. Reactor vs Proactor：两种事件循环范式

有了 I/O 多路复用，怎么把它组织成一个能写业务代码的框架？两种经典模式。

Reactor 模式（基于"准备就绪"）

核心思想：事件循环监听 fd → fd 可读时通知 handler → handler 自己 read + 处理。配套的是同步 I/O（epoll / kqueue）。

Reactor 单线程版（Redis 6.0 之前、Node.js 主循环）: ┌───────────────────────────────┐ │ event loop (epoll_wait) │ └──────────────┬────────────────┘ │ 发现 fd N 就绪 ▼ handler(fd) { read; 处理; write; } Reactor 多线程版（Nginx、Netty）: 主线程: accept + 分发 │ ├──→ worker 1 (epoll + handlers) ├──→ worker 2 (epoll + handlers) └──→ worker 3 (epoll + handlers)

Reactor 几乎统治了现代网络服务：Nginx 用多 worker Reactor、Redis 用单线程 Reactor、Node.js 用单线程 Reactor + 线程池、Netty 用主从 Reactor。Python 的 asyncio、Go runtime 的 netpoller 本质都是 Reactor。

Proactor 模式（基于"完成事件"）

核心思想：提交 I/O 请求 → 内核自己读完 → 完成时通知你，数据已经在你的 buffer 里。配套的是真异步 I/O（Windows IOCP、Linux io_uring）。

Reactor 是"你来读，我告诉你什么时候有"；Proactor 是"我帮你读完，读好了喊你拿"。Proactor 减少了一次同步阻塞的等待和一次用户态 syscall，理论上性能更高。但因为 Linux 长期没有像样的异步 I/O，Proactor 在 Linux 生态里直到 io_uring 出现前几乎没人用。

8. io_uring 与 aio：真正的异步 I/O

为什么 POSIX aio 失败了

POSIX 早就规定了 aio_read / aio_write 异步 I/O 接口，但 Linux 实现一直很差——只支持 O_DIRECT 文件 I/O、不支持 socket、底下用线程池模拟、性能差且兼容性糟。所以二十年里没人真用它。

io_uring：内核 5.1 (2019) 的革命

io_uring 的核心创新是两个共享内存环形队列——一个 SQ（Submission Queue）从用户态写、内核态读，一个 CQ（Completion Queue）从内核态写、用户态读。提交 I/O 和获取结果都不需要 syscall（用 IORING_SETUP_SQPOLL 模式时）。

io_uring 数据流: 用户态 内核态 ┌──────────────┐ ┌──────────────┐ │ App │ 写 SQ │ Worker │ │ ─────────────┼─→ ┌────────┐ ─→ │ 处理 I/O │ │ │ │ SQ │ │ ↓ │ │ │ └────────┘ │ 写 CQ │ │ 读 CQ │ ┌────────┐ ←─ │ │ │ ←────────────┼── │ CQ │ │ │ └──────────────┘ └────────┘ └──────────────┘ （内存映射，无需 syscall）

能力清单：

  • 支持所有 I/O——文件、socket、accept、connect、recvmsg、splice、定时器、信号……
  • 真异步：提交后线程立刻返回去做别的，内核处理完拿 CQ。
  • 支持链式——多个操作打包成一个链（A 完成自动触发 B），减少往返。
  • 支持缓冲池——内核自己挑空闲 buf 放数据，无需用户提前分配。
# Python 的 liburing 封装示例（仅示意，需安装 liburing-py 或 io-uring-py）
import liburing
ring = liburing.Ring(32)
sqe = ring.get_sqe()
sqe.prep_read(fd, buf, len(buf), 0)
ring.submit()
cqe = ring.wait_cqe()
print("read", cqe.res, "bytes")

所以你看到 ScyllaDB、TigerBeetle、新版本 PostgreSQL、Cloudflare quiche、Meta 的 wangle 都在切到 io_uring。但 io_uring 也是过去三年 Linux 内核安全漏洞集中爆发的子系统——Google 一度禁止在 ChromeOS / Android / GKE 默认开启，因为 sandbox 逃逸 CVE 太多。

🔗 与 AI / Agent 安全的连结

1. LLM 流式推理的 I/O 模型决定了"扛多少长连接"。OpenAI 兼容的 /v1/chat/completions 流式接口，每个用户对应一个 SSE 长连接，平均挂 30 秒、长的几分钟。你的网关如果用"每连接一线程"，1000 个并发就崩；用 epoll Reactor（FastAPI + uvicorn + uvloop / Go net/http / Nginx 反代），同样硬件能扛十万连接。读流式推理服务源码先看它的 I/O 范式，决定了它的容量上限——vLLM 的 HTTP 服务用 FastAPI + uvicorn（asyncio epoll），生产部署一定要在前面放个 nginx 或 envoy 做连接管理。

2. 慢连接 DoS 是 LLM 网关最常见也最便宜的攻击。Slowloris 攻击的本质是故意慢慢发请求 / 慢慢读响应，让服务端持续保留连接 + 上下文。对 LLM 服务尤其致命——每个挂着的连接都对应一份 KV cache 占着 GPU 显存。防御：(a) 网关层设连接级超时和 idle timeout；(b) limit_conn 限同一 IP 并发连接数；(c) 流式响应监控 客户端读取速率，长时间不读就强制断开（释放 KV cache）。这些都要在 epoll 这一层实现 —— 应用层 Python 抢不到这个时机。

3. SO_REUSEPORT + 多 worker 是热升级和负载均衡的基础。Linux 3.9 加入的 SO_REUSEPORT 允许多个进程绑同一个端口，内核负责把 accept 流量均匀分给各 worker。Nginx、Envoy、vLLM 多 worker 部署都靠它。但它也是攻击面——如果你的容器配置允许同 namespace 内多进程绑端口，恶意 sidecar 容器可以悄悄 bind 你的服务端口分流流量。这是云原生 LLM 部署的一个偏门 hardening 点：限制 worker pod 的 capabilities，禁止 CAP_NET_BIND_SERVICE 给非必要容器。

4. io_uring 是新攻击面。2022-2024 年 io_uring 子系统连续爆出多个本地提权漏洞（CVE-2022-1043、CVE-2022-29582、CVE-2024-0582 等），原因是它的复杂性（异步上下文 + 多种操作类型 + 共享内存）使得内核审计很难做。Google ChromeOS、Android、GKE 已经默认禁用普通用户访问 io_uring（kernel.io_uring_disabled = 2）。如果你的 Agent 沙箱跑用户代码，务必把 io_uring 加进 seccomp 黑名单，否则等于给攻击者一条全功能 syscall 旁路。

5. 事件循环里的阻塞调用是悄悄的性能黑洞。asyncio / Node.js 是单线程事件循环——一旦你在 handler 里写了个同步的数据库调用、同步的文件读、甚至一个 JSON.parse 几百 MB 数据，整个事件循环都被你这一个调用卡住，所有其他用户的请求全在排队。LLM 服务里最常见的是：在 asyncio handler 里调用一个同步的 tokenizer（HuggingFace tokenizer 默认不是 async）→ 一个 100KB prompt 的 tokenize 阻塞 50ms → 这 50ms 里 1000 个其他连接全冻结。修正：用 asyncio.to_thread() 或 run_in_executor 把 CPU 密集调用扔进线程池，让事件循环立刻回去处理别的 fd。

📝 今日小练习

120 分钟 · 动手：用 selectors 写一个 epoll 回声服务器，对比阻塞版

先用最朴素的"一线程一连接"写一个 echo server：

cat > echo_thread.py <<'EOF'
import socket, threading
srv = socket.socket(); srv.bind(("", 9001)); srv.listen()
def handle(conn):
    while True:
        data = conn.recv(4096)
        if not data: conn.close(); return
        conn.sendall(data)
while True:
    conn, _ = srv.accept()
    threading.Thread(target=handle, args=(conn,), daemon=True).start()
EOF

再用 epoll/selectors 重写（参考第 5 节的示例代码改成端口 9002）。然后用 ab -n 10000 -c 1000 http://127.0.0.1:9001/ 或简单 Python 客户端开 1000 个长连接打压测，对比：

  • 线程版的内存（top 看 RES）涨多少？epoll 版涨多少？
  • cat /proc/<PID>/status | grep Threads 看两边线程数。
  • 试着 strace -c -p <PID> 几秒，看主要 syscall 是 read/write 还是 epoll_wait/recvfrom。

# 找一个正在跑的 nginx / redis / 你的 Python LLM 服务
ps aux | grep -E "nginx|redis|uvicorn"

# 看它在用什么 I/O 多路复用
sudo strace -p <PID> -e trace=epoll_wait,epoll_ctl,kevent,io_uring_enter,select,poll 2>&1 | head -40

# 在 macOS 上换 dtruss
sudo dtruss -n nginx 2>&1 | grep -E "kevent|poll"

问题：(a) 你看到的是 epoll 还是 kqueue 还是 select？为什么？(b) epoll_wait 的 timeout 参数是多少？为什么不是 0 也不是 -1？(c) 如果完全看不到任何相关 syscall，可能在用什么？
