1. 从字节到文件：存储栈的全景与本质问题

先把今天要讲的东西放到一张图里。Linux 的存储栈从上到下大致是这样：

┌────────────────────────────────────────────────────────┐ │ 用户态：你的 Python / SQLite / LLM checkpoint 加载 │ │ open() / read() / write() / fsync() / mmap() │ ├────────────────────────────────────────────────────────┤ ← syscall 边界（Ring 3 → Ring 0） │ VFS 虚拟文件系统层（统一接口：file / inode / dentry） │ ├────────────────────────────────────────────────────────┤ │ 具体文件系统：ext4 / xfs / btrfs / tmpfs / nfs / fuse │ ├────────────────────────────────────────────────────────┤ │ Page Cache（页缓存：刚读过/写过的页留在内存里） │ ├────────────────────────────────────────────────────────┤ │ 块层 Block Layer（bio 请求、合并、IO 调度） │ │ 调度器：mq-deadline / kyber / bfq / none │ ├────────────────────────────────────────────────────────┤ │ 设备驱动（nvme / scsi / virtio_blk） │ ├────────────────────────────────────────────────────────┤ │ 硬件：NVMe SSD / SATA SSD / 机械盘 / 远程块存储 │ └────────────────────────────────────────────────────────┘

这一层一层的分工有一个共同主题：把"字节"和"持久化"拆开来谈。

  • 用户看到的是字节流，但磁盘只懂扇区（512B 或 4K）。
  • 用户调 write 时不一定真写到了盘——大概率只是写到 page cache，几秒后才落盘。
  • 用户调 read 时不一定真读了盘——大概率从 page cache 里直接拿到。
  • 真正"持久化"的承诺只在 fsync 返回后才生效。

这就解释了一件非常违反直觉的事：你 Python 写一个 1GB 文件可能只要 1 秒，但 fsync 单独一次可能要花几十毫秒甚至几秒。前者是写内存，后者才是真正把脏页推到 NVMe 上。

另一个核心问题：同一个 OS 怎么同时支持 ext4、xfs、btrfs、nfs、tmpfs 这么多不同的文件系统？答案就是 VFS——一层薄薄的抽象，把"什么是文件"、"什么是目录"、"什么是 inode"这些概念抽象出来，每种具体文件系统只要实现 VFS 定义的接口。这与你熟悉的 Python "duck typing" 极像：只要实现了 read() 方法，你就是个文件。

2. VFS：内核的"文件系统通用接口"

VFS（Virtual File System）是 Linux 内核里一个非常优雅的抽象层。它的目标是：给上层（syscall 入口）一个统一的"操作文件"的 API，不管下面是什么真实的文件系统。

类比：VFS 就像 Python 的 io.IOBase。你写代码用 f.read()，f 是磁盘文件、网络流、内存 BytesIO，对你来说都一样。这种"统一接口、底层多态"的思想，VFS 把它放到了内核里。

VFS 里的 4 个核心对象

读 Linux 内核代码你会反复看到这 4 个结构体（都定义在 include/linux/fs.h）：

对象它代表什么生命周期
struct super_block一个已挂载的文件系统实例（比如挂在 / 的 ext4）mount 时创建，umount 时销毁
struct inode磁盘上的一个文件/目录对象（无名）被引用时从盘上读到内存
struct dentry目录项缓存：一个名字到 inode 的映射路径解析时建立，LRU 回收
struct file一个"打开的文件"——open 返回的那个东西open 时创建，close 时销毁

这 4 个对象的关系大致是：一次 open("/etc/passwd") = 路径解析穿过若干个 dentry → 找到 inode → 创建一个 file 对象，把它放到进程的 files_struct 表里，返回 fd。

每个对象都有一组方法表（function pointer）

这是 VFS 多态的关键。struct file 里有一个字段 const struct file_operations *f_op，指向一组函数指针：

struct file_operations {
    ssize_t (*read)(struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write)(struct file *, const char __user *, size_t, loff_t *);
    int     (*open)(struct inode *, struct file *);
    int     (*release)(struct inode *, struct file *);
    int     (*fsync)(struct file *, loff_t, loff_t, int datasync);
    int     (*mmap)(struct file *, struct vm_area_struct *);
    // ... 30+ 操作
};

ext4 的 file_operations 里 read 指向 ext4_file_read_iter；NFS 的 read 指向 nfs_file_read；procfs 的 read 指向 proc_reg_read。同一个 syscall sys_read 进来，VFS 根据 file 对象的方法表把请求路由到正确的具体实现。

这就是为什么你可以用同一段 Python 代码读 ext4 上的文件、读 /proc/cpuinfo、读 NFS 挂载、读 FUSE 文件系统——上层是同一组 syscall，VFS 帮你把多态解决了。

3. inode 与 dentry：文件系统的两个核心对象

inode：文件的"身份"

很多新手以为「文件名」就是文件的身份。这是错的。在 Unix 设计里，文件的真实身份是 inode 号（一个整数）。文件名只是某个目录里指向这个 inode 的一个"标签"。

inode 里存了文件的所有元信息（除了名字）：

  • 文件大小、所有者 uid/gid、权限位 rwx
  • atime / mtime / ctime（访问时间 / 修改时间 / 元数据变更时间）
  • 硬链接数（nlink）——指向这个 inode 的目录项总数
  • 实际数据在磁盘上的位置（block pointer 或 extent tree）

看一个具体例子：

# 看文件的 inode 号
ls -i /etc/hostname
# 输出： 1234567 /etc/hostname

# stat 看完整的 inode 信息
stat /etc/hostname
# Size: 9   Blocks: 8   IO Block: 4096
# Inode: 1234567   Links: 1
# Access: (0644/-rw-r--r--)  Uid: (0/root)  Gid: (0/root)
# Access: 2026-05-24 ...
# Modify: 2024-01-01 ...
# Change: 2024-01-01 ...

注意 Links: 1。这是硬链接计数。ln a b 创建一条硬链接 → 同一个 inode 被两个名字指着，nlink=2。删除文件其实只是把目录里的那一项删掉、给 inode 的 nlink 减 1；只有 nlink 降到 0 且没有进程打开它时，文件才真正被回收。这就是 Unix 一个著名的小知识：rm 一个正在被进程用 tail -f 打开的日志文件，磁盘空间不会立即释放——直到那个进程退出。

dentry：路径解析的缓存

/usr/local/bin/python3 这样一个路径，要怎么从根目录走到那个 inode？答案是逐级查找：

  1. 从 /（根目录的 inode）开始
  2. 读它的目录数据，找到名字 "usr" → 拿到 usr 的 inode 号
  3. 读 usr 的目录数据，找到 "local" → 拿到 local 的 inode
  4. 读 local 的目录数据，找到 "bin" → 拿到 bin 的 inode
  5. 读 bin 的目录数据，找到 "python3" → 拿到 python3 的 inode

这是 5 次磁盘读。如果每次 open 都做一遍，性能完全崩溃。所以内核维护了一个dentry cache（dcache）：把"路径片段 → inode"的映射缓存在内存里，LRU 淘汰。下次再 open("/usr/local/bin/python3") 时，绝大概率直接命中 dcache，零次磁盘 IO。

实操查看你机器上的 dcache：

# Linux:
cat /proc/sys/fs/dentry-state
# 4 个数字：nr_dentry / nr_unused / age_limit / want_pages
# 第一个数字就是当前 dcache 里 dentry 的总数

slabtop -o | grep dentry
# 看 dentry slab 占了多少内存

4. ext4 / xfs / btrfs：三种主流文件系统的设计取舍

VFS 给上层统一接口，具体文件系统才是真正决定"数据在磁盘上怎么排"的地方。我们看三种主流的：

ext4xfsbtrfs
诞生2008（ext 家族 1992 起）1994（SGI IRIX 移植）2007
核心结构extent tree + journalB+ tree + journalCoW B-tree
最大文件16 TiB8 EiB16 EiB
快照不支持有限（reflink）原生支持
校验和仅元数据仅元数据数据 + 元数据
典型场景桌面 + 通用服务器大容量、并发写需要快照/校验

ext4：稳重的"主流选择"

ext4 是大部分 Linux 发行版的默认。它的核心创新是 extent——不再像 ext2/3 那样用「block pointer 数组」一格一格存，而是用「起始块 + 长度」表示一段连续的数据。顺序大文件的元信息开销大幅下降。

ext4 还有 journal（日志）：所有元数据修改先写到一个环形日志区，再写到目标位置。崩溃恢复时只需回放日志，避免 fsck 全盘扫描。三种模式：journal（数据和元数据都写日志）/ ordered（默认，元数据写日志、数据先于元数据写）/ writeback（只元数据写日志，性能最高、崩溃恢复后可能旧数据）。

xfs：高并发大文件之王

xfs 是 SGI 在 1994 年为 IRIX 设计的，2001 年移植到 Linux。它的特征：

  • Allocation Groups：把磁盘分成多个独立的"区"，每个 AG 有自己的空闲块管理和 inode 表 → 多线程并发分配不抢锁。
  • B+ tree 索引一切：空闲块、目录、extent 全部 B+ 树管理 → 大目录性能优秀。
  • 动态 inode 分配：不像 ext4 mkfs 时就定死 inode 数量。

红帽 RHEL 7+ 默认就是 xfs。大模型训练场景下存 checkpoint、log、shard 数据的目录，xfs 通常比 ext4 表现好。

btrfs：CoW + 快照 + 校验

btrfs 是 Oracle 2007 年开始做的，目标是和 ZFS 对标。核心是 Copy-on-Write（CoW）：任何修改都不覆盖原数据，而是写到新位置、再原子地更新指针。带来：

  • O(1) 快照：拍快照就是多记一个根指针。
  • 子卷（subvolume）：一个文件系统里隔出若干独立的"小文件系统"。
  • 数据校验和：每个数据块都有 CRC，自动检测位翻转（bit rot）。
  • RAID：内置 RAID 0/1/10。

代价是性能上常常不如 ext4/xfs，特别是数据库这种随机小写场景（CoW 会带来写放大）。所以工业界稳态部署里 btrfs 偏少，但 Synology / openSUSE / Fedora 桌面在推。

选型 cheat sheet：通用服务器/数据库 → ext4 或 xfs；大文件 + 高并发（数据湖、训练数据存储）→ xfs；需要频繁快照/防 bit rot → btrfs 或 ZFS；K8s 节点根分区 → 多数发行版默认 xfs。

5. 一次 read() 的完整磁盘 IO 路径

把前面所有概念串起来。看这一行 Python：

with open('/data/llama-7b.bin', 'rb') as f:
    chunk = f.read(4096)

从用户态到磁盘，可能发生的事情（最坏情况 = 缓存全 miss）：

[用户态] f.read(4096) ↓ libc → syscall read(fd, buf, 4096) [Ring 0] sys_read → vfs_read → file->f_op->read_iter ↓ [VFS / 具体 FS] ext4_file_read_iter ↓ 计算逻辑偏移 → 文件块号 ↓ 查 extent tree → 物理块号 = 12345678 ↓ [Page Cache] 查页缓存：这个 (inode, offset) 在不在？ ├── 在 → 直接 copy_to_user，返回（"热路径"，几微秒） └── 不在 ↓ [Block Layer] 构造 struct bio：目标设备 + 物理块号 + 长度 ↓ 加入请求队列 → IO 调度器排序合并 ↓ [Device Driver] NVMe 驱动：把 bio 翻译成 NVMe 提交队列条目 ↓ doorbell 通知设备 [Hardware] SSD 控制器读 NAND → DMA 到内存的 page cache 页 ↓ 完成中断 [Block Layer] bio 完成 → 唤醒等待的进程 [VFS] copy_to_user 把数据从 page cache 拷到用户 buf [用户态] f.read 返回，得到 bytes 对象

几个值得记的数字（典型量级）：

层级典型耗时
page cache 命中1-5 µs
NVMe SSD 一次随机读50-200 µs
SATA SSD 一次随机读100-500 µs
机械盘一次随机读5-10 ms（寻道 + 旋转）
远程网络盘 (NFS/EBS)1-10 ms

这就是为什么page cache 命中率几乎决定一切。Linux 默认把所有空闲内存都拿去做 page cache（free -h 里看到的 buff/cache 那一列）。所以你看到机器"用了 50GB 内存"未必是有进程吃了 50GB，可能是文件缓存。echo 3 > /proc/sys/vm/drop_caches 可以手动清掉，常用于性能测试时拿到"冷缓存"基线。

6. IO 调度器：Linux 块设备的策略大脑

当多个进程同时发 IO 请求，块层不会"先来先服务"傻乎乎地往设备发。中间有个 IO 调度器（I/O Scheduler），负责合并相邻请求、按某种策略排序、避免某些进程被饿死。

blk-mq 时代的 4 个调度器

Linux 4.x 之后切到了 blk-mq（多队列块层），目前 Linux 6.x 主线的可选调度器：

调度器设计目标适用场景
none不调度，直接透传NVMe / 高并发 SSD（设备自己有队列）
mq-deadline给每个请求一个 deadline，避免饿死SATA SSD / 数据库混合负载
kybertoken 机制控制读/写延迟低延迟要求场景
bfq预算公平队列，进程级别公平桌面 / 交互式

看你机器上的设置：

# 看 /dev/sda 当前用什么调度器
cat /sys/block/sda/queue/scheduler
# 输出例如: [none] mq-deadline kyber bfq
# 方括号里就是当前

# 临时改：
echo mq-deadline | sudo tee /sys/block/sda/queue/scheduler

# 看 NVMe：
cat /sys/block/nvme0n1/queue/scheduler
# NVMe 通常默认 [none] —— 因为 SSD 自己内部已经有几十个并发队列

合并（merge）：调度器最大的性能贡献

设想 100 个进程同时各发了一个 4K 读请求，恰好读的是相邻块。如果直接每个都下发，就是 100 次 IO。调度器会发现它们的物理块号相邻，合并成一个大请求一次性下发，吞吐立刻提升几十倍。看合并的统计：

iostat -x 1
# 关注列：
# rrqm/s wrqm/s : 每秒合并的读/写请求数
# r/s w/s       : 实际下发到设备的 IO 次数
# 如果 rrqm 远大于 r —— 合并率高，调度器在干活

7. fsync 与 O_DIRECT：持久化和绕过 page cache

fsync：把承诺变成事实

前面说过：write 通常只写到 page cache。真正落盘要靠 fsync(fd)。这件事对数据库、日志、checkpoint 极其重要。看一个典型的"WAL"模式（write-ahead log，几乎所有数据库都用）：

# 伪代码
write(log_fd, log_entry)    # 先写日志
fsync(log_fd)               # 确保日志落盘 ← 关键
apply_change_to_data()      # 再修改数据

这个 fsync 一旦返回，OS 就承诺即使下一秒断电，日志数据也已经在盘上了。代价是慢——典型 NVMe 一次 fsync 几百微秒，机械盘几十毫秒。所以高性能系统会想办法批量做 fsync（group commit）：攒一批写一起 fsync，把延迟摊薄。

注意一个常被忽略的细节：fsync 不仅要刷数据页，还要刷文件元数据（比如文件变大后的 inode 信息）。如果你只在意数据、不在意 mtime 之类的元信息，可以用 fdatasync，省一次元数据 IO。

O_DIRECT：绕过 page cache

有些场景不希望走 page cache：

  • 数据库（PostgreSQL/MySQL/Oracle）：自己有 buffer pool，再叠一层 page cache 是"双重缓存"浪费内存。
  • 大模型 checkpoint 一次性流式读：读完就丢，留在 cache 里反而把别人的热数据挤掉。
  • 性能测试：你想测"裸盘性能"，cache 干扰太大。

解决办法是 open 时加 O_DIRECT 标志。数据直接 DMA 在用户 buffer 和设备之间走，跳过 page cache。代价：

  • 用户 buffer 必须按设备块大小对齐（通常 4096 字节）。
  • 偏移和长度也要对齐。否则 EINVAL。
  • 实现复杂，错用比走 cache 还慢。

所以 O_DIRECT 是"专业工具"，不是日常 API。Python 里要用 os.open(path, os.O_RDONLY | os.O_DIRECT)，再配合 mmap 拿对齐 buffer。

一个让人惊掉下巴的事实：fsync 在很多文件系统的早期版本里语义并不一致。著名的 "fsyncgate" 2018 年事件揭示：Linux 上某些 ext4 配置下 fsync 失败后，对应的脏页会被默默丢弃，下次 fsync 会返回成功——程序以为数据已经在盘上，其实没有。PostgreSQL 因此专门改写了 fsync 错误处理逻辑。教训：fsync 错误必须当致命错误处理，不能 retry。

8. 零拷贝：sendfile / splice / mmap，与大模型 checkpoint 加速

典型的"读文件然后通过 socket 发出去"代码长这样：

buf = file.read(N)        # 1. 内核 page cache → 用户 buf（拷贝 1）
sock.send(buf)            # 2. 用户 buf → 内核 socket buf（拷贝 2）
                          # 3. 网卡 DMA 出去

数据在内核态和用户态之间来回拷贝了 2 次，还伴随 2 次 syscall 上下文切换。文件越大、网络流量越高，浪费越严重。这正是 nginx 静态资源、CDN、视频流服务最常见的瓶颈。

sendfile：内核里直接搬

sendfile(out_fd, in_fd, &offset, count) 让内核不经过用户空间，直接把数据从一个 fd 搬到另一个 fd。数据路径：

传统：磁盘 → page cache → 用户 buf → socket buf → 网卡 sendfile：磁盘 → page cache ─────────────→ socket buf → 网卡 ↑ 内核里直接转发（DMA gather 还能再省一次）

nginx 一行配置就开：

http {
    sendfile on;
}

静态文件 QPS 通常能提升 30-50%。

splice：管道中转

sendfile 只支持 "文件 → socket"。splice(in_fd, NULL, pipe_fd, NULL, len, 0) 更通用——通过管道作为内核里的中转，可以在任意两个支持 splice 的 fd 之间搬数据。Nginx 和 HAProxy 用 splice 做 TCP 转发，连 page cache 都不进。

mmap：把文件映射到地址空间

mmap 是另一种零拷贝技巧。mmap(NULL, len, PROT_READ, MAP_PRIVATE, fd, 0) 把文件映射到进程地址空间——之后你直接像访问数组一样访问内存，内核按需 page fault 把对应页读进来。好处：

  • 没有 read syscall 的开销
  • 没有从 page cache 到用户 buf 的拷贝（用户 buf 就是 page cache 那页的映射）
  • 多个进程 mmap 同一个文件 → 物理上只有一份，自动共享

大模型 checkpoint 加载的真实案例

Llama-2-7B 的 FP16 权重大约 13GB。比较两种加载方式：

方式 A：torch.load 默认

state = torch.load('llama-7b.pt')
# 内部：open + read 整个文件到用户 buf → 反序列化为 tensor
# 进程 RSS 立刻涨 13GB；冷启动慢（要等磁盘读完）

方式 B：mmap 加载（safetensors / weights_only=True 走 mmap）

from safetensors.torch import load_file
state = load_file('llama-7b.safetensors', device='cpu')
# 内部：mmap 整个文件，按需 page fault
# 进程 VSZ 涨 13GB，但 RSS 慢慢涨；冷启动快（不阻塞）
# 多 worker 加载同一份权重 → 物理只占一份内存

vLLM、TGI 等推理引擎在多 worker 场景下大量依赖 mmap 共享权重。如果你 8 个 worker 各自 torch.load，物理内存就是 8×13GB = 104GB；用 mmap 只占 13GB。这就是为什么 safetensors 一开始就被设计成 mmap-friendly 的二进制格式（头部有偏移表，可以直接定位到张量数据）。

🔗 与 AI / Agent 安全的连结

1. 模型权重供应链与 mmap。你下载的 .safetensors 或 .pt 文件实际上是一段二进制数据 + 一些反序列化逻辑。torch.load 在底层会调 pickle.load——这就是为什么不可信来源的 .pt 文件可以执行任意代码（pickle 反序列化是图灵完备的）。用 safetensors（纯数据 + mmap）才是安全的。你今天理解了 mmap，就知道 safetensors 为什么能既快又安全：它的设计强制把"代码"从"数据"中剥离。Week 22 周六会专门讲模型供应链攻击（log4shell 类比），今天打了基础。

2. Agent 沙箱里的文件系统隔离。Agent 经常要执行不可信的代码——这些代码会读文件、写文件、甚至 os.listdir('/') 探索宿主。OS 提供的隔离原语在这里至关重要：mount namespace 让 Agent 看到一个干净的 rootfs；只读绑定挂载防止它写宿主敏感目录；tmpfs 上限防止它把磁盘填满；overlayfs 把基础镜像做共享。Week 09 我们会讲 namespace，但今天 inode/dentry 的概念帮你理解为什么 "bind mount /etc/passwd 到容器里" 是绝对禁止的——因为本质上是给容器内进程一条直达宿主 inode 的引用。

3. 日志与审计的持久化保证。你的 Agent 安全产品要记录"谁在什么时候调了什么工具"，这是合规审计的硬要求。如果日志只写 page cache 不 fsync，一旦机器崩溃丢失最近几秒的记录，整个审计链就断了。你需要做 group commit fsync——攒一批审计事件一起落盘，平衡性能和耐久性。今天对 fsync 语义（fsyncgate 教训）的理解直接告诉你：fsync 失败必须当致命错误，不能简单重试。

4. 大模型推理服务的冷启动优化。LLM gateway 要支持"分钟级扩容"——突然来流量时新拉 GPU、加载模型、对外服务。冷启动里最大的耗时往往是权重加载。两个直接优化点都来自今天：(a) 用 mmap + safetensors，按需 page fault；(b) 模型权重存在 xfs 上（大文件顺序读优于 ext4），或更激进地用 GPUDirect Storage 让 NVMe 直接 DMA 进 GPU 显存，跳过 CPU。Week 08 讲推理引擎时这些都会用上。

5. 检索投毒中的"文件系统"隐患。RAG 系统从一个目录读知识库文档。攻击者如果有写文档目录的权限，可以放进一份"看起来无害但含 indirect prompt injection 的 markdown"。更隐蔽的版本：用符号链接、硬链接、特殊字符的文件名，让你的索引器爬到本不该爬的位置。今天 inode 和 dentry 的概念帮你设计正确的防御——不能只看路径字符串，要 realpath 解析后检查、要禁止跨文件系统边界、要拒绝符号链接（O_NOFOLLOW）。Week 19 周六（Day 132）会展开。

130 分钟 · sendfile vs read+write 性能对比

写两个 Python 脚本传一个 100MB 文件经过本地回环 socket。一个用 file.read() + sock.send()，一个用 os.sendfile(sock_fd, file_fd, 0, 100*1024*1024)。各跑 10 次取平均：

import os, socket, time

# 准备：
# dd if=/dev/urandom of=/tmp/100M bs=1M count=100

def serve_once(use_sendfile):
    srv = socket.socket(); srv.bind(('127.0.0.1', 0)); srv.listen(1)
    port = srv.getsockname()[1]

    pid = os.fork()
    if pid == 0:  # 子进程：客户端，收数据
        c = socket.socket(); c.connect(('127.0.0.1', port))
        total = 0
        while True:
            d = c.recv(64*1024)
            if not d: break
            total += len(d)
        os._exit(0)

    conn, _ = srv.accept()
    f = open('/tmp/100M', 'rb')
    t0 = time.time()
    if use_sendfile:
        os.sendfile(conn.fileno(), f.fileno(), 0, 100*1024*1024)
    else:
        while True:
            d = f.read(64*1024)
            if not d: break
            conn.sendall(d)
    conn.close(); f.close()
    os.waitpid(pid, 0)
    return time.time() - t0

print('read+write:', sum(serve_once(False) for _ in range(5))/5)
print('sendfile  :', sum(serve_once(True)  for _ in range(5))/5)

问题：哪种快？快多少？再用 strace -c 跑一次看 syscall 次数差异。

310 分钟 · 思考题

你的 LLM 推理服务部署在 K8s 上，每个 Pod 启动时要从对象存储下载 13GB 模型权重到本地磁盘，再 mmap 加载。冷启动总耗时 90 秒。运维要求降到 20 秒以内。下面 5 种思路，你会选哪几个组合，按收益排序：

  1. 把权重打进容器镜像（镜像直接 100GB+）
  2. 用 init container 提前并行下载（多线程 multipart）
  3. 把权重放共享 PVC（NFS / 块存储），同节点 Pod 复用
  4. 用 daemonset 在每个节点本地预热一份权重，pod 通过 hostPath 只读挂载
  5. 用 GPUDirect Storage + safetensors，从 NVMe 直接 DMA 到 GPU 显存

提示：考虑 (1) 网络带宽 (2) 节点磁盘 IOPS (3) 多 pod 共享内存的可能性 (4) 容器镜像 pull 缓存 (5) 你今天学的 mmap / page cache 在多容器场景下还能不能共享物理页。
