1. LAB 1：free + vmstat 字段全解读

这俩工具是你 ssh 到一台陌生机器后的第一组命令。30 秒内你就要决定：内存压力大不大、要不要扩容、是不是已经在 swap。今天把每个字段搞清楚。

步骤（Linux 优先；mac 等价命令在每步末尾）

A看一眼全局内存。
# Linux
free -h

# 你会看到类似：
#               total        used        free      shared  buff/cache   available
# Mem:           15Gi       4.2Gi       1.1Gi       320Mi        10Gi        10Gi
# Swap:         2.0Gi          0B       2.0Gi

# mac 等价
vm_stat
# 输出是按 page (4096 / 16384 字节) 计数，要自己乘 page size。
sysctl hw.memsize hw.pagesize
关键解读：used 不是"用掉了多少"——它不包括 buff/cache。真正决定"还能不能再开应用"的是 available 列（buff/cache 大部分可以瞬间回收）。新手最常见的恐慌是看 free 很小就以为要 OOM，其实只要 available 还多就没事——这是 Day 10 page cache 那节讲的"Linux 会用满你所有内存，但都是 cache"。
Bvmstat 看动态：内存、swap、IO、CPU 全在一张表。
# Linux: 每 1 秒采一次，连采 10 次
vmstat 1 10

# 输出关键列：
# procs ----memory---------- ---swap-- -----io---- -system-- ------cpu-----
#  r  b   swpd   free  buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  1  0      0   1.1G  120M  9.8G    0    0    12    34  301  847 12  3 84  1  0

# mac 没有 vmstat，但 vm_stat 1 也能周期采样
vm_stat 1
这 18 列里你只需要盯死 5 列：r=就绪队列长度（>CPU 核数就是 CPU 过载）；si/so=每秒从 swap 读入/写出的 KB（任何非零都是危险信号）；bi/bo=磁盘读写；cs=每秒上下文切换次数（高得离谱说明线程太多）；wa=CPU 等 IO 的百分比。"si 和 so 任意一个不是 0" 就立刻报警——这是 Day 9 swap 那节讲的"swap 一旦激活，性能崩塌"。
C动手制造一次内存压力。
# 起一个 Python 程序快速吃 8GB 内存
python3 -c "
import time
buf = bytearray(8 * 1024 * 1024 * 1024)  # 8 GiB
print('allocated 8GiB, sleeping 30s...')
time.sleep(30)
" &

# 另开终端，盯着 free -h 变化
watch -n 1 free -h

# 同时盯 vmstat
vmstat 1
观察：used 一路涨，available 跌；如果你的机器内存不够 8GiB，会先看到 buff/cache 被挤压（系统把可丢弃的缓存释放给你），然后 si/so 出现非零（开始 swap），最后可能触发 OOM killer。Day 9 第 7 节 OOM 那段，今天你能亲眼看到了。

2. LAB 2：/proc/[pid]/maps 与 smaps

Day 8 你画了 32 位 / 64 位地址空间的图。今天我们打开一个真实进程，对着图找每段。

mac 注意：mac 没有 /proc。等价工具是 vmmap <pid>，输出格式不同但信息量类似。本节用 Linux 演示，mac 用户用 vmmap 替代。

步骤（Linux / Linux Docker）

A挑一个正在跑的进程（拿你自己的 shell）。
echo "我的 shell PID: $$"
cat /proc/$$/maps | head -30

# 你会看到一堆形如：
# 555f8c800000-555f8c82c000 r--p 00000000 fd:01 1234567 /usr/bin/bash
# 7f6a3a000000-7f6a3a02d000 r--p 00000000 fd:01 7654321 /usr/lib/.../libc.so.6
# 7ffd1c000000-7ffd1c021000 rw-p 00000000 00:00 0       [stack]
# 7ffd1c0bb000-7ffd1c0be000 r--p 00000000 00:00 0       [vvar]
每一行 = 一段 VMA（virtual memory area）。列含义：起始-结束地址 权限(rwxp) 文件偏移 设备 inode 路径。r--p=只读私有（典型的代码段）；rw-p=可读写私有（堆/栈/匿名映射）；r-xp=可读可执行（旧风格代码段）。注意 [stack]、[heap]、[vvar]、[vdso] 这几个特殊段——这就是 Day 1 第 4 节进程地址空间图里画的那几块。
B对 maps 做统计：进程一共映射了多少 MB？
cat /proc/$$/maps | awk '
{
  split($1, a, "-")
  start = strtonum("0x" a[1])
  end   = strtonum("0x" a[2])
  total += end - start
}
END { printf "VSZ (虚拟内存大小): %.1f MB\n", total/1024/1024 }'
典型 shell 的 VSZ 在 10-30 MB。这只是"虚拟地址空间用了多大"，不是实际物理占用。VSZ ≠ RSS——这是 Day 8 / Day 10 反复强调的：很多 VMA 是"声明了但没真正读进物理页"的。
Csmaps 看每段的真实物理占用。
# /proc/[pid]/smaps 给每个 VMA 附加详细统计
cat /proc/$$/smaps | head -40

# 关键字段:
# Size:        虚拟大小（= maps 里那行的 end - start）
# Rss:         实际驻留物理内存大小
# Pss:         按共享比例分摊的 Rss（多个进程共享的库会被除以共享数）
# Shared_Clean / Shared_Dirty:  和其他进程共享的页
# Private_Clean / Private_Dirty: 私有页
# Swap:        被换出到 swap 的部分
D找出进程里"实际最占内存"的 5 个段。
cat /proc/$$/smaps | awk '
/^[0-9a-f]+-/ { name=$0; }
/^Rss:/       { rss=$2; print rss " KB  " name }
' | sort -rn | head -5
预期：通常排第一的是 [heap] 或某个大共享库（libc）。如果你的 LLM 推理进程做这个统计，第一名一定是某个匿名 rw 段——那就是权重 + KV cache。这也是 Day 11 mmap 模型加载相关讲过的：HuggingFace 的 safetensors 默认就是 mmap 进来的，在 smaps 里会看到一个几十 GB 的 r--p 文件映射段。
Emac 等价。
# mac 上用 vmmap
vmmap $$ | head -40
# 输出会按 region type 分组（MALLOC / Stack / __TEXT / __DATA / ...），
# 比 Linux 的 maps 更"语义化"

# 看一个进程所有 region 大小排序
vmmap $$ -summary

3. LAB 3：用 mmap 加速大文件读取

Day 10 讲了 mmap 把文件映射到进程地址空间，省去 read 的拷贝。今天对比一下 read vs mmap 在"统计一个 1GB 文件里某字符出现次数"上的差距。

步骤

A生成一个 1GB 测试文件。
# 用 /dev/urandom 写 1GB（mac / Linux 都能跑；慢一点用 zero 也行）
dd if=/dev/zero of=/tmp/bigfile bs=1M count=1024
ls -lh /tmp/bigfile  # 应该是 1.0G

# 别忘了清掉 page cache 才能公平测试 (仅 Linux)
sync; echo 3 > /proc/sys/vm/drop_caches   # 要 root
# mac 上重启或 purge（要 sudo）
sudo purge   # mac 等价的清缓存
B方案 1：传统 read 一次性读进来。
cat > /tmp/read_bench.py <<'EOF'
import time, sys
path = '/tmp/bigfile'
target = ord('a')

t0 = time.perf_counter()
with open(path, 'rb') as f:
    data = f.read()                  # 整个文件读进 user buffer
    count = data.count(bytes([target]))
print(f"read all + count: {time.perf_counter()-t0:.2f}s, count={count}")
EOF

python3 /tmp/read_bench.py
预期：5-10 秒（首次冷读），内存峰值占用 ~1GB。第二次跑（page cache 已热）会快很多。
C方案 2：mmap 映射，不显式拷贝。
cat > /tmp/mmap_bench.py <<'EOF'
import time, mmap, os
path = '/tmp/bigfile'

t0 = time.perf_counter()
with open(path, 'rb') as f:
    mm = mmap.mmap(f.fileno(), 0, prot=mmap.PROT_READ)
    count = mm.count(b'a')
    mm.close()
print(f"mmap + count: {time.perf_counter()-t0:.2f}s, count={count}")
EOF

# 先清缓存（如有权限），再跑
sudo purge   # mac
# echo 3 | sudo tee /proc/sys/vm/drop_caches   # Linux

python3 /tmp/mmap_bench.py
预期：冷启动下 mmap 不一定快多少（磁盘 IO 一样要做），但 user 内存峰值会显著低——你不再持有一份完整文件副本，OS page cache 才是真正持有者。在多个进程并发处理同一个文件时差距会拉开：mmap 多个进程共享同一份物理页，read 各自拷贝一份。
D用 madvise 进一步提示访问模式。
cat > /tmp/mmap_advise.py <<'EOF'
import time, mmap, os
with open('/tmp/bigfile', 'rb') as f:
    mm = mmap.mmap(f.fileno(), 0, prot=mmap.PROT_READ)
    # 告诉内核我会顺序读 → 内核会更积极地预读后面的页
    mm.madvise(mmap.MADV_SEQUENTIAL)
    t0 = time.perf_counter()
    count = mm.count(b'a')
    print(f"seq advised: {time.perf_counter()-t0:.2f}s, count={count}")
    mm.close()
EOF
python3 /tmp/mmap_advise.py
解读：madvise 是"我向内核保证我的访问模式，内核以此优化预读 / 回收策略"。MADV_SEQUENTIAL = 顺序读，MADV_RANDOM = 随机访问（关闭预读），MADV_DONTNEED = 用完丢弃（适合大模型 checkpoint 加载完就 evict）。vLLM、HuggingFace transformers 加载 safetensors 时都会用 madvise 优化——下周二讲推理引擎你会再见。

4. LAB 4：Python selectors 写 epoll 回声服务器

Day 11 你学了 5 种 IO 模型，epoll 是 Linux 上高并发服务的核心。今天用 Python selectors 模块（自动选最优后端：Linux=epoll，mac=kqueue，Windows=select）写一个能撑 1 万并发的回声服务器——50 行代码。

步骤

A先写一个最朴素的"一连接一线程"的版本作为对照。
cat > /tmp/echo_thread.py <<'EOF'
"""阻塞 + 多线程版：每连接一线程"""
import socket, threading

def handle(conn, addr):
    while True:
        data = conn.recv(4096)
        if not data: break
        conn.sendall(data)
    conn.close()

s = socket.socket()
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(('127.0.0.1', 9001))
s.listen(128)
print('thread-echo listening on 9001')
while True:
    conn, addr = s.accept()
    threading.Thread(target=handle, args=(conn, addr), daemon=True).start()
EOF

python3 /tmp/echo_thread.py &
B用 selectors 写 epoll/kqueue 版。
cat > /tmp/echo_epoll.py <<'EOF'
"""单线程 + selectors（背后是 epoll / kqueue）"""
import socket, selectors

sel = selectors.DefaultSelector()

def on_accept(server):
    conn, addr = server.accept()
    conn.setblocking(False)
    sel.register(conn, selectors.EVENT_READ, on_read)

def on_read(conn):
    try:
        data = conn.recv(4096)
    except (ConnectionResetError, BlockingIOError):
        data = None
    if not data:
        sel.unregister(conn)
        conn.close()
        return
    conn.sendall(data)

server = socket.socket()
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(('127.0.0.1', 9002))
server.listen(1024)
server.setblocking(False)
sel.register(server, selectors.EVENT_READ, on_accept)

print(f'selectors-echo listening on 9002 (backend: {type(sel).__name__})')
while True:
    for key, mask in sel.select(timeout=None):
        callback = key.data
        callback(key.fileobj)
EOF

python3 /tmp/echo_epoll.py &
注意打印的 backend：mac 上是 KqueueSelector，Linux 上是 EpollSelector。Python 替你选好了——这就是 selectors 模块的价值。
C用 nc 手动测一下两个服务器都能跑。
nc 127.0.0.1 9001
# 输入任何字符串回车，会被回显
# Ctrl+C 退出

nc 127.0.0.1 9002
# 同上
D对比并发能力（粗略 benchmark）。
# 写一个并发连接 + 收发的客户端
cat > /tmp/echo_client.py <<'EOF'
import socket, threading, time, sys
port = int(sys.argv[1])
N = int(sys.argv[2])

def one():
    s = socket.socket()
    try:
        s.connect(('127.0.0.1', port))
        for _ in range(20):
            s.sendall(b'x'*64)
            s.recv(64)
        s.close()
    except Exception as e:
        print(e)

t0 = time.perf_counter()
ts = [threading.Thread(target=one) for _ in range(N)]
for t in ts: t.start()
for t in ts: t.join()
print(f"N={N} port={port}: {time.perf_counter()-t0:.2f}s")
EOF

# 测 thread 版（容易在 N>500 时崩或变慢）
python3 /tmp/echo_client.py 9001 500

# 测 epoll 版（同样 500 个并发，更稳更快）
python3 /tmp/echo_client.py 9002 500

# 加大到 2000 个看差距
python3 /tmp/echo_client.py 9002 2000
预期：thread 版在 500 并发时已能感到吃力（线程切换 + GIL）；epoll/kqueue 版能轻松撑 2000+ 并发——所有事件在同一个线程循环里处理。这就是 Nginx / Redis / Node.js 高并发的秘密。Day 11 第 4 节那张"select vs poll vs epoll 复杂度对比"，今天你跑过两个版本就有手感了。
E清理：杀掉后台服务器。

5. LAB 5：valgrind / leaks 抓内存泄漏

Python 本身有 GC，不太容易写出内存泄漏；但如果你用 ctypes 调 C 库、或写 C 扩展、或调用未释放的第三方库，泄漏一样会发生。valgrind（Linux）/ leaks（mac）是定位泄漏的金标准。

步骤

A故意写一段 C 代码有泄漏。
cat > /tmp/leak.c <<'EOF'
#include <stdlib.h>
#include <string.h>

void leak_one() {
    char *p = malloc(1024);     // 1 KB
    strcpy(p, "hello leak");
    // 故意不 free
}

int main() {
    for (int i = 0; i < 100; i++) leak_one();
    return 0;
}
EOF

# Linux:
gcc -g -O0 /tmp/leak.c -o /tmp/leak

# mac (同样的 gcc 命令，clang 兼容):
clang -g -O0 /tmp/leak.c -o /tmp/leak
BLinux: 用 valgrind 跑。
# Linux Docker 里:
apt install -y valgrind
valgrind --leak-check=full --show-leak-kinds=all /tmp/leak

# 你会看到类似:
# ==12345== 102,400 bytes in 100 blocks are definitely lost in loss record ...
# ==12345==    by 0x4006A2: leak_one (leak.c:5)
# ==12345==    by 0x4006D9: main      (leak.c:11)
valgrind 的 4 种泄漏分级：definitely lost（找不到指针了，铁定泄漏）；indirectly lost（结构体指针丢了，里面的子结构也连带丢）；possibly lost（指针只指到内部某个偏移，可能是设计的，也可能是 bug）；still reachable（程序退出时还能找到指针，例如全局静态——通常不算 bug）。
Cmac: 用 leaks。
# mac 自带，无需安装
MallocStackLogging=1 leaks --atExit -- /tmp/leak

# 输出会列出每个泄漏 block 的 stack trace
DPython 端的内存"泄漏"诊断。
# Python 没有真泄漏（GC 兜底），但常见"内存只增不减"是因为
# 引用没断（全局 dict 一直加 key、closure 持有大对象、event handler
# 没解绑等）。用 tracemalloc 定位：

cat > /tmp/tracemalloc_demo.py <<'EOF'
import tracemalloc, time

tracemalloc.start()
cache = []

def work():
    cache.append(bytearray(10 * 1024 * 1024))  # 每次 10MB，永不释放

snap1 = tracemalloc.take_snapshot()
for _ in range(5): work()
snap2 = tracemalloc.take_snapshot()

# 对比两次快照，看新增分配集中在哪一行
top = snap2.compare_to(snap1, 'lineno')
for stat in top[:5]:
    print(stat)
EOF
python3 /tmp/tracemalloc_demo.py
预期：tracemalloc 会精确指出 cache.append(...) 那行是泄漏热点。在生产 LLM 服务里出现"内存随时间线性上涨"，十次有八次是某个 dict / list 越积越大——tracemalloc 是首选工具。objgraph 是补充：能画出对象引用图。

6. LAB 6：综合演练 · 诊断一个内存"过高"的 LLM 服务

把今天 5 个 LAB 串起来。模拟运维场景：业务报告"我们的 LLM 推理服务用了 70GB 内存，是不是泄漏？"——你需要 5 分钟内给出判断。

诊断流程

1外层先看 free -h 和 vmstat 1 5。

关注：available 是否充足？si/so 是否为 0？如果 si/so 非零，性能已经崩了，先扩内存再说。

2定位到具体 PID：ps aux --sort=-rss | head -10。

看 RSS 最大的进程，确认是不是 LLM 服务。注意 VSZ 远大于 RSS 是正常的（mmap 模型权重导致虚拟空间巨大）。

3看进程的 smaps，找出占大头的段。
cat /proc/<pid>/smaps_rollup
# 这个文件是 smaps 的汇总版（更易读）

# 或拆开看：
sudo cat /proc/<pid>/smaps | awk '
/^[0-9a-f]+.*\.safetensors/  { name=$NF; flag=1 }
/^[0-9a-f]+.*\[heap\]/        { name="[heap]"; flag=1 }
/^[0-9a-f]+.*\[stack\]/       { name="[stack]"; flag=1 }
/^[0-9a-f]+/ && !flag         { name="anon"; flag=1 }
/^Rss:/                       { rss=$2; print rss " KB  " name; flag=0 }
' | sort -rn | head
典型分布：safetensors 模型映射 ~14GB（7B fp16）、KV cache（匿名 rw 段）几 GB 到几十 GB、Python 堆几百 MB。如果 [heap] 占大头，才是真"内存泄漏嫌疑"；如果是模型映射或 KV cache，那是设计本就如此。
4用 tracemalloc / Py-Spy / memray 跟踪 Python 端动态。
# memray 是 Bloomberg 出品的 Python 内存 profiler，比 tracemalloc 更直观
pip install memray
memray run my_llm_server.py
memray flamegraph memray-*.bin
# 然后浏览器打开生成的 html，火焰图标识哪行代码持有最多内存
5区分"内存增长"和"内存泄漏"。
现象是否是泄漏常见原因
启动后稳定在 70GB 不变否模型 mmap + KV cache 池预分配（vLLM 默认 gpu_memory_utilization=0.9）
随请求数线性增长，请求结束也不降是session/历史 dict 持续 append，未做 LRU
峰值时高，闲时回落否批处理高峰期 KV cache 占用上去，请求完释放
分配增长但 RSS 不增否Python 内部分配器 free 了但没还给 OS（glibc 默认行为）
SWAP 在涨是危机不管是不是泄漏，立刻处理：扩容 / 限流 / 调小 gpu_memory_utilization
给业务的回答模板：
"70GB 中 14GB 是模型权重的 mmap 映射（多副本进程共享物理页，不是 70GB × N）；50GB 是 vLLM 预分配的 KV cache 池，这是配置项，不是漏；剩下 6GB 是 Python 运行时 + 请求中状态。没有泄漏。如果想降，调小 --gpu-memory-utilization 0.7 或 --max-model-len 8192。"

🔗 与 AI / Agent 安全的连结

1. /proc/[pid]/maps 是 LLM 服务的"X 光片"。安全审计时，你能用 maps 看清楚一个被攻陷的进程真正映射了哪些文件——有没有意外的 .so 被 LD_PRELOAD 注入？模型权重路径是不是被替换？业务 Python 进程是不是悄悄 mmap 了 /etc/shadow？这些只有 maps/smaps 能告诉你。日常巡检脚本应该把 /proc/<pid>/maps 里非白名单文件路径全部告警。

2. epoll 服务模型是 LLM 推理网关的标准形态。LAB 4 你写的 50 行 selectors 回声服务，把它换成"接受流式 HTTP 请求 → 转发到 vLLM → 把 token 流回写客户端"，就是一个最小可用 LLM 网关。高并发 LLM 服务的瓶颈很少在 epoll 层，几乎全在 GPU——这个直觉是后面 Phase 2 必须的。同时单线程事件循环也意味着一个慢回调能阻塞全部连接，这是后面讲 Nginx 反代时要避开的坑。

3. mmap 是模型加载的核心机制，但也是新攻击面。Llama-2 70B 的 safetensors 文件 140GB，全部 mmap 映射进进程。如果攻击者能往磁盘上"换"一份同名但是恶意权重文件（甚至只是悄悄 bit flip 几个数），下次 page fault 取页时进程内存里就是篡改后的内容——而内存里看起来还和"合法"hash 一致（因为校验通常只在加载时做一次）。这是模型权重供应链攻击在内存层面的延伸。防御：用 fsverity 给整个文件做按页校验，或者首次加载后立刻 mlock 全部映射防止换出再换入。

4. valgrind / leaks 不只是抓泄漏，还是"看看程序到底做了哪些 malloc"的窗口。Agent 安全产品常需要对一段未知二进制（比如 LLM 生成的可执行文件）做"无害化执行"，valgrind --tool=memcheck 或更强大的 valgrind --tool=helgrind / callgrind 是低成本的沙箱化分析手段——比起完整的 ptrace 跟踪开销小很多，又能拿到内存访问模式。明天讲 strace 深入和 eBPF 是更生产级的方案。

5. 内存监控是 P99 抖动的第一嫌疑人。LLM 推理服务延迟突然抖动，5 分钟诊断顺序：vmstat 1 看 si/so → 看 GPU 利用率（nvidia-smi） → 看进程 RSS 是否在涨 → 看 smaps 区分是 KV cache 增长还是 Python 堆增长 → 看请求队列长度。si/so 任一非零 = 立刻限流 + 扩容，没有第二条路。Day 9 讲 swap 那段：swap 一开就是性能崩塌，对延迟敏感服务等于"宕机"。
