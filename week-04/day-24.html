1. Nginx 是什么 · master/worker 进程模型

Nginx（读作 "engine-x"）是一个高性能的 HTTP 服务器 + 反向代理 + 负载均衡器。2004 年由俄罗斯工程师 Igor Sysoev 写出来，就是为了解决一个具体问题：C10K 问题——如何让一台机器同时处理一万条并发连接。老一代的 Apache 用「一个连接一个进程/线程」的模型，一万条连接要一万个线程，光是上下文切换（Day 04、Day 05 讲过）和内存就把机器压垮了。Nginx 换了一套完全不同的架构。

启动 Nginx 后，你 ps 会看到它是一组进程，而不是一个：

$ ps -ef | grep nginx
root      1201     1  0  nginx: master process /usr/sbin/nginx
www-data  1202  1201  0  nginx: worker process
www-data  1203  1201  0  nginx: worker process
www-data  1204  1201  0  nginx: worker process
www-data  1205  1201  0  nginx: worker process

这就是 master/worker 模型，分工非常清晰：

角色数量身份职责
master 进程1 个root（特权）读配置、绑定 80/443 端口、fork worker、平滑重载、监控并重启崩溃的 worker。不处理任何用户请求。
worker 进程通常 = CPU 核数非特权用户（www-data/nginx）真正处理所有连接与请求。彼此独立、无共享状态（除少量共享内存）。

类比：master 是餐厅经理，只做开门、排班、盯着谁病了顶上；worker 是服务员，每人负责一片桌子端菜。经理不端盘子，服务员不管排班。这种分离带来两个直接好处：

  • 安全：只有 master 用 root 去绑定特权端口（1024 以下需要 root），worker 全部降权成普通用户运行。就算某个 worker 被攻破，攻击者拿到的也只是 www-data 权限，碰不到 root——这正是 Day 06 讲的最小权限原则的教科书应用。
  • 可靠 + 可平滑升级：某个 worker 崩了，master 立刻 fork 一个新的补上，服务不中断。改配置后执行 nginx -s reload，master 用新配置启新 worker、让老 worker 处理完手头连接再退出——重载配置全程零丢包。

worker 数量由 worker_processes 指令控制，惯例设成 auto（等于 CPU 核数）。为什么正好是核数？因为每个 worker 是单线程的，一个核跑一个 worker，既跑满所有核，又避免 worker 之间抢核导致的上下文切换。这与你熟悉的「多开线程池」思路截然相反——Nginx 的哲学是进程数固定且少，靠单进程内的事件循环榨干每一个核。这就引出了第二节的核心问题。

2. 为什么 Nginx 这么快：事件驱动 + epoll

一个 worker 进程、单线程，怎么可能同时伺候上万条连接？如果你用 Python 的思路写一个服务器：

# 传统「阻塞式」写法——每条连接占一个线程
def handle(conn):
    data = conn.recv(4096)   # 这里会「阻塞」，线程卡住直到数据到达
    process(data)
    conn.send(response)

for conn in accept_loop():
    threading.Thread(target=handle, args=(conn,)).start()  # 一连接一线程

问题在 conn.recv() 这一行：网络数据没到时，这个线程什么也做不了，只能干等（阻塞）。一万条连接就要一万个线程干等，绝大多数时间都浪费在「等」上。Nginx 不这么干。它用的是 Day 11 你已经学过的 I/O 多路复用（epoll）——一个 worker 的核心就是一个 事件循环（event loop）：

worker 进程内部（单线程事件循环）： ┌─────────────────────────────────────────────┐ │ epoll_wait() ← 一次系统调用，问内核： │ │ 「我盯着的这 8000 条连接里，哪些现在有事？」 │ │ │ │ 内核返回：conn#12 可读, conn#88 可写, │ │ conn#301 有新连接...（就绪列表） │ └───────────────┬─────────────────────────────┘ │ 只处理「就绪」的那几条，不碰其余 ▼ for ev in ready_events: if ev.readable: 读请求、解析、转发给后端 if ev.writable: 把响应写回客户端 │ └──► 处理完立刻回到 epoll_wait，继续下一轮

关键转变：线程再也不会「卡」在某一条连接上。谁有数据就处理谁，没数据的连接只是内核里的一个 fd，不占任何 CPU。回顾 Day 11 的核心结论：epoll 的开销与「就绪的连接数」成正比，而不是与「总连接数」成正比。所以维护 5 万条空闲连接几乎不花 CPU，只花一点内存。这就是单线程扛住 C10K 的秘密：用一个 epoll 循环代替一万个干等的线程。

一句话对比：Apache（prohibit-fork 模型）是「10000 个服务员每人守一桌，多数时间在发呆」；Nginx 是「1 个眼观六路的服务员，哪桌举手就冲哪桌，从不站着发呆」。前者被上下文切换和内存压垮，后者把单核 CPU 榨到极致。这也是为什么 worker_connections（单 worker 能开的连接数）默认能设到 10240 甚至更高。

你在 Day 22 已经知道，LLM 流式响应（SSE）会让每条连接长时间挂着（几十秒地吐 token）。这种「连接多、每条又慢又长」的场景，正是 Nginx 事件驱动模型的主场——挂着的连接不占线程，只占 fd。

3. 配置文件的层级结构：上下文与指令继承

Nginx 的配置（默认 /etc/nginx/nginx.conf）之所以让新手崩溃，是因为没抓住它的树状嵌套结构。它由两种东西组成：指令（directive）和上下文（context / block）。指令是「键 值;」一行一条；上下文是用 { } 括起来的一块，可以嵌套。核心的层级从外到内是：

# ==== main 上下文（文件最外层，全局设置）====
user  www-data;
worker_processes  auto;          # ← 第 1 节讲的 worker 数

events {                          # ==== events 上下文：连接处理方式 ====
    worker_connections  10240;   # ← 第 2 节讲的单 worker 连接上限
    use epoll;                   # Linux 上用 epoll（通常自动选好）
}

http {                           # ==== http 上下文：所有 HTTP 相关配置 ====
    include       mime.types;
    gzip on;                     # ← 在 http 层开，下面所有 server 都继承

    upstream llm_backend {       # ==== upstream 上下文：定义一组后端 ====
        server 127.0.0.1:8000;
        server 127.0.0.1:8001;
    }

    server {                     # ==== server 上下文：一个「虚拟主机」====
        listen       443 ssl;
        server_name  api.example.com;

        location /v1/ {          # ==== location 上下文：按 URL 路径匹配 ====
            proxy_pass http://llm_backend;
        }
        location /healthz {
            return 200 "ok";
        }
    }
}

把它想象成一棵树：main → events / http → server → location。理解配置的唯一心法是这条规则：

指令向内层「继承」，内层可以「覆盖」外层。你在 http 层写 gzip on;，则所有 server 和 location 默认都开了 gzip；某个 location 里写 gzip off; 就单独关掉它。

各层的语义：http 是所有 HTTP 配置的总容器；一个 server 块 = 一个虚拟主机，靠 listen（端口）+ server_name（域名）来区分——同一个 80 端口上，Nginx 靠请求里的 Host 头把请求分发给对应的 server；location 则按 URL 路径进一步匹配，是你写规则最多的地方。location 的匹配有优先级：=（精确）> ^~（前缀且不再正则）> ~/~*（正则/忽略大小写正则）> 普通前缀。改完配置后务必两步走：

sudo nginx -t          # 先测试配置语法是否正确（生产环境救命命令）
sudo nginx -s reload   # 语法 OK 再平滑重载（第 1 节讲的零丢包重载）

4. 正向代理 vs 反向代理

「代理」就是中间人：客户端和服务器不直接说话，中间夹一层替它们转发。方向不同，名字就不同，这是最容易搞混的一对概念。

正向代理（forward proxy）—— 代理「客户端」，服务器不知道真实客户是谁 你的公司内网 [员工浏览器] ──► [公司代理服务器] ──► [google.com] (翻墙梯子/科学上网也是这类) 服务器看到的来源是「代理」，客户端主动配置要走这个代理 反向代理（reverse proxy）—— 代理「服务器」，客户端不知道真实后端是谁 [用户浏览器] ──► [Nginx 反向代理] ──┬─► [后端 A: vLLM :8000] (公网 443) (api.example.com) ├─► [后端 B: vLLM :8001] └─► [后端 C: Python API] 用户以为只在跟 Nginx 说话，完全看不到背后有几台后端

一句话记忆：正向代理站在客户端这边、替客户端出去；反向代理站在服务器这边、替服务器接客。判断标准是「代理离谁近、替谁隐身」。你做 LLM 服务用到的几乎全是反向代理——Nginx 挡在最前面，对外暴露一个统一的 `api.example.com:443`，背后可能是一群 vLLM 实例、几个 Python 网关。反向代理带来的价值：

  • 隐藏与保护后端：后端 IP、端口、拓扑对外全不可见，攻击面收窄到 Nginx 一层。
  • 统一入口做横切：TLS 终止、限流、缓存、压缩、鉴权、日志，全在这一层集中做，后端只管业务逻辑。
  • 负载均衡与灰度：把流量分发到多个后端（下一节），或按比例切到新版本。

最简反向代理只要一条 proxy_pass：

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:8000;     # 把请求转发给本机 8000 的后端
        proxy_set_header Host $host;           # 把原始 Host 头传给后端
        proxy_set_header X-Real-IP $remote_addr;  # 把真实客户端 IP 告诉后端
    }
}

proxy_set_header 很关键：请求经过代理后，后端直接看到的「来源 IP」是 Nginx 的 IP。要让后端知道真实用户 IP，必须由 Nginx 用 X-Real-IP / X-Forwarded-For 头传过去。这也是安全上的一个坑——后端如果无脑信任 X-Forwarded-For，攻击者可以伪造它来绕过基于 IP 的限流或封禁（这个我们放到连结部分细说）。

5. upstream：反向代理 + 负载均衡

当后端不止一个实例时，用 upstream 块定义一组后端，Nginx 就成了负载均衡器（这与 Day 23 的 L4/L7 负载均衡呼应，Nginx 是典型的 L7 应用层 LB）：

upstream llm_backend {
    # 默认调度算法：加权轮询（weighted round-robin）
    server 10.0.0.1:8000 weight=3;      # 权重 3，性能好的机器多分点
    server 10.0.0.2:8000 weight=1;
    server 10.0.0.3:8000 backup;        # 备用，只有前面全挂了才启用

    # 被动健康检查：某后端连续失败 3 次，摘除 30 秒
    # （写在每个 server 上：max_fails=3 fail_timeout=30s）
}

server {
    location /v1/ {
        proxy_pass http://llm_backend;   # 指向 upstream 的名字，而非具体 IP
        proxy_read_timeout 300s;         # LLM 推理慢，读超时要调大！
    }
}

Nginx 内置几种调度算法（呼应 Day 23）：

算法指令怎么分适用
加权轮询（默认）按权重依次轮流后端无状态、性能相近
最少连接least_conn;发给当前活跃连接最少的后端LLM 场景首选——请求耗时差异大
IP 哈希ip_hash;同一客户端 IP 固定打到同一后端需要会话粘性（sticky session）

对 LLM 服务，least_conn 通常比轮询更合理：一次推理请求可能跑 200ms 也可能跑 30s，轮询会把新请求盲目塞给一个正在跑长任务、已经排满队的后端；最少连接则天然把请求导向最空闲的实例。健康检查也至关重要——Nginx 开源版自带「被动」健康检查（max_fails + fail_timeout）：转发失败到某后端达到阈值，就临时把它摘出去，隔一阵再试。这样某个 vLLM 实例 OOM 崩了，Nginx 会自动绕开它，用户几乎无感。

6. gzip 压缩与 proxy_cache 缓存

这两个都是「省带宽 / 提速度」的经典手段，在 Nginx 里各一行开关背后是重要的机制。

gzip 压缩

gzip 让 Nginx 在把响应发回客户端前先压缩，文本类内容（JSON、HTML、JS、CSS）通常能压到原来的 20%~30%。它靠 HTTP 的内容协商完成：浏览器在请求头带 Accept-Encoding: gzip 表示「我能解压」，Nginx 压缩后在响应头回 Content-Encoding: gzip。

http {
    gzip on;
    gzip_types application/json text/plain text/css application/javascript;
    gzip_min_length 1024;     # 小于 1KB 的不压（压缩本身也有 CPU 开销）
    gzip_comp_level 5;        # 压缩级别 1~9，越高越小但越费 CPU，5 是平衡点
}
LLM 服务的一个反直觉点：SSE 流式响应不要开 gzip。gzip 需要攒够一块数据才能压缩输出，这会让「逐 token 吐出」的流式效果被缓冲住、卡顿。流式接口通常显式关掉 gzip 并设 proxy_buffering off;，让 token 一产生就立即透传给用户——这是保证「打字机效果」流畅的关键配置。

proxy_cache 缓存

反向代理可以把后端的响应缓存在 Nginx 本地，下次相同请求直接由 Nginx 返回，根本不打扰后端。对「结果稳定、重复请求多」的接口（模型列表、静态配置、公开的 embedding 查询）能极大减轻后端压力：

http {
    # 定义一块缓存区：磁盘路径 + 内存里的 key 索引（10MB）+ 最大 1GB
    proxy_cache_path /var/cache/nginx keys_zone=api_cache:10m max_size=1g
                     inactive=60m;

    server {
        location /v1/models {
            proxy_pass http://llm_backend;
            proxy_cache api_cache;
            proxy_cache_valid 200 5m;      # 200 响应缓存 5 分钟
            add_header X-Cache-Status $upstream_cache_status;  # HIT/MISS 可观测
        }
    }
}

响应头里的 X-Cache-Status 会告诉你这次是 HIT（命中缓存，没打后端）还是 MISS（未命中，回源了）。但对 LLM 要极其小心：绝大多数推理请求是「一次一变」的（prompt 不同、带随机采样），缓存命中率极低还可能返回错误的旧结果；缓存只适合真正幂等、无个性化的接口。缓存与安全的交叉点——缓存投毒（cache poisoning）和跨用户缓存泄露——我们放到连结部分。

7. 限流：limit_req（漏桶）与 limit_conn

限流是 Nginx 作为守门人最重要的防御能力，用来抵挡突发流量、爬虫、以及针对 LLM 的资源耗尽攻击。Nginx 提供两个正交的维度。

limit_req：限制「请求速率」（漏桶算法）

limit_req 实现的是漏桶算法（leaky bucket）：请求像水一样倒进桶里，桶底以恒定速率漏出（= 允许的处理速率），桶满了就溢出（多余请求被拒或排队）。

http {
    # 定义一个限流区：以客户端 IP 为 key，平均速率 10 请求/秒，索引占 10MB
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    server {
        location /v1/ {
            limit_req zone=api_limit burst=20 nodelay;
            # rate=10r/s：稳态每秒放行 10 个
            # burst=20  ：允许瞬时积压 20 个的缓冲（应对正常抖动）
            # nodelay   ：burst 内的请求立即处理，不强行排队等待
            proxy_pass http://llm_backend;
        }
    }
}

三个参数的配合是精髓：rate 是长期平均速率，burst 是允许的短时突发桶容量，nodelay 决定突发的请求是「立即放行」还是「匀速排队」。超出 burst 的请求会被返回 503。$binary_remote_addr 是用二进制形式存客户端 IP（比字符串省内存），10MB 的 zone 大约能存 16 万个 IP 的状态。

limit_conn：限制「并发连接数」

limit_conn 限制的是「同一时刻」某个 key 的活跃连接数，与速率无关：

http {
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        location /v1/chat {
            limit_conn conn_limit 5;   # 每个 IP 最多 5 条并发连接
            proxy_pass http://llm_backend;
        }
    }
}

两者解决不同问题：limit_req 防「打得太快」，limit_conn 防「占得太多」。对 LLM 流式接口，limit_conn 尤其有用——单条 SSE 连接可能挂 30 秒并持续消耗一个后端的推理槽位，限制每个 IP 的并发连接数，能防止单个用户开几百条流式连接把你的 GPU 推理槽全占满（这是一种典型的慢速资源耗尽攻击）。生产实践通常两者叠加：既限速率又限并发。

8. SSL / TLS 终止（SSL termination）

SSL 终止（也叫 TLS termination / TLS offloading）指的是：加密的 HTTPS 连接在 Nginx 这一层被解密，Nginx 到后端之间走明文 HTTP。回顾 Day 18 的 TLS 知识——TLS 握手、证书验证、对称加解密都是有成本的 CPU 密集操作，把它集中在 Nginx 层做，后端 Python/vLLM 就只处理明文、专注业务。

用户 Nginx（TLS 终止点） 后端 [浏览器] ══HTTPS/443══► [解密 · 验证 · 加密] ──HTTP──► [vLLM :8000] (TLS 加密隧道) 证书 & 私钥在这里 (内网明文，同一可信网络) ↑ 加密到此为止
server {
    listen 443 ssl http2;              # 443 端口，启用 SSL 和 HTTP/2
    server_name api.example.com;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;   # 证书链
    ssl_certificate_key /etc/nginx/certs/privkey.pem;     # 私钥（严格权限 600!）
    ssl_protocols       TLSv1.2 TLSv1.3;   # 禁用老旧不安全的 TLS 1.0/1.1
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        proxy_pass http://127.0.0.1:8000;  # 注意是 http://，到后端已是明文
    }
}

# 把所有 80 的明文请求 301 跳转到 443
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}

SSL 终止的好处：证书和私钥集中管理在一处（续期、轮转只改 Nginx），后端无需各自配 TLS；CPU 密集的加解密集中优化。它的安全前提是：Nginx 到后端的那段明文链路必须在可信网络内（同一台机器的 loopback、或同一 VPC 的内网）。如果这段链路会跨越不可信网络，就要升级成 proxy_pass https://... 做「TLS 重加密」，或按 Day 18 讲的用 mTLS（双向 TLS）保证后端只接受来自可信 Nginx 的连接——这正是 Week 16 零信任 / service mesh 会展开的话题。

🔗 与 AI / Agent 安全的连结

1. Nginx 是 LLM 服务的第一道安全闸门。你的 vLLM/推理网关不应该直接裸奔在公网。把 Nginx 放在最前面，就把「限流、TLS、鉴权、日志、隐藏后端」这些横切安全能力收敛到一层。攻击者扫描到的只有 Nginx，看不到后端拓扑、端口、版本——攻击面从「一群后端」收窄成「一层代理」。设计 Agent 安全产品的入口层时，这是默认起点。

2. limit_req / limit_conn 是抵御 LLM 资源耗尽攻击的第一层。Day 01 就埋过伏笔：攻击者用超长 prompt 或高频请求把你的 GPU 推理槽、显存打满，让正常用户无法服务（一种 DoS）。在 Nginx 层用 limit_req 卡请求速率、limit_conn 卡并发连接，是最廉价、最不可绕过的第一道闸——因为它在请求还没到达昂贵的推理层之前就拦掉了。但要记住这只是「第一层」：真正的 token 级配额、按用户的成本控制，还需要在应用层（Day 156 推理网关）做。

3. X-Forwarded-For 伪造会击穿你的 IP 限流。第 4 节埋的坑在这里引爆：如果你的限流或封禁是基于客户端 IP，而后端/Nginx 无脑信任请求头里的 X-Forwarded-For，攻击者只要每次请求伪造一个不同的 XFF 值，就能让每个「假 IP」都刚好卡在限流阈值以下，绕过整个限流体系。正确做法：只信任由你自己可信代理链设置的 XFF（用 real_ip_header + set_real_ip_from 指定可信来源），对外部传入的 XFF 一律覆盖或忽略。这是 Agent 网关做速率控制时必查的一个点。

4. proxy_cache 的跨用户泄露与缓存投毒。如果你给带鉴权/个性化的 LLM 接口错误地开了缓存，用户 A 的响应可能被缓存后返回给用户 B——跨用户数据泄露。更隐蔽的是「缓存投毒」：攻击者构造一个带特殊头的请求，让 Nginx 缓存下一个被污染的响应，之后所有用户都拿到这个坏结果。原则：任何带用户身份、带敏感数据的接口默认不缓存；要缓存必须把用户身份纳入 cache key，并严格控制哪些请求头参与缓存。

5. SSL 终止后的「内网明文」是常被忽视的暴露面。很多人以为「上了 HTTPS 就安全了」，但 SSL 在 Nginx 终止后，Nginx→后端那段是明文 HTTP。如果 Agent 系统里有一个被攻破的 sidecar、或攻击者拿到了内网某个立足点，就能在这段明文链路上嗅探到完整的 prompt、模型输出、甚至用户会话。做威胁建模（Day 167 会正式学 STRIDE）时，这段「可信内网明文」必须显式标注为一个信任边界——高敏场景要用 mTLS 把它也加密上。

115 分钟 · 动手：起一个反向代理 + 限流

用 Docker 跑一个 Nginx，把它配成一个带限流的反向代理，后端用 Python 一行起个假服务。

# 终端 A：起一个假后端（返回自己的时间戳）

# 写一个最小配置 mini.conf（放当前目录）：
cat > mini.conf <<'EOF'
events {}
http {
  limit_req_zone $binary_remote_addr zone=t:10m rate=2r/s;
  server {
    listen 80;
    location / {
      limit_req zone=t burst=3 nodelay;
      proxy_pass http://host.docker.internal:8000;
    }
  }
}
EOF

# 终端 B：用官方 nginx 镜像挂载这个配置
docker run --rm -p 8080:80 -v "$PWD/mini.conf:/etc/nginx/nginx.conf:ro" nginx
