1. REST：把一切看成「资源」

REST（Representational State Transfer，表现层状态转移）不是一个协议，而是 Roy Fielding 在 2000 年博士论文里提出的一组架构风格约束。它复用 HTTP 已有的语义，核心思想只有一句话：把系统里的每个东西都建模成「资源」，用 URL 定位它，用 HTTP 动词操作它。

类比：把后端想象成一个图书馆。每本书是一个资源，有固定地址（URL，如 /books/42）。你不会说「请帮我执行借书函数」，而是说「对这本书的位置做某个标准动作」：看（GET）、新增（POST）、整本替换（PUT）、改一部分（PATCH）、撤掉（DELETE）。动作是固定那几个，资源是无穷的——这就是 REST 的优雅之处。

动词语义幂等?例子
GET读取资源是GET /users/7
POST新建（在集合下）否POST /users
PUT整体替换/创建是PUT /users/7
PATCH局部更新否PATCH /users/7
DELETE删除是DELETE /users/7

幂等（idempotent）是个关键概念：同一个请求发 1 次和发 N 次，服务器最终状态一样。这直接决定了「失败能不能安全重试」——GET/PUT/DELETE 可以无脑重试，POST 不行（会重复下单）。REST 的另外两条重要约束：

  • 无状态（stateless）：服务器不在两次请求之间记住客户端是谁。每个请求自带全部上下文（如 token）。好处是任意一台服务器都能处理任意请求 → 水平扩展简单。
  • 用状态码表达结果：2xx 成功、3xx 重定向、4xx 客户端错（401 未认证、403 没权限、404 没找到、429 限流）、5xx 服务器错。
# 一个典型 REST 交互（curl）
curl -X POST https://api.example.com/v1/users \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"Ada","email":"ada@x.io"}'
# → 201 Created  + Location: /v1/users/7

REST 的代价：用文本 JSON 传输（冗余、慢），没有强类型契约（字段拼错了运行时才发现），一个页面常要打多个接口（over/under-fetching）。这三条恰好是后面 gRPC 和 GraphQL 要解决的痛点。

2. HTTP/2：gRPC 的地基

要理解 gRPC，必须先懂 HTTP/2（Day 17 已概览，今天补足细节）。HTTP/1.1 的致命问题是队头阻塞（head-of-line blocking）：一个 TCP 连接上同一时刻只能跑一个请求-响应，前面的慢请求会卡住后面所有请求。浏览器只好开 6 个并行连接硬撑。

HTTP/2 在同一个 TCP 连接上引入了流（stream）+ 帧（frame）的概念：

一条 TCP 连接 ├── Stream 1 ┃HEADERS┃ → ┃DATA┃ ┃DATA┃ (请求 A) ├── Stream 3 ┃HEADERS┃ → ┃DATA┃ (请求 B) └── Stream 5 ┃HEADERS┃ ┃DATA┃ → ... (请求 C) ↑ 多个 stream 的帧交错(multiplexing)在同一连接上飞
  • 多路复用：一条连接同时跑多个独立的 stream，互不阻塞（在应用层解决了队头阻塞，TCP 层仍有，由 HTTP/3 解决）。
  • 二进制分帧：不再是纯文本，而是结构化的二进制帧，解析更快、更省。
  • HPACK 头压缩：HTTP 头里大量重复（如 User-Agent），HPACK 用动态表只传增量。
  • 服务器推送 / 流式：服务器可在一个 stream 上持续推 DATA 帧——这正是 gRPC streaming 的物理基础。

记住这条因果链：gRPC 的双向流、客户端流，本质都是「一个 HTTP/2 stream 上持续发多个 DATA 帧」。没有 HTTP/2 就没有 gRPC 的流式能力。

3. gRPC + Protobuf：强类型的二进制 RPC

RPC（Remote Procedure Call，远程过程调用）的理想是：调远程服务就像调本地函数一样自然，网络细节藏起来。gRPC 是 Google 开源的现代 RPC 框架 = HTTP/2 传输 + Protobuf 序列化 + 代码生成。

Protobuf（Protocol Buffers）是核心：你先在 .proto 文件里用 IDL（接口定义语言）写下契约——有哪些消息、哪些方法。然后 protoc 编译器为你的语言（Python/Go/Java...）自动生成 stub 代码。

// greeter.proto —— 单一事实来源，前后端共享
syntax = "proto3";
package greet;

message HelloRequest  { string name = 1; }   // 字段后的数字是「字段编号」
message HelloReply    { string message = 1; }

service Greeter {
  rpc SayHello (HelloRequest) returns (HelloReply);
}

那个 = 1 不是默认值，而是字段在二进制流里的标识号（tag）——序列化时不传字段名，只传 tag + 值，所以又小又快。字段名可以改，tag 一旦上线绝不能改，这是向后兼容的关键。

维度REST + JSONgRPC + Protobuf
编码文本，人可读二进制，紧凑快
契约靠文档/口头约定.proto 强类型，编译期检查
传输HTTP/1.1 或 2必须 HTTP/2
流式弱（需 SSE/WS）原生四种模式
浏览器直连原生支持需 grpc-web 代理
典型场景对外开放 API内部微服务互调

一句话选型：对外给浏览器/第三方用 REST，内部服务之间高性能互调用 gRPC。

4. gRPC 的 4 种调用模式

这是 gRPC 相对 REST 的杀手锏。借助 HTTP/2 的流，gRPC 支持 4 种通信形状：

① Unary（一元） Client ──req──▶ Server Client ◀─resp── Server ② Server streaming Client ──req──▶ Server Client ◀─r1─r2─r3── Server ③ Client streaming Client ─r1─r2─r3─▶ Server Client ◀─resp── Server ④ Bidirectional（双向） Client ◀─r/r/r─▶ Server （两边同时随便发）
  • Unary：最常见，等价于普通函数调用。如「查用户信息」。
  • Server streaming：一次请求，服务器持续返回多个响应。如「订阅一支股票的实时行情」「LLM 逐 token 返回生成结果」。
  • Client streaming：客户端持续上传，服务器最后给一个汇总响应。如「分块上传大文件」「批量埋点上报」。
  • Bidirectional streaming：两端独立地随时收发。如「实时语音转写」「多人协作」。
# Python gRPC server streaming —— 逐条 yield 即可
def Generate(self, request, context):
    for tok in llm.stream(request.prompt):
        yield GenReply(token=tok)       # 每 yield 一次 = HTTP/2 一个 DATA 帧

注意这里的 yield——server streaming 在 Python 里就是把 RPC 方法写成生成器。这与你做 LLM 流式输出的心智模型完全一致：模型每解出一个 token，就 yield 一帧出去。

5. GraphQL：让客户端决定要什么字段

REST 的痛点之一是 over/under-fetching：移动端首页只要用户的「昵称 + 头像」，但 GET /users/7 把 30 个字段全返回了（over-fetching）；或者一个页面要拼用户 + 订单 + 推荐，得打 3 个接口（under-fetching）。GraphQL（Facebook 2015）反转了控制权：服务器只暴露一个端点 /graphql，由客户端用查询语言精确声明「我要哪些字段」。

# 客户端发一个查询，服务器返回结构完全一致的 JSON
query {
  user(id: 7) {
    name
    avatar
    orders(last: 3) { id, total }   # 一次拿到嵌套数据，不用多打接口
  }
}

类比：REST 像点套餐（菜单固定，多了浪费、少了不够）；GraphQL 像自助餐报菜单（要什么夹什么，一趟取齐）。代价是：服务器端解析查询更复杂；缓存比 REST 难（URL 不再天然是缓存键）；一个恶意深层嵌套查询可能拖垮后端（后面安全连结会讲）。GraphQL 适合字段需求多变、前端形态多样（Web/iOS/Android 各要不同字段）的场景，对 LLM 服务这种「一个 prompt 进、一段文本出」的简单接口反而是杀鸡用牛刀。

6. WebSocket：全双工长连接

到目前为止的协议（REST/gRPC unary/GraphQL）都是客户端发起的。但有些场景需要服务器主动找客户端：聊天消息到达、协作文档别人改了。HTTP 本身做不到（服务器不能凭空找客户端）。WebSocket 解决这个问题。

它的精妙之处在于「借壳上市」：先发一个普通的 HTTP 请求带特殊头，请求「升级」协议；服务器同意后，这条 TCP 连接就脱离 HTTP，变成一条持久的、全双工的字节通道，双方可随时互发消息。

GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket            ← 关键：请求升级
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZQ==

HTTP/1.1 101 Switching Protocols   ← 服务器同意，从此不再是 HTTP
Upgrade: websocket
  • 全双工：两端同时收发，无需谁先请求。
  • 长连接：一次握手，连接一直在，省去反复建连开销，延迟低。
  • 代价：服务器要为每个连接维护状态（占内存），负载均衡和水平扩展更麻烦（连接有粘性），要自己处理心跳保活和断线重连。

适合真正双向、低延迟的场景：在线游戏、协作编辑、实时聊天。但如果只是「服务器单向往客户端推」，WebSocket 就过重了——这就引出今天的高潮。

7. SSE 与 LLM 流式：服务器单向推送

SSE（Server-Sent Events，服务器发送事件）是被低估的协议。它就是一个永不结束的 HTTP 响应：客户端发一个普通 GET，服务器保持连接打开，用特殊的 text/event-stream 格式持续往下写数据，每条以 data: 开头、空行分隔。

HTTP/1.1 200 OK
Content-Type: text/event-stream

data: 你好

data: ，世界

data: [DONE]

这正是 ChatGPT 那个「逐字蹦出」效果的底层。为什么 OpenAI/Anthropic 的流式 API 用 SSE 而不是 WebSocket？因为 LLM 生成是严格单向的：客户端发一次 prompt，之后只需要服务器一个 token 一个 token 地推回来，客户端不需要中途插话。这正好是 SSE 的形状。SSE 相比 WebSocket 的优势：

对比SSEWebSocket
方向仅服务器→客户端双向
底层普通 HTTP，走标准 HTTP/2升级后脱离 HTTP
断线重连浏览器自动重连（自带 Last-Event-ID）需自己实现
基础设施兼容对代理/CDN/防火墙友好常需特殊配置
实现复杂度低（就是流式 HTTP）高（连接状态管理）

最后对比长轮询（long polling）——SSE 出现前的「土办法」推送：客户端发请求，服务器故意不立刻回，挂起等到有数据才返回；客户端收到后立刻再发下一个请求。它用「反复的请求-响应」模拟推送，缺点是每条消息都要一次完整 HTTP 往返、延迟高、头开销大。今天它基本被 SSE/WebSocket 取代，但在不支持流式的老环境里仍是兜底方案。

一图选型：一次请求一次答 → REST / gRPC unary；内部服务高性能互调 → gRPC；前端字段需求多变 → GraphQL；服务器单向流式推（LLM 输出、进度条）→ SSE；双向实时对话（聊天、协作）→ WebSocket；环境太老啥都不支持 → 长轮询 兜底。

与 AI / Agent 安全的连结

① LLM 流式接口的截断与提前终止攻击。你的推理服务用 SSE 往外推 token。如果客户端在收到敏感内容前主动断开连接，服务端是否停止了昂贵的 GPU 生成？反过来，攻击者可以建立大量 SSE 连接但极慢消费（slowloris 变种），每条连接长时间占用一个推理 slot，造成资源耗尽 DoS。流式接口的连接生命周期管理是 LLM 服务的独特攻击面。

② gRPC 内部信任边界的滥用。微服务之间常用 gRPC 互调，且因为「在内网」而省略认证。一旦攻击者通过某个入口服务（如对外的 REST 网关）实现 SSRF 或落地，就能直接调用内部 gRPC 服务——这些服务往往假设调用方可信，缺少鉴权和限流。你的 Agent 工具调用后端如果走 gRPC，务必做服务间 mTLS（Day 18 学的双向认证）。

③ GraphQL 的查询深度炸弹。GraphQL 允许客户端构造任意嵌套查询，攻击者可以写一个 user → friends → friends → friends ... 的深层递归查询，单个请求就让服务器执行指数级的数据库查询而崩溃。Agent 如果暴露 GraphQL 接口给 LLM 自主调用，必须设查询深度限制、复杂度评分、超时——否则一个被注入的 prompt 就能让 Agent 自己打垮后端。

④ Protobuf 契约即攻击面文档。.proto 文件精确描述了所有内部 RPC 方法和字段，对防御方是清晰的资产清单，对攻击者则是现成的攻击地图。如果 gRPC server 开了 reflection（反射，允许运行时查询有哪些方法），攻击者无需源码就能枚举所有接口。生产环境应关闭 reflection，并把 .proto 当作敏感资产管理。

⑤ WebSocket 绕过 HTTP 安全控制。很多 WAF、鉴权中间件、CORS 策略是针对 HTTP 请求设计的，WebSocket 升级后脱离 HTTP，这些控制可能失效。「跨站 WebSocket 劫持」（CSWSH）就是利用 WebSocket 握手不受 CORS 限制、且自动带 Cookie 的特性。Agent 若用 WebSocket 与前端通信，需在握手阶段校验 Origin 和 token，不能依赖浏览器的同源策略。

今日小练习（3 道）

1动手：用 Python 标准库写一个极简 SSE 服务器，每秒推一行时间戳，浏览器访问能看到逐行出现。提示骨架如下，补全并用浏览器打开 http://localhost:8000 观察：

from http.server import BaseHTTPRequestHandler, HTTPServer
import time
class H(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(200)
        self.send_header("Content-Type", "text/event-stream")
        self.end_headers()
        for i in range(10):
            self.wfile.write(f"data: tick {i}\n\n".encode())
            self.wfile.flush()      # 关键：flush 才会立刻推出去
            time.sleep(1)
HTTPServer(("", 8000), H).serve_forever()

再用 curl -N http://localhost:8000 对比，体会 -N（禁用缓冲）的作用。

2观察：打开任意一个支持流式输出的 LLM 网页（或你公司的推理服务），按 F12 → Network 面板，找到那个一直 pending 的请求。查看它的 Content-Type 是不是 text/event-stream，点开 EventStream/Response 标签看 token 是怎么一条条到达的。它是 SSE 还是 WebSocket（看协议列是否为 ws）？

3思考：假设你要为公司的 Agent 平台设计「Agent 实时执行进度」功能——用户提交任务后，要在前端看到「正在搜索 → 正在调用工具 → 正在生成回答」的实时进度，同时用户可以随时点「停止」中断任务。这个场景该选 SSE 还是 WebSocket？为什么？（提示：「随时停止」这个需求改变了通信的方向性——它还是纯单向的吗？）
