0. 今天要搭的架构

先看全景。我们要在一台 mac 上启动三个进程，用三种不同的通信方式把它们连成一条链路。这正是真实 LLM 推理服务的极简骨架：

┌──────────────────┐ curl / 浏览器 ──────▶ │ Nginx :8080 │ 反向代理 · 限流 · SSE 不缓冲 └────────┬─────────┘ │ HTTP/1.1（明文） ┌────────▼─────────┐ │ gateway.py :8000 │ Flask 门面 │ POST /v1/predict│ ← 一元 │ POST /v1/stream │ ← SSE 流式（逐 token 吐） └────────┬─────────┘ │ gRPC（HTTP/2 + protobuf 二进制） ┌────────▼─────────┐ │ server.py :50051 │ gRPC 推理服务（假装是 vLLM） │ Predict() │ ← 一元 RPC │ PredictStream() │ ← 服务端流式 RPC └───────────────────┘

三段链路各用了本周学的一种协议范式：浏览器↔Nginx↔门面走人类友好的 HTTP/JSON；门面↔推理服务走机器友好的 gRPC（HTTP/2 多路复用 + Protobuf 紧凑二进制）；而流式那条线用 SSE 把 gRPC 的 server-streaming 一路透传到浏览器。做一遍你就理解了：为什么内部服务间用 gRPC、对外用 HTTP/SSE——这不是随便选的，是各自协议特性决定的。

1. LAB 1：写第一个 .proto 并生成 stub

gRPC 的哲学是「接口即契约，契约即代码」。你不写 JSON 文档靠嘴约定字段，而是写一份 .proto 文件——它同时是文档、数据格式定义、和代码生成的源头。写完用编译器一键生成两端的类型化代码，客户端和服务端永远对得上。

LAB 1 步骤

A新建 inference.proto，定义一个「推理服务」。
// inference.proto
syntax = "proto3";

package inference;

// service = 一组 RPC 方法的集合（相当于一个「类」）
service Inference {
  // 一元 RPC：一个请求 → 一个响应
  rpc Predict (PredictRequest) returns (PredictReply);
  // 服务端流式 RPC：一个请求 → 一串响应（关键字 stream）
  rpc PredictStream (PredictRequest) returns (stream Token);
}

// message = 一个结构体。等号后面的数字是「字段编号」(tag)，不是默认值！
message PredictRequest {
  string prompt = 1;
  int32  max_tokens = 2;
}

message PredictReply {
  string text = 1;
  int32  tokens_used = 2;
}

message Token {
  string piece = 1;   // 一个 token 片段
  int32  index = 2;   // 它是第几个 token
}
关键理解：字段后面的 = 1 = 2 是字段编号（tag），不是赋默认值。Protobuf 在网络上传输时不传字段名，只传「编号 + 值」，这就是它比 JSON 省一半以上字节的原因（JSON 每条消息都要重复传 "prompt": 这样的键名）。编号一旦上线就永不能改——改了老客户端就解析错乱，这是 protobuf 向后兼容的铁律。
B用 grpc_tools.protoc 编译，生成 Python stub。
python3 -m grpc_tools.protoc -I. \
    --python_out=. \
    --grpc_python_out=. \
    inference.proto

# 看生成了什么
ls -1 inference_pb2*.py
预期输出：多出两个文件——inference_pb2.py（消息类：PredictRequest、Token 等）和 inference_pb2_grpc.py（服务的骨架 InferenceServicer 和客户端 InferenceStub）。你一行手写代码都没写，两端的类型定义就都有了。打开 inference_pb2_grpc.py 扫一眼，你会看到里面已经把序列化/反序列化函数都接好了。

2. LAB 2：gRPC server + client 跑起来

有了 stub，服务端只要「填空」——继承生成的 InferenceServicer，实现里面的方法。我们不接真模型，用一个假的「echo 推理」占位（真实场景这里就是调 vLLM）。

LAB 2 步骤

A写服务端 server.py。
# server.py
import time
from concurrent import futures
import grpc
import inference_pb2
import inference_pb2_grpc

class InferenceServicer(inference_pb2_grpc.InferenceServicer):
    # 一元 RPC：填这个方法
    def Predict(self, request, context):
        text = f"echo「{request.prompt}」"
        return inference_pb2.PredictReply(text=text, tokens_used=len(text))

    # 服务端流式 RPC：用 yield 一次吐一个 token
    def PredictStream(self, request, context):
        words = f"这是对『{request.prompt}』的流式回答 结束".split()
        for i, w in enumerate(words):
            time.sleep(0.3)                       # 模拟逐 token 生成的延迟
            yield inference_pb2.Token(piece=w + " ", index=i)

def serve():
    # gRPC 用线程池并发处理请求；HTTP/2 单连接可多路复用
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    inference_pb2_grpc.add_InferenceServicer_to_server(InferenceServicer(), server)
    server.add_insecure_port("[::]:50051")        # 明文，生产要用 TLS
    server.start()
    print("gRPC 推理服务已启动 :50051")
    server.wait_for_termination()

if __name__ == "__main__":
    serve()
B写客户端 client.py，两种调用都试。
# client.py
import grpc
import inference_pb2
import inference_pb2_grpc

with grpc.insecure_channel("localhost:50051") as channel:
    stub = inference_pb2_grpc.InferenceStub(channel)

    # 一元调用：像调本地函数一样
    reply = stub.Predict(inference_pb2.PredictRequest(prompt="你好", max_tokens=50))
    print("[一元] ", reply.text, "| tokens=", reply.tokens_used)

    # 流式调用：返回值是个可迭代对象，服务端 yield 一个你就收一个
    print("[流式] ", end="", flush=True)
    for tok in stub.PredictStream(inference_pb2.PredictRequest(prompt="流式测试")):
        print(tok.piece, end="", flush=True)
    print()
C开两个终端窗口跑起来。
# 终端 1（记得先 source .venv/bin/activate）
python3 server.py

# 终端 2
python3 client.py
预期现象：一元调用瞬间返回 echo「你好」；流式调用会一个词一个词地蹦出来（每个间隔 0.3 秒）——这就是你在 ChatGPT 里看到的打字机效果的服务端本质。注意：stub.Predict(...) 用起来和调本地函数一模一样，但底层已经是一次跨进程的 HTTP/2 网络往返了。这种「远程调用伪装成本地调用」正是 RPC（Remote Procedure Call）的核心卖点。
gRPC vs REST 你刚亲手体会到的差别：① 你没有手写任何"解析 JSON / 拼 URL / 设 header"的代码——契约生成搞定了一切；② 流式是一等公民，一个 stream 关键字就有了，REST 要靠 SSE/WebSocket 额外造；③ 传的是二进制，抓包看不到明文（这既是性能优点，也是调试痛点）。

3. LAB 3：.proto 进阶——四种调用模式

gRPC 一共有四种调用模式，由 .proto 里 stream 关键字加在请求侧还是响应侧决定。你已经玩过前两种，把四种放一起对比记忆：

模式proto 写法典型 LLM 场景
一元 Unaryrpc F(Req) returns (Rep)短文本分类、embedding 单条
服务端流式rpc F(Req) returns (stream Rep)逐 token 生成（最常用）
客户端流式rpc F(stream Req) returns (Rep)上传长音频/大文档分块喂入
双向流式rpc F(stream Req) returns (stream Rep)实时语音对话、多轮 Agent

LAB 3 步骤（观察 protobuf 到底省了多少字节）

A直接在 Python 里量一条消息的二进制大小，和等价 JSON 对比。
python3 - <<'PY'
import json, inference_pb2

msg = inference_pb2.PredictRequest(prompt="hello world", max_tokens=128)
pb_bytes = msg.SerializeToString()                 # protobuf 二进制
js_bytes = json.dumps({"prompt": "hello world",
                       "max_tokens": 128}).encode() # 等价 JSON

print("protobuf:", len(pb_bytes), "字节 →", pb_bytes)
print("json    :", len(js_bytes), "字节 →", js_bytes)
PY
预期：protobuf 大约 17 字节，JSON 大约 40+ 字节。差距来自——JSON 把 "prompt"、"max_tokens" 这些键名当数据反复传，还要传引号、冒号、括号；protobuf 只传「字段编号 1 + 长度 + 内容」。消息越大、字段越多，差距越夸张。这就是内部高频服务间选 gRPC 的硬理由：同样带宽能塞更多请求。
B用 grpcurl（gRPC 版的 curl）在命令行直接调服务，无需写客户端。
brew install grpcurl
# proto3 默认没开反射，用 -proto 指定契约文件即可
grpcurl -plaintext -proto inference.proto \
  -d '{"prompt": "命令行调用", "max_tokens": 20}' \
  localhost:50051 inference.Inference/Predict
意义：gRPC 因为是二进制，不能像 REST 那样直接 curl 调试——这是它最大的工程痛点。grpcurl 就是为解决这个而生的工具。记住这个工具，以后排查内部 gRPC 服务时你会天天用它。

4. LAB 4：套一个 HTTP/SSE 门面

gRPC 好用，但浏览器原生调不了 gRPC（浏览器无法直接控制 HTTP/2 帧，只能走 gRPC-Web + 代理）。所以真实架构里，对外那一层几乎都是 HTTP/JSON。我们写一个 Flask 门面：对内用 gRPC 调推理服务，对外暴露普通 HTTP，并用 SSE（Server-Sent Events）把流式透传给浏览器。

SSE 是本周重点：它就是一个「一直不结束的 HTTP 响应」，服务器持续往里写 data: xxx\n\n 格式的文本，浏览器用 EventSource 逐条收。比 WebSocket 简单得多，且天然适配「服务器单向推、客户端只收」的 LLM 流式场景。

LAB 4 步骤

A写门面 gateway.py。
# gateway.py
import json
import grpc
from flask import Flask, request, Response, stream_with_context
import inference_pb2
import inference_pb2_grpc

app = Flask(__name__)
channel = grpc.insecure_channel("localhost:50051")
stub = inference_pb2_grpc.InferenceStub(channel)

@app.route("/v1/predict", methods=["POST"])
def predict():
    prompt = request.json.get("prompt", "")
    reply = stub.Predict(inference_pb2.PredictRequest(prompt=prompt, max_tokens=100))
    return {"text": reply.text, "tokens": reply.tokens_used}

@app.route("/v1/stream", methods=["POST"])
def stream():
    prompt = request.json.get("prompt", "")
    def gen():
        # 把 gRPC 服务端流 → 逐条转成 SSE 事件
        for tok in stub.PredictStream(inference_pb2.PredictRequest(prompt=prompt)):
            data = json.dumps({"piece": tok.piece, "index": tok.index},
                              ensure_ascii=False)
            yield f"data: {data}\n\n"        # SSE 格式：data: + 两个换行
        yield "data: [DONE]\n\n"             # 约定的结束标记（OpenAI 也这么干）
    return Response(stream_with_context(gen()), mimetype="text/event-stream")

if __name__ == "__main__":
    app.run(port=8000, threaded=True)
B启动门面（server.py 要保持运行），直连 :8000 测试。
# 终端 3
python3 gateway.py

# 终端 4：一元
curl -s -X POST http://127.0.0.1:8000/v1/predict \
  -H "Content-Type: application/json" -d '{"prompt":"你好"}'

# 终端 4：SSE 流式（-N = 关掉 curl 自己的缓冲，才能看到逐条到达）
curl -N -X POST http://127.0.0.1:8000/v1/stream \
  -H "Content-Type: application/json" -d '{"prompt":"流式"}'
预期现象：一元返回一坨 JSON；流式那条会每 0.3 秒蹦出一行 data: {"piece":"...","index":0}，最后一行 data: [DONE]。务必加 -N——不加的话 curl 会把整个流攒完再一次性打印，你就误以为"没流式"。这个 [DONE] 结束标记不是随便写的，OpenAI 的 /v1/chat/completions 流式响应用的就是同一套 SSE + [DONE] 协议。

5. LAB 5：Nginx 反向代理 + SSE 不缓冲的坑

现在把 Nginx 摆到最前面当反向代理。为什么要这层？生产里它负责：TLS 终止、限流、负载均衡多个门面实例、隐藏内部拓扑。今天先做最基础的反代，并踩一个90% 的人第一次做 SSE 都会踩的坑。

LAB 5 步骤

A装 Nginx，找到配置目录（mac homebrew 路径）。
brew install nginx
# Apple Silicon 配置在 /opt/homebrew/etc/nginx/，Intel 在 /usr/local/etc/nginx/
nginx -V 2>&1 | tr ' ' '\n' | grep conf-path   # 确认你的路径
B新建 servers/gateway.conf（Apple Silicon 放 /opt/homebrew/etc/nginx/servers/）。
upstream gateway_pool {
    server 127.0.0.1:8000;      # 有多个门面实例就在这里加几行，自动轮询
}

server {
    listen 8080;

    # 普通接口：正常反代
    location /v1/predict {
        proxy_pass http://gateway_pool;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # SSE 流式接口：必须关闭缓冲！否则 Nginx 会把整个流攒着，一次性吐给客户端
    location /v1/stream {
        proxy_pass http://gateway_pool;
        proxy_buffering off;            # ← 核心：关掉响应缓冲
        proxy_cache off;
        proxy_http_version 1.1;         # SSE 需要 HTTP/1.1 长连接
        proxy_set_header Connection ''; # 清掉 Connection，保持连接不被关
        chunked_transfer_encoding off;
    }
}
C校验配置、重载、测试。
nginx -t                 # 语法检查，必须 syntax is ok
nginx                     # 启动（已启动则用 nginx -s reload）

# 走 Nginx（:8080）测一元
curl -s -X POST http://127.0.0.1:8080/v1/predict \
  -H "Content-Type: application/json" -d '{"prompt":"过网关"}'

# 走 Nginx 测流式——注意观察是逐条还是一次性
curl -N -X POST http://127.0.0.1:8080/v1/stream \
  -H "Content-Type: application/json" -d '{"prompt":"过网关流式"}'
动手验证这个坑：把 proxy_buffering off; 那行注释掉，nginx -s reload，再跑一次流式 curl。你会看到——token 不再逐条蹦出来了，而是等 server 全部生成完，一次性哗啦全出来。原因：Nginx 默认开缓冲，会先把上游响应攒进 buffer 再转发。对 SSE 这种"实时性就是产品体验"的场景，这等于把流式退化成了一元。生产上无数"流式为什么不流式了"的工单，根子都在这一行。验证完记得把它加回去。

6. LAB 6：ab / wrk 压测量化性能

服务能跑 ≠ 服务能扛。上线前必须回答：它每秒能处理多少请求（QPS）？延迟分布怎样？并发拉高会不会崩？ 这就是压测。我们用两个经典工具量化这条链路。

LAB 6 步骤

Aab（Apache Bench，mac 自带）——最快上手。
# 准备请求体
echo '{"prompt":"benchmark"}' > body.json

# -n 总请求数  -c 并发数  -p POST体  -T content-type
ab -n 2000 -c 50 -p body.json -T application/json \
   http://127.0.0.1:8080/v1/predict
怎么读结果：重点看四个数——Requests per second（吞吐/QPS）、Time per request（平均延迟）、Failed requests（失败数，理想是 0）、以及底部的百分位延迟表（50%、95%、99% 各是多少 ms）。P99 比平均值重要得多——平均 20ms 但 P99 500ms，说明有长尾，1% 的用户体验很差。
Bwrk（更现代，多线程，能压出更高负载）。
# 写个 lua 脚本描述 POST 请求
cat > post.lua <<'LUA'
wrk.method = "POST"
wrk.body   = '{"prompt":"benchmark"}'
wrk.headers["Content-Type"] = "application/json"
LUA

# -t 线程数  -c 连接数  -d 持续时间  --latency 打印延迟分布
wrk -t4 -c100 -d15s --latency -s post.lua \
    http://127.0.0.1:8080/v1/predict
对比实验：分别压 :8000（直连门面）和 :8080（过 Nginx），看 QPS 差多少。Nginx 会引入一点点开销，但换来了限流、TLS、负载均衡的能力——这就是工程权衡。再把 -c 从 50 一路加到 500，观察 QPS 到某个点不再上升反而延迟飙升——那个拐点就是你这套配置的容量上限（瓶颈通常是 Flask 的 threaded 线程数 + GIL）。
C顺手加一层 Nginx 限流，再压一次看效果。
# 在 gateway.conf 的 server{} 外层（http 上下文）加：
#   limit_req_zone $binary_remote_addr zone=api:10m rate=100r/s;
# 在 location /v1/predict 里加：
#   limit_req zone=api burst=20 nodelay;
# reload 后用 wrk 高并发压，会看到部分请求返回 503——限流生效了
意义：这就是本周三讲的 limit_req（令牌桶/漏桶）在真机上的效果。对 LLM 服务尤其关键——一个请求可能吃几秒 GPU，不限流的话几十个恶意并发就能把你的推理集群拖垮。你亲手看到 503 出现的那一刻，就理解了"限流是保护后端的最后一道闸"。
收尾清理：nginx -s stop 停 Nginx；两个 Python 进程 Ctrl-C；deactivate 退出虚拟环境。今天的四个进程（server / gateway / nginx + 压测）合起来就是一套最小可用的推理网关雏形，Week 16 讲 API 网关、Week 23 讲推理引擎时会回到它。

🔗 与 AI / Agent 安全的连结

1. gRPC 的"二进制黑盒"是双刃剑。你今天体会到 gRPC 抓包看不到明文——这对性能是优点，但对安全审计是盲区。内部服务间全走 gRPC，传统的 HTTP WAF、日志中间件全都失效，看到的只是一堆二进制。做 Agent 安全产品时，如果要审计"Agent 调了哪个内部工具、传了什么参数"，你得在 gRPC 的 interceptor 层埋点解码，而不是指望网络层抓包。记住 grpcurl 和 interceptor 这两个词，它们是你审计 gRPC 流量的钥匙。

2. SSE 流式是 prompt injection 数据外泄的隐蔽通道。流式响应逐 token 吐、且经常关闭缓冲（你今天亲手关了 proxy_buffering），意味着输出内容是边生成边发出的。如果 Agent 被 indirect injection 诱导去泄露敏感信息，流式模式下数据会在你的输出过滤器还没看到完整响应前就已经发出去了一部分。所以对高敏感场景，"流式 + 输出审查"要么改成攒一段再审再放（牺牲体验），要么在 token 级别做实时检测——这是本周架构直接引出的防御设计难题。

3. Nginx 反代层是限流与身份校验的黄金位置。你今天在 Nginx 加了 limit_req 看到 503。真实的 LLM 网关会在这一层做得多得多：JWT 校验、per-user 配额、按 token 数计费的预扣。把安全策略下沉到网关（而非每个业务代码里重复写），是"安全左移"和"纵深防御"的工程落地。攻击者要打你的推理服务，第一道墙就是这层——它的配置质量直接决定资源耗尽攻击（Day 55 会讲）能不能得逞。

4. 服务间明文通信 = 内网横向移动的高速路。今天为了简单，gRPC 和 HTTP 都用了明文（insecure_channel、listen 8080 无 TLS）。生产里这是大忌：一旦攻击者进了内网（下周六 Day 27 讲反弹 shell 就是干这个的），明文的服务间通信让他能随意嗅探、篡改、重放内部 RPC。这就是为什么零信任架构要求服务间 mTLS（Day 111 专题）——不因为"在内网"就信任任何流量。

5. 压测数字是安全容量规划的基线。你今天用 wrk 找到的"QPS 拐点"，正是防御 DoS 的关键参数。不知道自己能扛多少 QPS，就无法合理设限流阈值：设太松挡不住攻击，设太紧误伤正常用户。Agent 安全里更微妙——LLM 请求的成本极不均匀（一个 8k token 的请求可能是普通请求的百倍开销），所以对 LLM 服务的"压测"和"限流"不能只按请求数算，得按 token 预算算。今天的 QPS 思维是起点，Week 23 会升级到 token-aware 的容量模型。

1动手 · 给链路加一个「慢客户端」并观测背压

把 server.py 里 PredictStream 的 time.sleep(0.3) 改成 0.05，让 token 吐得飞快。然后用一个"慢读"的客户端接收——在 client 的流式循环里加 time.sleep(1)。观察：服务端会不会一口气生成完所有 token，还是被客户端拖着慢慢来？（提示：gRPC 有流控/背压机制，服务端 yield 会被下游消费速度反压。）思考这对"用户关掉了浏览器但 GPU 还在狂算"这个真实浪费场景意味着什么。

2观察 · 对比三种传输的字节数与延迟

同一个 prompt="hello world"，分别测量：① gRPC protobuf 序列化后字节数（LAB 3 已给方法）；② 等价 JSON 字节数；③ 用 ab -n 1000 -c 1 分别压门面的一元接口（JSON over HTTP）。把"字节数"和"平均延迟"列成一张小表。问题：字节数省一半，端到端延迟省了多少？为什么延迟的节省远小于字节的节省？（提示：想想延迟由哪几部分组成——序列化只是其中一小块。）

3思考 · 内部 gRPC，对外 HTTP/SSE，为什么这样分

今天的架构是「内部服务间 gRPC，对外网关 HTTP/SSE」。请从三个维度论证这个划分的合理性：(a) 性能——内部为什么能承受二进制的调试代价？(b) 兼容性——对外为什么不能用 gRPC？(c) 安全——把"面向不可信外部"和"面向可信内部"用不同协议+不同信任级别隔开，对纵深防御有什么好处？用本周学的协议特性支撑你的论证。
