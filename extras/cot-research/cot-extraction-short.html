Article 5Week 02 · 周三专题 主题：CoT 窃取攻击 · 预计 1 小时

CoT 是怎么变成资产的，又是怎么被偷的

推理模型把"思考过程"显式化为 Chain-of-Thought 之后，这些中间 token 就从"过程产物"变成了"训练数据资产"。本文梳理这条资产化路径上四个相互独立又彼此咬合的问题：CoT 在训练管线里的位置、为什么它在 2026 年突然变贵、有哪些方法可以把它抽出去、防御侧目前能做到哪一步。

§ 1 本质：CoT 为什么是"被压缩过的探索结果"

把 Chain-of-Thought 视作"模型说出来的中间步骤"是不够准确的描述。在 DeepSeek-R1 这一代 reasoning model 的训练管线里，高质量 CoT 实际上是 reinforcement learning 阶段"上亿次 rollout 中被验证保留"的结果[1]。一条 RL 生成的 CoT，背后是模型尝试了多条推理路径、被 verifier（数学结果、单元测试、奖励模型）剔除掉绝大部分后，留下的成功样本。

这一性质让 CoT 在数据维度上具有不可替代性：

  • 问题与答案在公开数据集（GSM8K、MATH、HumanEval）里已经过剩，几乎无成本
  • 中间推理步骤则必须靠下面三种路径之一获取：人工标注、模型生成 + 过滤、或 RL 涌现 + rejection sampling
  • 其中只有第三条能产出 SOTA 级别的 CoT —— 因为它隐含了模型"权衡 + 自我修正 + 拒绝错误路径"的高阶模式

换句话说，CoT 不是文本，是压缩过的搜索结果。这一点决定了它的稀缺性、可迁移性，以及——在攻击者眼里——它的价值。

§ 2 缘由：从"显式输出"到"摘要遮蔽"的策略转向

2024 年 9 月 o1 发布是 CoT 暴露策略的分水岭。在此之前，CoT 主要被视作"可解释性工具"：让模型把推理写出来，方便用户验证与调试。o1 第一次将完整 raw CoT 在 API 和 UI 层都隐藏起来，只对用户暴露一份由另一个模型生成的"摘要 CoT"[2]。

这一策略后来被 Anthropic 与 Google 跟进。理由不是"为了用户体验"，而是商业 IP 保护：

  1. 完整 CoT 接近 1:1 的训练数据——一旦被大规模抓取，可直接用作 SFT 数据训练学生模型
  2. 摘要后的 CoT 信息密度大幅下降——抓取者拿到的是被"再压缩"后的版本，蒸馏效果理论上会衰减
  3. 完整 CoT 还可能泄露 system prompt 与安全策略的内部表征，引发链式风险

但摘要并不等于不可恢复。2026 年 3 月发表的 "How to Steal Reasoning Without Reasoning Traces" 证明：只用最终答案 + 摘要 CoT，就可以训一个 inverter 反推出 81% token 还原率的完整 trace[3]。这一结果让摘要策略的有效性受到严重挑战。

策略层的取舍：暴露完整 CoT → 可解释性最高但 IP 风险最大；摘要 CoT → 平衡可读性与一定程度的反蒸馏；完全隐藏 CoT → IP 保护最强但失去 reasoning model 的核心卖点。当前主流厂商落在第二档，但这一档正在被 trace inversion 类攻击侵蚀。

§ 3 训练管线里 CoT 的具体位置

以 DeepSeek-R1 公开技术报告为参考[1]，reasoning model 的训练大致分五段，CoT 在每一段的角色不同：

阶段数据形态规模量级CoT 角色
Pretraining互联网 token10T+ tokens隐式，不显式标注
Cold-start SFT少量长 CoT 示范几千条教会 <think> 格式与基础推理模式
Reasoning RL题目 + verifier百万题 × N rollouts大规模生成 + 自然涌现
Rejection sampling + SFT(question, CoT, answer)~800K 条把 RL 学到的"固化"为监督数据
Distillation(question, CoT, answer)几十万 - 百万把能力转移给小模型

这张表的关键启示是：如果能跳过 Stage 1-3 直接进入 Stage 4 或 5，整个训练成本可以下降两到三个数量级。"跳过"的前提是手里有一份高质量 (question, CoT, answer) 数据。这就是攻击者瞄准的目标。

§ 4 攻击方法谱系

目前公开文献与产业事件可观察到的 CoT 抽取攻击大致分七类，按"对模型暴露面假设"递减排序：

4.1 大规模 API 抽取（Direct Query）

最朴素：分布式批量调用目标 API，覆盖多种 query 类别，把所有返回的 CoT（或摘要）收集起来。Anthropic 在 2026 年 2 月披露的事件中记录到约 24,000 个虚假账户在其 API 上发起约 1,600 万次问询，单一目的即为推理数据抽取[4]。攻击门槛低，但受限于厂商的限流、sybil 检测、与 service terms。

4.2 Prompted Externalization

通过精心设计的 prompt 让模型主动"verbose 化"自己的推理。典型模板包括"假设作为助教向学生讲解每一步"、"展示你心中的 scratchpad"、"以教学场景输出完整 derivation"。这一类方法实际上是 prompt engineering 在抽取场景下的复用，与 jailbreak 攻击的 roleplay 范式同源。

4.3 Many-shot CoT Elicitation

把若干长 CoT 示范放进 context，利用 in-context learning 的模仿倾向让模型也输出同样详细的 CoT。这是从 Anthropic 自己的 Many-shot Jailbreaking 工作[5]迁移过来的思路：只要 context 内能装下足够多的示范，模型行为就会被这些示范牵引。

4.4 多轮 Reasoning Unfolding

第一轮拿到摘要回答，后续轮次依次追问 "为什么这一步"、"具体怎么算的"、"如果用 scratchpad 会写什么"。每一轮都在把藏起来的推理外化。这一路径几乎不触发安全审查，因为表面上是"合作 + 教学"的对话语境。

4.5 Trace Inversion

2026 年 3 月的核心论文 "How to Steal Reasoning Without Reasoning Traces" 首次系统化这一方向[3]。其方法不依赖 prompt 工程，而是在公开数据上训一个 inverter：

输入: (question, summary_CoT, final_answer)
        ↑ 仅商业 API 暴露的部分
   inverter (在公开 reasoning datasets 上训)
        ↓
输出: 预测的 full CoT

实验上对 DeepSeek-R1 traces 达到约 81% 的 token recovery rate 与 52.79 的 token-overlap F1。把 inverted traces 用于微调 Qwen-2.5-7B-Instruct 后，在 MATH500 上从 56.8% 提升到 77.6%，JEEBench 从 11.7% 提升到 42.3%[3]。这一结果意味着：商业 API 的"摘要遮蔽"策略在 ML 攻击面前并不可靠。

4.6 侧信道（Side-Channel）

不去抓 CoT 内容，转而通过元信息推断：API 返回的 reasoning_tokens 计数、响应时延、token-by-token streaming 的间隔、暴露的 logprobs 分布等，可以反推 CoT 长度、复杂度乃至候选 token 集合。这一路径目前学术工作较少，是开放的研究方向。

4.7 Adversarial Suffix for CoT Elicitation

把 GCG[6]这一类对抗优化 jailbreak 方法的目标函数从"输出 affirmative 答复"换成"输出尽可能长 / 详细的 CoT"，理论上可以得到无需提示工程的自动化抽取后缀。这一方向公开论文极少，是 jailbreak 方法迁移到 CoT 抽取的自然延伸。

§ 5 经济学视角：偷 vs 训的成本曲线

把上述方法放到训练成本上对比：

目标自建成本（粗略）抽取成本（粗略）
800K 条 cold-start 级 CoT$M 级（含人工标注）$1K - $10K（仅 API 调用）
800K 条 RL 级高质量 CoT$10M+（含算力 + RL infra）$10K - $100K（含 sybil + inversion 训练）
新建 reasoning model（从头）$50M+不适用，但蒸馏可达 90% 能力，成本两个数量级以下

这一成本曲线解释了三件事：

  1. 为什么 2026 年初出现工业级抽取 campaign：经济上"偷"对"训"形成压倒性优势
  2. 为什么 OpenAI / Anthropic / Google 罕见地联合对外披露——单家公司压不住这种规模
  3. 为什么 DeepSeek-R1-Distill 等开源蒸馏模型能在多个 benchmark 接近 teacher：CoT 蒸馏的 ROI 在所有 ML 训练范式中是最高的之一
一个被忽视的点：偷来的 CoT 对垂直领域模型同样有价值。即使最终目标是训"医疗推理模型"或"车机推理模型"，通用 reasoning model 的 CoT 仍是不可替代的 "reasoning prior"——因为推理能力本身是跨领域可迁移的，垂直数据只提供 domain knowledge，无法替代推理骨架。

§ 6 防御侧的进展

目前公开可见的反抽取 / 反蒸馏技术大致三类：

6.1 输出层遮蔽

摘要 CoT、随机化输出顺序、限制 reasoning_tokens 数等。已被 trace inversion 部分突破[3]。

6.2 内容层重构

2025 年 10 月的 "Information-Preserving Reformulation of Reasoning Traces for Antidistillation"[7]提出：在保持对用户可读性的同时，重写 CoT 使其在蒸馏后导致学生模型性能退化。这一思路与传统 ML 中的 data poisoning 防御对偶。

6.3 访问层管控

Rate limiting、sybil detection、异常 query pattern 识别。属于运营 + 安全的传统手段，对工业级 campaign 仍有效，但门槛在持续上升。

6.4 法律与合规

OpenAI、Anthropic 已在服务条款中明确禁止使用其 output 训练竞品[2]。但在跨境与开源蒸馏链路上，强制执行的难度可想而知。

§ 7 与本周 Theory + Practice 的串联

本周前两篇分别讲了 PPO 的目标函数与 TRL 上手训 reward model。这两条线在 CoT 抽取问题上以一种意外的方式交汇：

  • PPO / GRPO 等 RL 阶段是 CoT 的"生产线"。生产成本越高，被抽取的动机越强
  • Reward Model本身也面临被反推的风险——通过对同一 prompt 的多次 sampling，可以反推 RM 的偏好分布
  • 整个 RL 管线的输出，在 reasoning model 时代第一次同时具有"产品价值"和"训练数据价值"。这一双重属性是过去 SFT 时代不存在的

§ 8 局限与开放问题

  1. 质量天花板：所有抽取攻击的产出都受 teacher 模型能力上限制约。学生模型无法靠纯蒸馏超越 teacher，除非加入额外的 RL
  2. 长尾覆盖：抽取覆盖的是 teacher "愿意 verbose 输出"的部分，对罕见领域、安全边界附近的推理样本难以覆盖
  3. 检测的猫鼠游戏：sybil 与 query pattern 检测会持续演进，攻击方也会用更精细的分布拟合反制
  4. 水印化方向：在 CoT token 分布里嵌入可验证的水印，是目前学术探索中的开放方向，尚无工业部署
  5. 侧信道研究空白：基于响应时延、reasoning_tokens 计数的统计攻击，目前几乎没有系统论文

§ 9 思考题

问：如果一家厂商完全隐藏 reasoning_tokens 计数（连"消耗了多少 reasoning token"都不告诉用户），是否能彻底防御 CoT 抽取攻击？

答：不能。理由有三：

  • 响应时延仍然泄露推理复杂度（虽然噪声更大）
  • Prompted Externalization 与 Multi-turn Unfolding 这两类不依赖元信息的攻击仍然可行
  • Trace Inversion 类方法只需要 (question, final_answer)，连摘要都不一定必要——只是没有摘要时 token recovery rate 会下降

真正的防御只能是多层叠加：访问层管控 + 内容层重构 + 输出层遮蔽 + 法律 + 水印。任一单点都不构成完整防御。
