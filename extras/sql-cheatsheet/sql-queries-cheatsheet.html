01 SELECT 完整骨架

1.1 SQL 查询执行顺序（核心 · 必须背）

写出来的顺序 ≠ 执行顺序。SQL 查询的逻辑执行顺序是：

FROM     表 + JOIN              -- 1. 先确定数据来源
WHERE    行级过滤              -- 2. 在聚合前剔除不要的行
GROUP BY 分组                  -- 3. 聚合的"桶"
HAVING   组级过滤              -- 4. 在聚合后剔除不要的组
SELECT   选列 / 计算列         -- 5. 投影
DISTINCT 去重                  -- 6.
ORDER BY 排序                  -- 7.
LIMIT    取前 N                 -- 8.
实战意义：能用 WHERE 过滤就不要等到 HAVING 才过滤——WHERE 在聚合前，能少处理大量数据。

1.2 最完整的 SELECT 模板

SELECT [ALL | DISTINCT]
       column1, column2, AGG(column3) AS alias
FROM   table1
JOIN   table2 ON  table1.id = table2.t1_id
WHERE  条件
GROUP BY column1
HAVING AGG(column3) > 10
ORDER BY column1 [ASC|DESC]
LIMIT 100 OFFSET 200;

2.2 LIKE 通配符

符号含义例子
%匹配 0 个或多个字符'Z%' → "Zhu", "Zhang"
_匹配恰好 1 个字符'Z_u' → "Zhu", 但不匹配 "Zhou"
ILIKE (PostgreSQL)不区分大小写name ILIKE 'zhu%'
LIKE '%xxx%'（开头通配）无法用 B-tree 索引，全表扫描。考虑改用全文索引（PG 的 tsvector、MySQL 的 FULLTEXT）。

2.3 IN 与 NOT IN 的 NULL 陷阱

-- 看起来正常,但 NULL 会让你失望:
SELECT * FROM users
WHERE id NOT IN (1, 2, NULL);
-- 结果: 0 行！因为 id != NULL 永远是 NULL (不是 TRUE)

-- 正确做法:
SELECT * FROM users
WHERE id NOT IN (1, 2)
  AND id IS NOT NULL;

03 JOIN 七种姿势

3.1 五种核心 JOIN

JOIN 类型含义视觉
INNER JOIN两边都有匹配的行A ∩ B
LEFT JOIN左表全保留，右表没匹配 → NULLA（含未匹配）
RIGHT JOIN右表全保留，左表没匹配 → NULLB（含未匹配）
FULL OUTER JOIN两边都保留，缺失方 → NULLA ∪ B
CROSS JOIN笛卡尔积（每行配每行）A × B

3.2 代码示例

-- 表准备
users(id, name)
orders(id, user_id, amount)

-- INNER JOIN: 只要有订单的用户
SELECT u.name, o.amount
FROM   users u
INNER JOIN orders o ON o.user_id = u.id;

-- LEFT JOIN: 所有用户,包括没下单的(amount 为 NULL)
SELECT u.name, o.amount
FROM   users u
LEFT JOIN orders o ON o.user_id = u.id;

-- "反 JOIN": 找出从未下单的用户
SELECT u.name
FROM   users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE  o.id IS NULL;

3.3 两种特殊 JOIN

SELF JOIN（同表自联）

-- 找出员工与其经理的对应关系
SELECT e.name AS employee,
       m.name AS manager
FROM   employees e
LEFT JOIN employees m ON m.id = e.manager_id;

LATERAL JOIN（横向应用 · PostgreSQL）

-- 对每个用户,关联他的最近 3 笔订单
SELECT u.name, o.*
FROM   users u
CROSS JOIN LATERAL (
    SELECT * FROM orders
    WHERE user_id = u.id
    ORDER BY created_at DESC
    LIMIT 3
) o;

04 GROUP BY + 聚合函数

4.1 全部聚合函数

函数含义NULL 处理
COUNT(*)行数不忽略
COUNT(col)非 NULL 行数忽略 NULL
COUNT(DISTINCT col)不同值的数量忽略 NULL
SUM(col)求和忽略 NULL
AVG(col)平均（=SUM/COUNT）忽略 NULL
MIN/MAX(col)最小/最大忽略 NULL
STRING_AGG(col, ',') / GROUP_CONCAT字符串拼接忽略 NULL
ARRAY_AGG(col)聚成数组（PG）保留 NULL
JSON_AGG(col)聚成 JSON 数组（PG）保留 NULL
BOOL_AND/OR(col)逻辑与/或忽略 NULL

4.2 GROUP BY 经典用法

-- 每个用户的订单数 + 总金额
SELECT user_id,
       COUNT(*) AS order_count,
       SUM(amount) AS total
FROM   orders
GROUP BY user_id;

-- 多列分组
SELECT user_id, status, COUNT(*)
FROM   orders
GROUP BY user_id, status;

4.3 GROUP BY 进阶：ROLLUP / CUBE / GROUPING SETS

-- 按 (city, product) 分组,同时计算 city 小计 + 总计
SELECT city, product, SUM(sales)
FROM   sales
GROUP BY ROLLUP (city, product);

-- 等价于:
GROUP BY GROUPING SETS (
  (city, product),
  (city),
  ()
);

05 ORDER BY + LIMIT + 分页

5.1 ORDER BY 完整语法

ORDER BY
  price DESC,                    -- 主排序
  name ASC,                      -- 次排序
  CASE WHEN status='urgent' THEN 0 ELSE 1 END;  -- 自定义排序

-- NULL 排序
ORDER BY price DESC NULLS LAST;
ORDER BY price ASC NULLS FIRST;

5.2 分页两种姿势

OFFSET + LIMIT（最常见但慢）

-- 第 5 页,每页 20 条
SELECT * FROM posts
ORDER BY created_at DESC
LIMIT 20 OFFSET 80;
OFFSET 大数性能差：OFFSET 100000 数据库要先扫前 10w 行再丢掉。深翻页用游标分页。

游标分页（Keyset Pagination · 推荐）

-- 第一页
SELECT * FROM posts
ORDER BY id DESC
LIMIT 20;

-- 下一页 (用上一页最后一条的 id)
SELECT * FROM posts
WHERE  id < 12345            -- 上一页最后一条 id
ORDER BY id DESC
LIMIT 20;
游标分页用索引 seek 替代 sort，OFFSET 1,000,000 时性能差距可达 100×+。

06 子查询全谱系

6.1 标量子查询（返回单个值）

-- 每个商品价格相对于平均价格的差
SELECT name,
       price - (SELECT AVG(price) FROM products) AS diff
FROM   products;

6.2 列子查询（IN）

-- 找出有 VIP 订单的用户
SELECT * FROM users
WHERE  id IN (SELECT user_id FROM orders WHERE vip=true);

6.3 行子查询

SELECT * FROM products
WHERE  (category, price) = ('书', 99);

6.4 派生表 / 子查询作 FROM

SELECT t.user_id, t.total
FROM (
  SELECT user_id, SUM(amount) AS total
  FROM   orders
  GROUP BY user_id
) t
WHERE t.total > 1000;

6.5 EXISTS / NOT EXISTS（推荐用法）

-- 有订单的用户
SELECT * FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.user_id = u.id
);

-- 没下过单的用户
SELECT * FROM users u
WHERE NOT EXISTS (
  SELECT 1 FROM orders o
  WHERE o.user_id = u.id
);
EXISTS vs IN 的选择：
  • 子查询结果集小 → IN 更直观
  • 子查询结果集大 / 有 NULL 风险 / 需要相关子查询 → EXISTS 更稳
  • 大多数现代数据库优化器会把两者优化成同样的执行计划

07 CTE 与递归 CTE

7.1 普通 CTE（提高可读性）

WITH high_spenders AS (
  SELECT user_id, SUM(amount) AS total
  FROM   orders
  GROUP BY user_id
  HAVING SUM(amount) > 10000
)
SELECT u.name, h.total
FROM   users u
JOIN   high_spenders h ON h.user_id = u.id;

7.2 多 CTE 链式

WITH
  step1 AS (SELECT ... FROM ...),
  step2 AS (SELECT ... FROM step1),
  step3 AS (SELECT ... FROM step2)
SELECT * FROM step3;

7.3 递归 CTE（处理层级 / 树 / 图）

-- 找出某个员工的所有下级 (任意层级)
WITH RECURSIVE subordinates AS (
  -- 锚定 (anchor)
  SELECT id, name, manager_id, 1 AS level
  FROM   employees
  WHERE  id = 42           -- 起始员工

  UNION ALL

  -- 递归部分
  SELECT e.id, e.name, e.manager_id, s.level + 1
  FROM   employees e
  JOIN   subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
递归 CTE 是 SQL 处理树结构、图遍历、生成序列、爬层级的核心武器。学会它能解决 80% "看似要写循环"的问题。

7.4 生成数字序列（递归 CTE 妙用）

-- 生成 1 到 100
WITH RECURSIVE nums(n) AS (
  SELECT 1
  UNION ALL
  SELECT n + 1 FROM nums WHERE n < 100
)
SELECT n FROM nums;

-- PostgreSQL 简化版:
SELECT * FROM generate_series(1, 100);

08 窗口函数全集 ⭐

窗口函数 = SQL 现代化的分水岭。能优雅完成"对每行计算一个相对于某个窗口的值"——比如排名、累计、相对前一行、移动平均。是替代复杂自联和子查询的核心武器。

8.1 通用语法

function() OVER (
  [PARTITION BY 分组列]
  [ORDER BY 排序列]
  [ROWS|RANGE BETWEEN 起点 AND 终点]
)

8.2 排名类函数

函数遇到相等怎么办
ROW_NUMBER()给每行一个唯一序号（即使值相同）
RANK()相同值同名次，下一名次跳号（1, 2, 2, 4）
DENSE_RANK()相同值同名次，不跳号（1, 2, 2, 3）
NTILE(N)等分为 N 组（用于百分位）
PERCENT_RANK()百分比排名 0-1
CUME_DIST()累积分布
 
-- 每个分类下按价格排名前 3 名
SELECT *
FROM (
  SELECT *,
         ROW_NUMBER() OVER (
           PARTITION BY category
           ORDER BY price DESC
         ) AS rn
  FROM products
) t
WHERE rn <= 3;

8.3 取相邻行：LAG / LEAD

-- 每笔订单 + 该用户上一笔订单的金额
SELECT user_id, amount,
       LAG(amount, 1) OVER (
         PARTITION BY user_id
         ORDER BY created_at
       ) AS prev_amount
FROM   orders;

-- 下一笔金额
LEAD(amount, 1) OVER (...)

8.4 首尾值：FIRST_VALUE / LAST_VALUE / NTH_VALUE

SELECT user_id, order_date,
       FIRST_VALUE(amount) OVER (
         PARTITION BY user_id
         ORDER BY order_date
       ) AS first_order
FROM orders;

8.5 累计 / 移动窗口

-- 累计销售额 (running total)
SELECT date, sales,
       SUM(sales) OVER (
         ORDER BY date
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total
FROM daily_sales;

-- 7 天移动平均
SELECT date, sales,
       AVG(sales) OVER (
         ORDER BY date
         ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS moving_avg_7d
FROM daily_sales;

8.6 窗口帧边界关键字

关键字含义
UNBOUNDED PRECEDING分区开始
N PRECEDING前 N 行
CURRENT ROW当前行
N FOLLOWING后 N 行
UNBOUNDED FOLLOWING分区结束

操作含义去重
UNION并集是
UNION ALL并集否（更快，推荐用 ALL 除非确实要去重）
INTERSECT交集是

10 NULL 完整处理

10.1 NULL 的特殊行为（务必记住）

NULL 不等于任何值，包括它自己：NULL = NULL 的结果是 NULL（不是 TRUE）。 
-- 永远用 IS NULL / IS NOT NULL,不要用 = NULL
WHERE deleted_at IS NULL;

-- 任何与 NULL 的算术返回 NULL
SELECT 1 + NULL;           -- → NULL
SELECT 'abc' || NULL;        -- → NULL

-- 聚合函数会忽略 NULL (除了 COUNT(*))
SELECT AVG(salary) FROM employees;
-- 等价于: SUM(salary) / COUNT(salary), 不算 NULL

10.2 NULL 处理函数

函数用途等价
COALESCE(a, b, c, ...)第一个非 NULL逐层 fallback
NULLIF(a, b)a==b 时返回 NULL，否则 a反向 fallback
IFNULL(a, b) (MySQL)等同 COALESCE(a, b)2 参数版
ISNULL(a, b) (SQL Server)同上
 
SELECT COALESCE(phone, email, '未填写') AS contact
FROM users;

11 CASE WHEN 与条件表达式

11.1 CASE WHEN（标准）

SELECT
  user_id,
  CASE
    WHEN total >= 100000 THEN '钻石'
    WHEN total >= 10000  THEN '金'
    WHEN total >= 1000   THEN '银'
    ELSE '普通'
  END AS tier
FROM user_summary;

-- 简单形式
CASE status
  WHEN 1 THEN '草稿'
  WHEN 2 THEN '已发布'
  ELSE '未知'
END

11.2 CASE WHEN 用作条件聚合

-- 一次性算多个状态的数量 (横表)
SELECT
  COUNT(CASE WHEN status='paid'   THEN 1 END) AS paid,
  COUNT(CASE WHEN status='unpaid' THEN 1 END) AS unpaid,
  SUM(CASE WHEN status='paid' THEN amount ELSE 0 END) AS revenue
FROM orders;

13 DDL：CREATE / ALTER / DROP

13.1 CREATE TABLE 完整模板

CREATE TABLE orders (
  id          BIGSERIAL PRIMARY KEY,
  user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  amount      NUMERIC(10, 2) NOT NULL CHECK (amount >= 0),
  status      VARCHAR(20) NOT NULL DEFAULT 'pending',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, status),
  CHECK (status IN ('pending', 'paid', 'refunded'))
);

-- 创建索引
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_created ON orders(created_at DESC);

14 DML：INSERT / UPDATE / DELETE / UPSERT

14.1 INSERT

-- 单条
INSERT INTO users (name, email) VALUES ('Alice', 'a@x.com');

-- 多条
INSERT INTO users (name, email) VALUES
  ('Alice', 'a@x.com'),
  ('Bob',   'b@x.com');

-- INSERT ... SELECT
INSERT INTO archive_orders
SELECT * FROM orders WHERE created_at < '2020-01-01';

-- 返回插入后的值 (PG)
INSERT INTO users (name) VALUES ('Eve')
RETURNING id, created_at;

14.2 UPDATE

UPDATE orders
SET    status = 'paid',
       paid_at = NOW()
WHERE  id = 42;

-- 用 JOIN 更新
UPDATE orders o
SET    total = total * 0.9
FROM   users u
WHERE  o.user_id = u.id
  AND  u.vip = true;

14.3 DELETE

DELETE FROM orders WHERE status = 'cancelled';

-- 配 JOIN
DELETE FROM orders o
USING users u
WHERE o.user_id = u.id
  AND u.deleted = true;

14.4 UPSERT（关键！）

-- PostgreSQL: ON CONFLICT
INSERT INTO users (email, name)
VALUES ('a@x.com', 'Alice')
ON CONFLICT (email) DO UPDATE
  SET name = EXCLUDED.name,
      updated_at = NOW();

-- 只在冲突时跳过
ON CONFLICT (email) DO NOTHING;

-- MySQL: ON DUPLICATE KEY UPDATE
INSERT INTO users (email, name) VALUES ('a@x.com', 'Alice')
ON DUPLICATE KEY UPDATE name = VALUES(name);

-- SQL 标准: MERGE
MERGE INTO users u
USING (VALUES ('a@x.com', 'Alice')) v(email, name)
ON u.email = v.email
WHEN MATCHED     THEN UPDATE SET name = v.name
WHEN NOT MATCHED THEN INSERT (email, name) VALUES (v.email, v.name);

15 索引、视图、事务

15.1 索引完整种类

索引类型适用引擎
B-tree等值、范围、排序（默认）全部
Hash仅等值，特定场景比 B-tree 快PG, MySQL(MEMORY)
GINJSONB、数组、全文PG
GiST地理空间、自定义PG
BRIN大表、值与物理位置相关PG
FULLTEXT全文搜索MySQL, PG (tsvector)

15.2 索引使用规律

-- 联合索引: 最左前缀原则
CREATE INDEX idx_user_status_time
  ON orders(user_id, status, created_at);

-- 能用上:
WHERE user_id = ?
WHERE user_id = ? AND status = ?
WHERE user_id = ? AND status = ? AND created_at > ?

-- 用不上:
WHERE status = ?               -- 跳过了 user_id
WHERE created_at > ?           -- 跳过了前两个
索引失效场景：
  • 列上有函数：WHERE UPPER(email) = ...
  • 类型不匹配自动转换：字符串列被当数字比
  • 前导通配：LIKE '%xxx%'
  • OR 不同列：WHERE a=1 OR b=2 通常各走一次

15.3 视图

-- 普通视图 (执行时实时查询)
CREATE VIEW v_active_users AS
SELECT * FROM users WHERE deleted_at IS NULL;

-- 物化视图 (PG, 缓存结果,需手动刷新)
CREATE MATERIALIZED VIEW mv_daily_sales AS
SELECT DATE_TRUNC('day', created_at), SUM(amount)
FROM orders GROUP BY 1;

-- 刷新
REFRESH MATERIALIZED VIEW mv_daily_sales;

15.4 事务

BEGIN;  -- 或 START TRANSACTION

  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;

  -- 保存点
  SAVEPOINT sp1;
  UPDATE ...;
  ROLLBACK TO SAVEPOINT sp1;  -- 仅回滚到此

COMMIT;   -- 或 ROLLBACK 全部撤销

-- 隔离级别
SET TRANSACTION ISOLATION LEVEL
  READ COMMITTED       -- PG/Oracle 默认
  | REPEATABLE READ    -- MySQL InnoDB 默认
  | SERIALIZABLE;     -- 最严, 性能代价

16 18 个高频业务模式

16.1 Top-N per Group（每组前 N）⭐

-- 每个分类下销量前 3
WITH ranked AS (
  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY category ORDER BY sales DESC) AS rn
  FROM products
)
SELECT * FROM ranked WHERE rn <= 3;

16.2 累计求和（Running Total）

SELECT date, sales,
       SUM(sales) OVER (ORDER BY date) AS cumulative
FROM daily_sales;

16.3 移动平均

SELECT date, sales,
       AVG(sales) OVER (
         ORDER BY date
         ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS ma7
FROM daily_sales;

16.4 行转列（Pivot）

-- 月销售透视
SELECT product_id,
       SUM(CASE WHEN EXTRACT(MONTH FROM d) = 1  THEN sales ELSE 0 END) AS jan,
       SUM(CASE WHEN EXTRACT(MONTH FROM d) = 2  THEN sales ELSE 0 END) AS feb,
       ...
FROM monthly_sales
GROUP BY product_id;

16.5 列转行（Unpivot）

-- PostgreSQL: 用 UNION ALL 或 unnest
SELECT id, 'jan' AS month, jan AS sales FROM t
UNION ALL
SELECT id, 'feb', feb FROM t
UNION ALL
...

16.6 找重复行

-- 哪些 email 重复了 + 重复几次
SELECT email, COUNT(*) AS cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- 保留每组第一条,删除其他重复 (PG)
DELETE FROM users
WHERE id NOT IN (
  SELECT MIN(id) FROM users GROUP BY email
);

16.7 留存分析（次日留存）

WITH day1 AS (SELECT DISTINCT user_id FROM events WHERE date = '2026-01-01'),
     day2 AS (SELECT DISTINCT user_id FROM events WHERE date = '2026-01-02')
SELECT
  (SELECT COUNT(*) FROM day1 JOIN day2 USING (user_id))::float /
  (SELECT COUNT(*) FROM day1) AS retention;

16.8 漏斗分析

SELECT
  COUNT(DISTINCT CASE WHEN event='view'     THEN user_id END) AS step1_view,
  COUNT(DISTINCT CASE WHEN event='cart'     THEN user_id END) AS step2_cart,
  COUNT(DISTINCT CASE WHEN event='checkout' THEN user_id END) AS step3_checkout,
  COUNT(DISTINCT CASE WHEN event='paid'     THEN user_id END) AS step4_paid
FROM events;

16.9 时间窗口对比（同比 / 环比）

WITH curr AS (SELECT SUM(amount) AS total FROM orders WHERE d >= '2026-05-01'),
     prev AS (SELECT SUM(amount) AS total FROM orders
                WHERE d BETWEEN '2026-04-01' AND '2026-04-30')
SELECT curr.total, prev.total,
       (curr.total - prev.total) / prev.total::float AS growth
FROM curr, prev;

16.10 找缺失的序号（Gap Detection）

-- 找出 id 序列中的空洞
SELECT id + 1 AS missing_start,
       LEAD(id) OVER (ORDER BY id) - 1 AS missing_end
FROM t
WHERE LEAD(id) OVER (ORDER BY id) - id > 1;

16.11 中位数

SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM employees;

16.12 随机抽样

-- 抽 100 条
SELECT * FROM large_table ORDER BY RANDOM() LIMIT 100;

-- PG: 用 TABLESAMPLE 更快
SELECT * FROM large_table TABLESAMPLE BERNOULLI (1);  -- 1%

16.13 行号、相邻行差

-- 跟上一次订单间隔多少天
SELECT user_id, order_date,
       order_date - LAG(order_date) OVER (
         PARTITION BY user_id ORDER BY order_date
       ) AS days_since_last
FROM orders;

16.14 连续登录天数（岛屿与缝隙问题）

-- 经典岛屿问题: 算每个用户的最长连续登录
WITH marked AS (
  SELECT user_id, date,
         date - ROW_NUMBER() OVER (
           PARTITION BY user_id ORDER BY date
         ) * INTERVAL '1 day' AS grp
  FROM logins
)
SELECT user_id, MAX(streak) AS longest_streak
FROM (
  SELECT user_id, grp, COUNT(*) AS streak
  FROM marked
  GROUP BY user_id, grp
) t
GROUP BY user_id;

16.15 第 N 高的薪水

-- 第 N 高 (含并列)
SELECT DISTINCT salary
FROM employees
ORDER BY salary DESC
OFFSET N - 1 LIMIT 1;

-- 或用 DENSE_RANK
SELECT salary
FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) t
WHERE rk = N;

16.16 JSON 字段查询（PG）

-- JSON 字段提取
SELECT data->'user'->>'name' AS name
FROM events
WHERE data->'user'->>'role' = 'admin';

-- JSON 数组展开
SELECT jsonb_array_elements(data->'tags') FROM events;

16.17 全文搜索（PG）

SELECT * FROM articles
WHERE to_tsvector('simple', title || ' ' || body)
   @@ to_tsquery('simple', 'postgres & index');

16.18 批量更新（数据迁移场景）

-- 用 VALUES 列表批量更新
UPDATE products p
SET    price = v.price
FROM (
  VALUES (1, 99.0), (2, 59.0), (3, 88.0)
) AS v(id, price)
WHERE p.id = v.id;

17 性能优化要点

17.1 永远先看执行计划

EXPLAIN              your_query;   -- 静态分析
EXPLAIN ANALYZE      your_query;   -- 实际跑一遍, 给真实耗时 (PG)
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) ...;

17.2 优化清单

🟢 查询层

  • WHERE 列建索引
  • JOIN 列建索引
  • 避免 SELECT *，只取需要的列
  • EXISTS 替代 IN（大数据集）
  • 用 UNION ALL 替代 UNION（如果不需去重）
  • 用游标分页替代深 OFFSET

🟢 数据库层

  • VACUUM / ANALYZE（PG）
  • ANALYZE TABLE（MySQL）
  • 统计信息要更新
  • 分区表（大表场景）
  • 物化视图缓存常用聚合
  • 读写分离 / 副本

17.3 索引数量的平衡

索引不是越多越好。每个索引让写入慢一点（每次 INSERT/UPDATE/DELETE 都要维护）。一般经验：单表索引数控制在 5-7 个以内。

18 SQL 注入防御与红队视角

18.1 SQL 注入是什么

把用户输入拼接到 SQL 字符串里，导致用户输入被当成 SQL 代码执行。

-- ❌ 致命错误的写法
sql = f"SELECT * FROM users WHERE name='{user_input}'"

-- 用户输入 admin' OR '1'='1
-- 拼接结果:
SELECT * FROM users WHERE name='admin' OR '1'='1';
-- → 返回全部用户

18.2 正确做法：参数化查询

# Python (psycopg2)
cur.execute(
  "SELECT * FROM users WHERE name = %s",
  (user_input,)        # 参数,会被自动转义
)

# Node.js
db.query(
  'SELECT * FROM users WHERE name = $1',
  [user_input]
)
核心原则：永远不要拼接 SQL 字符串。所有用户输入都通过参数占位符传入。这一条做好了，SQL 注入基本就死了。

18.3 红队视角的 5 类注入手法

类型原理典型 payload
Union-based通过 UNION 直接拼接读数据' UNION SELECT 1,2,3--
Boolean Blind通过响应差异推断' AND 1=1-- vs ' AND 1=2--
Time-based Blind通过响应时间推断' AND SLEEP(5)--
Error-based故意触发错误获取信息各种类型转换错误
Stacked Queries分号执行额外语句'; DROP TABLE--

18.4 ORM 的安全保证

正经 ORM（SQLAlchemy、Django ORM、TypeORM、GORM）默认参数化，安全。但要警惕这些反模式：

# ❌ 任何"原生 SQL + 字符串拼接"都危险
db.execute(f"SELECT * FROM t WHERE id = {user_id}")

# ❌ 动态表名/列名 ORM 也保护不了
db.execute(f"SELECT * FROM {user_input_table}")
# → 必须自己用白名单 + 严格校验

20 术语表 + 参考

20.1 术语表

DDL (Data Definition Language)
定义数据结构的语句：CREATE / ALTER / DROP / TRUNCATE / RENAME。
DML (Data Manipulation Language)
操作数据的语句：INSERT / UPDATE / DELETE / SELECT / MERGE。
DCL (Data Control Language)
权限控制：GRANT / REVOKE。
TCL (Transaction Control Language)
事务控制：BEGIN / COMMIT / ROLLBACK / SAVEPOINT。
OLTP / OLAP
联机事务处理 / 联机分析处理。OLTP 重高并发短事务，OLAP 重大数据复杂查询。
ACID
事务四性质：原子性 / 一致性 / 隔离性 / 持久性。
CTE (Common Table Expression)
WITH 子句定义的临时结果集，提高可读性 + 支持递归。
Window Function（窗口函数）
对一组相关行计算单行结果的函数，不像聚合函数那样合并行。
Index Seek vs Index Scan
Seek 是直接跳到目标位置，O(log n)；Scan 是按顺序扫整个索引。Seek 更快。
EXPLAIN
查询计划解释器，看 SQL 实际怎么跑。优化的第一步。
UPSERT
UPDATE + INSERT 的组合，存在则更新，不存在则插入。
MVCC (Multi-Version Concurrency Control)
多版本并发控制，让读不阻塞写、写不阻塞读。PG / InnoDB 都用。
